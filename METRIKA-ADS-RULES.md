# Правила работы с Яндекс Директ API (MCP: yandex-direct)

## Получение кампаний — ВАЖНО

Когда пользователь просит "покажи кампании", "получи кампании", "что у меня запущено" — **всегда фильтруй по активным**:

```
get_campaigns с states: ["ON"]
```

**НЕ вызывай get_campaigns без фильтра states** — без фильтра API вернёт ВСЕ кампании (включая архивные, завершённые, выключенные), и результат будет бесполезным.

**НЕ ставь маленький limit** (5, 10) — можно пропустить нужные кампании. Лучше не указывать limit вообще (по умолчанию 100) или ставить достаточно большой.

## Фильтры states

| Значение    | Что значит                          |
|-------------|-------------------------------------|
| `ON`        | Активные, показываются              |
| `SUSPENDED` | Остановлены рекламодателем          |
| `OFF`       | Выключены (например, нет средств)   |
| `ENDED`     | Завершены по дате                   |
| `CONVERTED` | Конвертированные (устаревший формат)|

## Типовые сценарии

- **"Покажи кампании" / "Мои кампании"** → `states: ["ON"]`
- **"Покажи все кампании включая остановленные"** → `states: ["ON", "SUSPENDED"]`
- **"Покажи вообще всё"** → без фильтра states (только если пользователь явно просит ВСЕ)
- **"Какие кампании остановлены?"** → `states: ["SUSPENDED"]`

## Атрибуция — ОБЯЗАТЕЛЬНО

**Есть две разные настройки, не путать:**

1. **Настройка кампании** (в интерфейсе Директа) — влияет на обучение алгоритма. У всех кампаний стоит **AUTO** — не менять.
2. **Атрибуция отчёта** (параметр `attribution` в get_campaign_stats) — влияет только на то, как мы смотрим данные. Не трогает кампанию.

**Правило для отчётов:** всегда передавай `attribution: "LYDC"` в get_campaign_stats и get_search_queries. Дефолт API — LC, данные будут другими и несовместимыми между чатами.

В файлах кампаний и задачах писать «атрибуция отчёта: LYDC», не «атрибуция: LYDC».

## Экономия токенов — field_names

`get_campaigns` поддерживает параметр `field_names` — используй его чтобы запрашивать только нужные поля. Без `field_names` API возвращает ВСЕ поля (стратегия, минус-фразы, настройки, уведомления), что переполняет контекст.

**Правила:**
- Для получения списка кампаний (обзор, каннибализация) → `field_names: ["Id", "Name"]`
- Для анализа конкретной кампании → не указывай `field_names` (нужны все данные)
- Всегда думай какие поля реально нужны для текущей задачи

## UTM-метки — ОБЯЗАТЕЛЬНО

При создании кампании **всегда добавляй UTM-метки** через `tracking_params` на уровне группы (update_adgroup). Это применяется ко всем ссылкам: основная + быстрые ссылки.

**Шаблон:**
```
utm_source=yandex&utm_medium=cpc&utm_campaign={slug}&utm_content={ad_id}&utm_term={keyword}
```

Где `{slug}` — короткое латинское название кампании (например `moy-produkt`, `usluga-msk`, `brand-campaign`).

**Динамические параметры Директа:**
- `{ad_id}` — ID объявления
- `{keyword}` — ключевая фраза
- `{campaign_id}` — ID кампании (можно использовать вместо slug)

**Не забывай:** UTM ставится после создания группы через `update_adgroup(adgroup_id, tracking_params: "...")`.

## Порядок создания кампании через API — СТРОГО ПОСЛЕДОВАТЕЛЬНО

Кампания содержит **несколько групп объявлений** (по кластерам семантики). Шаги 2–7 повторяются для каждого кластера.

```
1. add_campaign → campaign_id

--- ЦИКЛ ПО КЛАСТЕРАМ (для каждого кластера семантики): ---
2. add_adgroup(campaign_id) → adgroup_id
3. add_keywords(adgroup_id, keywords="фразы кластера")  ← ОБЯЗАТЕЛЬНО!
4. update_adgroup(adgroup_id, tracking_params=...)  ← UTM
5. add_ad × 2 (A/B тест, заголовки индивидуальны для кластера!) → banner_ids
6. add_sitelinks + привязка
7. add_ad_extension + привязка
--- КОНЕЦ ЦИКЛА ---

8. add_labels(campaign_id, ВСЕ banner_ids, labels="...")  ← на объявления!
```

**ВАЖНО:** `add_adgroup` НЕ принимает параметр `keywords` — ключевые фразы добавляются ОТДЕЛЬНО через `add_keywords`. Без этого шага группа работает только на автотаргетинге.

**ВАЖНО:** `add_labels` назначает метки на объявления (banner_ids), а не на кампанию. В интерфейсе метки отображаются на уровне групп. Одни и те же метки — на ВСЕ объявления всех кластеров.

**Кластеры → группы:** каждый кластер семантики (гео-район, интент, общий) — отдельная группа объявлений. Гео-кластеры получают название района в заголовках объявлений.

## Семантика — сбор и кластеризация ключевых фраз

**Целевой объём:** 30–80 ключевых фраз на кампанию, распределённых по 3–10 кластерам (группам объявлений).

**Сбор:**
1. `get_semantic_cluster(theme)` → seed-матрица: {модификатор} × {объект} + город → 10 seeds
2. `check_search_volume(phrases, region_ids)` → SearchedWith расширяет seeds (батчи по 10 фраз!)
3. Фильтрация: Shows >= 10 (>= 5 для малых городов), минус нерелевантные
4. Опционально: второй раунд check_search_volume по высокочастотным находкам

**Кластеризация (после сбора):**
1. Гео-кластеры (приоритет): фразы с районами/микрорайонами → отдельные группы
2. Интент-кластеры: «купить», «цена/недорого», «от собственника/вторичка»
3. Общий кластер: широкие фразы без уточнений
4. Минимум 3 фразы на кластер или 1 фраза с Shows >= 50

**Гео-кластеры:** название района идёт в Title объявления, посадочная URL — с фильтром района (если сайт поддерживает).

**Лимиты:** `check_search_volume` принимает до 10 фраз за раз, всегда передавай `region_ids` города.

## Порядок работы с API (чтение)

1. **get_campaigns** — начинай с получения кампаний (с правильным фильтром!)
2. **get_adgroups** — группы объявлений внутри кампании (нужен campaign_id)
3. **get_ads** — объявления (фильтр по campaign_ids или adgroup_ids)
4. **get_keywords** — ключевые фразы (фильтр по campaign_ids или adgroup_ids)
5. **get_campaign_stats** — статистика за период (нужен campaign_id, date_from, date_to)

## Управление кампаниями

| Действие | Инструмент | Условие |
|----------|-----------|---------|
| Остановить показы | **suspend_campaign** | Работает только для кампаний в статусе `ON` |
| Возобновить показы | **resume_campaign** | Работает только для кампаний в статусе `SUSPENDED` |
| Архивировать | **archive_campaign** | Сначала нужно остановить (suspend), потом архивировать |
| Переименовать | **update_campaign** | Передать `name` |
| Изменить бюджет | **update_campaign** | Передать `daily_budget_amount` в рублях (целое число) |

**Цепочки действий:**
- Остановить: `suspend_campaign(campaign_id)`
- Возобновить: `resume_campaign(campaign_id)`
- Архивировать: `suspend_campaign(campaign_id)` → `archive_campaign(campaign_id)`

## Редактирование объявлений — ВАЖНО

**Для изменения текста, заголовков, ссылок — ВСЕГДА используй `update_ad`.** Это partial update: отправляются только указанные поля, остальные остаются без изменений.

**НИКОГДА не удаляй объявление и не создавай новое** вместо обновления — это уничтожает:
- Быстрые ссылки (`sitelink_set_id`)
- Уточнения (`ad_extension_ids`)
- Изображения (`ad_image_hash`)
- Видеодополнения, цены и другие настройки, заполненные вручную

**Если по какой-то причине нужно создать новое объявление:**
1. Сначала `get_ads` с `text_ad_field_names` — получить текущие привязки
2. Перенести `sitelink_set_id`, `ad_extension_ids`, `ad_image_hash` и другие поля в новое объявление

## Места показа — ВАЖНО

При создании кампаний **всегда выбираем только "Продвижение в поисковой выдаче"**. Всё остальное отключаем:
- **Динамические места на поиске** — ВЫКЛЮЧЕНЫ (нерелевантные размещения, размывают бюджет)
- **Товарная галерея на поиске** — ВЫКЛЮЧЕНА (не наш формат)
- **РСЯ (Рекламная сеть Яндекса)** — ВЫКЛЮЧЕНА

В API это значит: `network_strategy: "SERVING_OFF"` (показы в сетях отключены).
Не включать `NETWORK_DEFAULT`, `WB_MAXIMUM_CLICKS` для сетей и не использовать РСЯ-стратегии без явной просьбы пользователя.

## GEO-кампании (зарубежная аудитория) — ВАЖНО

При создании кампаний на зарубежное гео **обязательно отключать** `ENABLE_AREA_OF_INTEREST_TARGETING`:

```
settings: [{"option": "ENABLE_AREA_OF_INTEREST_TARGETING", "value": false}]
```

**Почему:** по умолчанию Директ показывает рекламу не только тем, кто физически находится в регионе, но и тем, кто им «интересуется». Для GEO-кампаний это приводит к показам пользователям из РФ, что размывает бюджет и смешивает аудитории.

**Доступные страны для таргетинга** (проверено):
- Казахстан (159), Беларусь (149), Грузия (169), Литва (117)
- Израиль (181), Германия (96), Турция (983), ОАЭ (210)

**Недоступные для таргетинга:** Латвия, Эстония (API возвращает ошибку 5120).

## Категории запросов на Поиске — ВАЖНО

При настройке автотаргетинга выбираем **только узкие и целевые** запросы. Всё остальное отключаем:

- **Целевые запросы** — ВКЛЮЧЕНЫ
- **Узкие запросы** — ВКЛЮЧЕНЫ
- **Широкие запросы** — ВЫКЛЮЧЕНЫ (размывают бюджет на нецелевой трафик)
- **Сопутствующие запросы** — ВЫКЛЮЧЕНЫ
- **Альтернативные запросы** — ВЫКЛЮЧЕНЫ

Не включать широкие/сопутствующие/альтернативные без явной просьбы пользователя.

## Бюджет — РУБЛИ, НЕ микроюниты

**В Яндекс Директе бюджеты НЕДЕЛЬНЫЕ, а не дневные.**

**КРИТИЧЕСКИ ВАЖНО:** MCP-инструменты `add_campaign` и `update_campaign` принимают бюджет в **РУБЛЯХ**. Передавай число как есть. **НИКОГДА не умножай на 1 000 000** и не конвертируй в микроюниты — MCP-сервер сам обрабатывает конвертацию.

Параметр `daily_budget_amount` — это на самом деле **недельный** бюджет.
Примеры: 300 руб/нед → `daily_budget_amount: 300`, 1000 руб/нед → `daily_budget_amount: 1000`.

Когда пользователь говорит "бюджет 400 рублей" — уточни: в неделю? Если да, передавай `400`.

## Выбор стратегии — порядок запуска

WB_MAXIMUM_CONVERSION_RATE и AVERAGE_CPA — это **два режима одной стратегии** "Максимум конверсий":

| Режим | Что делает | Когда использовать |
|-------|-----------|-------------------|
| **WB_MAXIMUM_CONVERSION_RATE** | Максимум конверсий в рамках бюджета, без потолка CPA | Старт кампании, нет данных о конверсиях |
| **AVERAGE_CPA** | Максимум конверсий с удержанием целевой цены | Есть 10+ конверсий в неделю стабильно |

**Правило: всегда запускай с WB_MAXIMUM_CONVERSION_RATE, переключай на AVERAGE_CPA только после накопления данных.**

Почему:
- Для обучения стратегии нужно **минимум 10 конверсий в неделю** (документация Яндекса)
- AVERAGE_CPA без данных либо режет показы (мало трафика → замкнутый круг), либо превышает заданный CPA **в несколько раз**
- Яндекс прямо рекомендует: "Если нет накопленной статистики, используйте Максимум конверсий с ограничением только недельного бюджета"

**Порядок:**
1. Запуск → `WB_MAXIMUM_CONVERSION_RATE` + правильная цель + недельный бюджет
2. Ждём 2-3 недели, пока стратегия обучится (10+ конверсий/нед)
3. Переключаем → `AVERAGE_CPA` с целевой ценой на основе реальных данных

**НЕ ставь AVERAGE_CPA на кампанию с 0 конверсий** — это контрпродуктивно.

## Несколько целей (PriorityGoals) — ВАЖНО

Когда кампания оптимизируется по **нескольким целям** (например, form_sum_leads + received_real_calls), используй параметр `priority_goals` в `add_campaign` / `update_campaign`:

```
priority_goals: [{"goal_id": 34275273, "value": 0}, {"goal_id": 500485089, "value": 0}]
```

**Правила:**
- `value` — ценность конверсии в рублях (0 = минимум 1₽, сервер ставит 1 000 000 микроюнитов)
- При передаче `priority_goals` GoalId в стратегии **автоматически ставится 13** (= «оптимизация по приоритетным целям»)
- Не передавай отдельно `goal_id` при использовании `priority_goals` — сервер сам подставит 13
- В интерфейсе Директа это выглядит как «Несколько целей» с настройкой ценности для каждой

**API-нюансы:**
- При **создании** кампании: Items содержат GoalId, Value, IsMetrikaSourceOfValue
- При **обновлении** кампании: Items дополнительно требуют поле `"Operation": "SET"` — сервер добавляет автоматически
- `Value` в API — в микроюнитах (1₽ = 1 000 000), минимум 300 000 (0.3₽)
- `IsMetrikaSourceOfValue: "NO"` — ценность задаётся вручную, не из Метрики

**Не путай:** `goal_id` в стратегии (одна цель) vs `priority_goals` (несколько целей). При нескольких целях goal_id=13.

**Расчёт value:** используй `get_conversion_values(client_login, counter_id)` — он автоматически рассчитает ценности на основе реальных CPA по действующим кампаниям города. Если кампаний нет — вернёт средние по сети.

## Изменения, влияющие на обучение — СОГЛАСОВАНИЕ ОБЯЗАТЕЛЬНО

**НИКОГДА не вносить самостоятельно без явного подтверждения пользователя:**

1. **Смена цели оптимизации** (goal_id) — сбрасывает обучение полностью
2. **Смена стратегии** (WB_MAX_CONV → AVERAGE_CPA и т.д.)
3. **Добавление нового объявления** (A/B тест)
4. **Резкое изменение бюджета** (более +30% за раз)
5. **Добавление новых ключевых фраз**

**Порядок действий:**
1. Сначала **описать** что именно будет изменено и как это повлияет на обучение
2. Дождаться **явного «ок»** от пользователя
3. Спросить: **«Внести сейчас или поставить задачу на чт-пт?»**
4. Только после подтверждения — вносить изменения

**Почему чт-пт:** эти изменения вызывают «разведку» — алгоритм начинает тратить больше обычного, чтобы собрать данные. Если это происходит в понедельник, весь недельный бюджет уходит на разведку и к выходным кампания голодает. В чт-пт разведка попадает на начало нового бюджетного цикла.

**При обновлении через API:** если нужно изменить только бюджет — передавать ТОЛЬКО бюджет. НЕ менять попутно цель, стратегию или другие параметры, даже если кажется что «так правильнее». Каждое изменение — отдельное согласование.
