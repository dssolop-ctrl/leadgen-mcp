## Файлы кампаний

Для каждой кампании есть файл `campaigns/<utm_метка>.md`. Если файла нет — создать по шаблону `campaigns/template.md` до начала работы.

**Правила работы с файлом:**

1. После каждой проверки — обновить файл: добавить результаты в таблицу и обновить «Где сейчас»
2. Задачи НЕ хранятся в файле кампании — только история и текущее состояние

## Яндекс Директ API (MCP: yandex-direct) — ОБЯЗАТЕЛЬНО

При создании и управлении рекламой в Яндекс Директе — **всегда сверяйся с `METRIKA-ADS-RULES.md`**. Там правила получения кампаний, фильтры, бюджеты, стратегии, UTM-метки, атрибуция и нюансы API.

**При обнаружении новых общих правил и нюансов Яндекс Директ API** (не частных для конкретной кампании) — **обновляй `METRIKA-ADS-RULES.md`**.

### Краткие правила (подробности в METRIKA-ADS-RULES.md)

- **Получение кампаний** → всегда `states: ["ON"]`, не ставить маленький limit
- **field_names** → для обзора `["Id", "Name"]`, для анализа — все поля
- **Атрибуция отчётов** → всегда `attribution: "LYDC"` в get_campaign_stats и get_search_queries
- **UTM-метки** → обязательно через `tracking_params` на уровне группы
- **Места показа** → только поиск, `network_strategy: "SERVING_OFF"`
- **Автотаргетинг** → только целевые + узкие, остальные выключены
- **Бюджет** → НЕДЕЛЬНЫЙ, указывается в РУБЛЯХ (число). НЕ умножать на 1 000 000, НЕ конвертировать в микроюниты
- **Стратегия** → старт с WB_MAXIMUM_CONVERSION_RATE, переход на AVERAGE_CPA после 10+ конверсий/нед
- **Изменения** → смена цели/стратегии/бюджета >30% — только с подтверждением пользователя, лучше в чт-пт
- **GEO-кампании** → отключать `ENABLE_AREA_OF_INTEREST_TARGETING`

### Правило: всегда называй цели и счётчики по-человечески

При обсуждении кампаний, целей и счётчиков — **никогда не кидай голые ID**. Всегда используй понятные названия. Формат: «цель Отправка заявки (123456789)» или «счётчик основной (987654321)».

## Скиллы

Пять скиллов для управления, аналитики и оптимизации кампаний.

### Яндекс Директ (`yandex-direct`)

Workflow-гайд для управления кампаниями через MCP `yandex-direct`: создание, оптимизация, масштабирование.

- **Триггеры:** «создай кампанию», «оптимизируй кампанию», «статистика директа», «поисковые запросы», «ключевые слова», «минус-слова»
- **MCP-сервер:** `yandex-direct` (включает Директ + Метрику + Wordstat + задачи)
- **Workflows:** создание кампании, проверка здоровья, оптимизация по запросам, масштабирование

### VK Ads (`vk-ads`)

Workflow-гайд для управления рекламой через MCP `vk-ads`: кампании, таргетинг, ретаргетинг, контекстные фразы.

- **Триггеры:** «создай кампанию VK», «VK реклама», «таргетинг VK», «ретаргетинг», «look-alike»
- **MCP-сервер:** `vk-ads` (отдельный от yandex-direct, 76 инструментов)
- **Workflows:** создание кампании, ретаргетинг, контекстный таргетинг, анализ, масштабирование

### Яндекс Метрика (`yandex-metrika`)

Аналитика кампаний через MCP `metrika_*` инструменты: отчёты по UTM, CPA с оценкой здоровья, конверсии, сравнение периодов.

- **Триггеры:** «метрика», «аналитика», «CPA», «конверсии», «отчёт по кампании»
- **MCP-сервер:** `yandex-direct` (Метрика — часть единого MCP, токен не нужен отдельно)
- **Каденция:** пн — недельный отчёт, чт — сравнение периодов
- **Пороги CPA:** из `PROJECTS.md`

### Исследование спроса (`demand-research`)

Анализ семантики: упущенный спрос из поисковых запросов, сезонный прогноз бюджета, каннибализация ключевых, верификация интента.

- **Триггеры:** «исследование спроса», «упущенный спрос», «каннибализация», «сезонность»
- **MCP-сервер:** `yandex-direct` (Wordstat — часть единого MCP: `wordstat_top_requests`, `wordstat_dynamics`, `wordstat_regions`)
- **Ключевой скрипт:** `missed_demand.py` — парсинг CSV поисковых запросов → opportunities + waste

### Мониторинг позиций (`serp-monitor`)

Проверка позиций в Яндексе, анализ конкурентов, отслеживание динамики.

- **Триггеры:** «позиции в выдаче», «SERP», «конкуренты», «где я показываюсь»
- **Ключевой скрипт:** `search.sh` — поиск в Яндексе с выделением рекламных блоков
- **Трекинг:** `track_position.sh` — CSV-история позиций с трендами
- **Конфиг:** Yandex XML ключи в `tokens.env`
- **MCP-интеграция:** `get_campaign_stats`, `get_campaigns` для расхода и бюджетов

### Общая инфраструктура

Скиллы serp-monitor и demand-research используют `_shared/common.sh` (HTTP с retry/429, SHA256 кеш с TTL 6ч) и `_shared/json_helpers.py` (парсинг JSON, таблицы). Токены для SERP хранятся в `tokens.env` (шаблон: `tokens.env.example`).

## CI/CD

### Авто-релиз при пуше в main

Workflow `.github/workflows/release-on-push.yml` — при каждом пуше в `main` автоматически:

- Определяет следующую patch-версию (на основе последнего тега `vX.Y.Z`)
- Создаёт GitHub Release с полными архивами проекта (`.tar.gz` + `.zip`)
- Версионирование: `v1.2.1` → `v1.2.2` → `v1.2.3` и т.д.

Для поднятия minor/major — создать тег вручную (`git tag v1.3.0 && git push origin v1.3.0`), дальше patch пойдёт от него.

## Задачи и напоминания

В MCP-сервере есть система задач — используй её для ВСЕХ задач, напоминаний и планов проверки.

**Инструменты:**

- `add_tasks` / `vk_add_tasks` — создать задачи (с промптом для будущей проверки)
- `get_tasks` / `vk_get_tasks` — получить список задач
- `complete_task` / `vk_complete_task` — отметить выполненной/отклонённой

**Правила:**

1. Когда пользователь просит «напомни», «создай задачу», «проверить через N дней», «поставь на контроль» — ВСЕГДА используй `add_tasks` / `vk_add_tasks`. НЕ используй файлы, заметки, память или другие способы.
2. В начале разговора проверь задачи через `get_tasks` (due: "overdue") — если есть просроченные, сообщи пользователю.
3. После создания/изменения рекламной кампании — предложи создать задачу на проверку (но не создавай без подтверждения).
4. Каждая задача содержит `prompt` — готовую инструкцию для ИИ. Для выполнения задачи используй промпт, затем отметь через `complete_task`.
