# Дефолты РСЯ-кампаний (ветка R)

> Справочник приоритетных настроек для ветки **R (РСЯ)** скилла leadgen.
> Основано на анализе 395 активных РСЯ-кампаний агентства Этажи (апр 2026) + рыночных рекомендациях.
> Загружается в контексте: R1 (параметры), R2 (конфиг), R7 (создание через API).

---

## 1. Тип и размещение

| Параметр | Значение API | Комментарий |
|---|---|---|
| Тип кампании | ЕПК (`UNIFIED_CAMPAIGN`) | С мая 2024 все новые кампании Директа — ЕПК (и Поиск, и РСЯ). API в режиме совместимости создаёт ЕПК даже из payload `TextCampaign`. См. `METRIKA-ADS-RULES.md` → «Тип кампании: всегда ЕПК». В UI: «Единая перфоманс-кампания». |
| Стратегия поиска | `SERVING_OFF` | Поиск выключен — РСЯ-кампании чистые. Если нужен гибрид — пользоваться веткой C (поиск). |
| Стратегия сетей | `WB_MAXIMUM_CONVERSION_RATE` (старт) → `AVERAGE_CPA` (после обучения) | См. раздел 3. |
| `network_strategy` | активируется через сам API-вызов | Противоположно поисковой ветке. |

## 2. Бюджеты (консервативные, без запаса «на всякий случай»)

**Принцип:** автостратегии Директа стабильно обучаются при **~10 конверсий/нед**. Бюджет = ожидаемый CPA × количество конверсий.

### Таблица дефолтов (недельный бюджет)

| Уровень | РСЯ-бюджет/нед | Мин. конверсий/нед | Когда выбирать |
|---|---:|---:|---|
| **Микро-тест** | 3 000–5 000 ₽ | 5–8 | Новый город, неизвестный CPA, гипотеза |
| **Стандарт** | 5 000–10 000 ₽ | 10–15 | Типовой региональный город (Омск, Курган, Тюмень) |
| **Расширенный** | 15 000–25 000 ₽ | 20+ | Крупный город (Казань, Краснодар, Уфа) |
| **Премиум** | 40 000–80 000 ₽ | 40+ | Москва, СПб, премиум-ЖК |

### Формула для первой кампании

```
weekly_budget = целевой_CPA × 10 × 1.2   # коэффициент 1.2 — страховой запас на обучение
округлить до ближайших 500 ₽
```

Пример Омск-вторичка: целевой CPA ≈ 716 ₽ (факт из PROJECTS.md) → 716 × 10 × 1.2 ≈ **8 600 ₽/нед** → округляем до **8 500 ₽/нед**.

### Правила изменения бюджета

- **Старт** — по таблице выше.
- **Рост больше чем на 30% за раз** — только с явного подтверждения пользователя, желательно в чт-пт (меньше риска перекрутить обучение).
- **Снижение** — допустимо в любой момент, если CPA > порога CRITICAL из PROJECTS.md.

## 3. Стратегии: дерево решений

```
┌────────────────────────────────────────────────────────┐
│  История конверсий по поиску в этом городе/тематике?  │
└──────────────────────┬─────────────────────────────────┘
                       │
         ┌─────────────┴─────────────┐
         │                           │
      НЕТ                           ДА
         │                           │
         ▼                           ▼
  ┌────────────────┐       ┌─────────────────────┐
  │ Старт без      │       │ Есть 10+ конв/нед   │
  │ целевой CPA:   │       │ стабильно 2+ недели?│
  │ WB_MAX_CONV    │       └──────┬──────────────┘
  │ + BidCeiling   │              │
  │ + PriorityGoal │       ┌──────┴──────┐
  └────────────────┘      НЕТ           ДА
                           │             │
                           ▼             ▼
                  ┌────────────────┐ ┌──────────────┐
                  │ WB_MAX_CONV    │ │ AVERAGE_CPA  │
                  │ + BidCeiling   │ │ с tCPA=факт  │
                  │ + PriorityGoal │ │ × 0.9        │
                  └────────────────┘ └──────────────┘

Имиджевая / охватная (редко)  →  WB_MAXIMUM_CLICKS
```

### Коэффициент CPA для РСЯ

**Важно:** CPA в РСЯ обычно **ниже поискового на 20–40%** (холодная аудитория, но релевантные таргеты). Используем эмпирический коэффициент:

```
tCPA_РСЯ = CPA_поиска_факт × 0.7
```

**Если истории поиска нет** (новый город/тематика): старт с `WB_MAXIMUM_CONVERSION_RATE` без явного tCPA, через 2 недели при 10+ конверсий/нед — перевод на `AVERAGE_CPA`.

### BidCeiling (максимальная ставка клика) — отдельная сущность, НЕ tCPA

⚠️ **ВАЖНО.** BidCeiling — это потолок **цены клика**, не CPA конверсии. Старая формула `BidCeiling = tCPA × 1.5` была ошибкой (см. `lessons_registry.md` запись о 2026-04-29) — она путает клик и конверсию и даёт ставки в **5–15 раз выше реальных рыночных**. Реальный клик = CPA × CR (conversion rate), для РСЯ-недвижимости CR обычно 3–7%.

**Правильная формула:**

```
expected_click = tCPA × CR_estimate
BidCeiling     = expected_click × ceiling_factor
```

| Параметр | Значение по умолчанию |
|---|---|
| `CR_estimate` (РСЯ недвижимость) | `0.05` (5% — медианная конверсия из клика в форму) |
| `ceiling_factor` (страховой запас сверху) | `2.5` |
| **Итог** | **`BidCeiling ≈ tCPA × 0.125`** (~12.5% от tCPA) |

**Калибровка по tier-у** (минимум, ниже которого автостратегия теряет показы):

| Tier | Минимальный BidCeiling | Хард-кап (макс) |
|---|---:|---:|
| tier_1 (Москва, СПб, Казань…) | 80 ₽ | `tCPA × 0.30` |
| tier_2 (Тюмень, Уфа, Омск…) | 40 ₽ | `tCPA × 0.30` |
| tier_3 (малые города) | 20 ₽ | `tCPA × 0.30` |

```
BidCeiling_final = clamp(tCPA × 0.125, tier_min, tCPA × 0.30)
```

**Если фактический CR известен** (счётчик связан с Директом, есть история кликов и конверсий по форме за 30 дней) — используй его вместо дефолта 0.05.

**Анти-паттерны (НИКОГДА):**

- ❌ `BidCeiling = tCPA × 1.5` (старая ошибочная формула — путает клик с конверсией).
- ❌ `BidCeiling = CallCPA × что-то` — звонки в РСЯ редкость, форма доминирует; брать **только** form CPA как базу.
- ❌ `BidCeiling > tCPA × 0.30` — означает, что одна заявка должна окупиться меньше чем за 4 клика. Нереалистично.
- ❌ Создавать кампанию без `network_bid_ceiling` — без потолка автостратегия уйдёт в 500+ ₽/клик в регионах и 2000+ ₽/клик в Москве.

**Примеры расчёта (правильно):**

| Город | Tier | tCPA РСЯ | BidCeiling (×0.125) | Финал (после clamp) |
|---|---|---:|---:|---:|
| Омск | tier_2 | 2 500 ₽ | 312 ₽ | 312 ₽ |
| Москва | tier_1 | 4 200 ₽ | 525 ₽ | 525 ₽ |
| Курган | tier_3 | 1 800 ₽ | 225 ₽ | 225 ₽ |
| Курган (низкий tCPA) | tier_3 | 1 200 ₽ | 150 ₽ | 150 ₽ (клик-флор 20 не активен, всё ок) |

### Sanity-check бенчмарка перед использованием как target

`get_conversion_values` возвращает `BenchmarkFormCPA` (факт по аккаунту) и `TargetFormCPA` (= benchmark, если override не задан). Перед тем как взять benchmark как target — обязательно сравни с network-fallback того же tier-а:

```
ratio = BenchmarkFormCPA / NetworkExpected[theme/tier]
```

Network-ожидание для тематики/тира уже захардкожено в MCP (`server/data/network_benchmarks.json`); для вторички tier_1 = 3 500 ₽, tier_2 = 2 500 ₽, tier_3 = 1 800 ₽.

**Правило выбора target — детерминированное, без пауз даже в auto-режиме:**

| ratio | target_form_cpa | Логика |
|---|---|---|
| < 0.7 | `BenchmarkFormCPA` | benchmark ниже ожидаемого — норма (хорошая работа аккаунта). |
| 0.7 — 1.3 | `BenchmarkFormCPA` | Норма, аккаунт в рамках tier-а. |
| 1.3 — 2.0 | **`NetworkExpected`** | Аккаунтный benchmark задран — ориентир на норму tier-а. Не наследуем дорогие исторические периоды в новую кампанию. |
| > 2.0 | `NetworkExpected × 0.9` | Сильная аномалия (учётка с высоким спендом без атрибуции / тестовый период / неполные данные). Стартуем чуть ниже нормы, чтобы автостратегия не пыталась тянуть исторический потолок. |

**Почему так в auto-режиме:**
- Ранее правило заставляло агента приостанавливаться при ratio > 1.3 и спрашивать пользователя. Это ломает автономную работу (ночные/массовые прогоны, сторонние оркестраторы), которую пользователь явно запрашивает фразами «делай сам без вмешательства», «без модерации».
- Теперь правило детерминированное: при аномальном benchmark target не «угадывается», а прижимается к норме tier-а. Это безопасно (бюджет защищён), предсказуемо и автоматизируемо.
- В режиме `review` агент всё равно показывает расчёт перед `apply` — пользователь может перебить через `target_form_cpa_override`.

**Фиксация решения:** в файле кампании (раздел «Решения по target») всегда отрази значения `BenchmarkFormCPA`, `NetworkExpected`, `ratio`, выбранный бакет и итоговый `target_form_cpa`. Это нужно при пост-старт-аудите.

### Параметры создания для API

Для старта новой РСЯ (без истории) минимум:

```json
{
  "search_strategy": "SERVING_OFF",
  "network_strategy": "WB_MAXIMUM_CONVERSION_RATE",
  "network_bid_ceiling": <BidCeiling из формулы выше>,
  "priority_goals": "[{\"goal_id\":<form_goal>,\"value\":<target_form_cpa>},{\"goal_id\":<call_goal>,\"value\":<target_call_cpa>}]",
  "daily_budget_amount": <weekly_rub>,
  "counter_ids": "<counter_id>"
}
```

⚠️ `priority_goals.value` = **target CPA** для соответствующей цели (форма / звонок). **НЕ** переписывать туда click-ceiling.

**BidCeiling** — задаётся параметрами `network_bid_ceiling` / `search_bid_ceiling` в `add_campaign` и `update_campaign`. В рублях, внутренне приводится к микроюнитам, минимум 0.3 ₽. Применим к `WB_MAXIMUM_CONVERSION_RATE`, `AVERAGE_CPA`, `WB_MAXIMUM_CLICKS` (неприменим к `SERVING_OFF` и ручным).

Правило для РСЯ: **всегда задаём `network_bid_ceiling` при создании** — страховка от перекрута ставок на первых аукционах обучения. Без потолка автостратегия может уйти в 500+ ₽/клик в регионах и 2000+ ₽/клик в Москве. Но и не задирать выше `tCPA × 0.30` — см. предыдущий раздел.

Чтобы изменить только BidCeiling у существующей РК через `update_campaign` — обязательно продублируй `network_strategy` в вызове (API принимает Network-блок целиком, не партиально).

## 4. UTM (tracking_params) — обязательно

**Нарушение в текущей проде:** 40% РСЯ-кампаний работают без UTM. В новой ветке R это недопустимо.

- Получаем готовую строку через `get_utm_config(city, theme, placement="рся")`.
- Ставим на уровне **кампании** (наследуется группами) — поле `tracking_params` в `add_campaign`.
- НЕ на уровне группы (исключение: нужен override для A/B теста).
- После создания — `get_campaigns` с `text_campaign_field_names=TrackingParams` для верификации.

## 5. Расширенный геотаргетинг

⚠️ **Дефолт — выключено (`false`).** Опция `ENABLE_AREA_OF_INTEREST_TARGETING` показывает рекламу пользователям, которые СЕЙЧАС не в целевом регионе, но интересуются им. Для лидогенерации недвижимости это в основном утечка бюджета: житель Москвы, однажды искавший «квартира Омск», получает показ — конверсия по такому показу крайне маловероятна.

Включаем явно (`true`) ТОЛЬКО если пользователь явно просит «ловить приезжих / командировочных» или для очень узких сценариев (премиум-ЖК для приезжих топ-менеджеров и т. п.).

```
ENABLE_AREA_OF_INTEREST_TARGETING:
  ├── Региональная РСЯ (Омск, Казань, Курган и т.п.)  → NO (по умолчанию)
  ├── Районная / точечная (Москва-Раменки, СПб-Парнас) → NO
  └── Федеральная / РФ без города                      → NO
```

Логика: дефолт OFF, потому что в недвижимости показ житель Москвы по запросу «квартира Омск» — почти всегда нецелевой, конверсия околонулевая.

Задаётся через `settings` в `add_campaign`:

```json
[{"option":"ENABLE_AREA_OF_INTEREST_TARGETING","value":false}]
```

Включи `value: true` ТОЛЬКО при явном запросе пользователя на «приезжих / командировочных».

## 6. Счётчики Метрики и цели

- `counter_ids` обязательно — подключаем городской счётчик через `get_city_config` / `get_conversion_values`.
- `priority_goals` — две цели (форма + звонок), с ценностями из `get_conversion_values(...)`.
- НЕ использовать `goal_id=13` (системная «любое целевое действие» Директа) — только явные цели Метрики.

## 7. Структура групп (обязательный минимум)

Для РСЯ кластеризация идёт **по аудитории** (не по интенту, как в поиске):

| Группа (обязательная) | Что внутри | Примечание |
|---|---|---|
| **LAL** | Ключевые + look-alike сегмент на 3–5% по текущим конвертерам | Требует аудиторный таргет (`add_audience_targets`). Опция — см. раздел 8. |
| **Интерес-премиум** | Ключевые премиум-сегмента + интересы (для Москвы/СПб) или общие интересы (регионы) | Только если тематика позволяет сегментацию |
| **Общие-купить** | Широкие целевые ключи + автотаргетинг EXACT+ALTERNATIVE | База кампании |
| **По ЖК / по сегменту** | 1-комн / 2-комн / конкретный ЖК / котлован / вторичка-класс | Если есть посадочные под них |
| **Ретаргет-посетителей** | Сегмент Метрики «был на сайте, не оставил заявку» | Опция — при наличии истории трафика |

**Минимум для первой кампании:** `Общие-купить` + 1–2 сегментные группы.

### Правило наполнения группы

- **Количество объявлений:** 5–15 для стандартной группы; до **45** для LAL (питает ML-стратегию).
- **Автотаргетинг:** `EXACT=YES, ALTERNATIVE=YES`, остальные выключены.
- **Tracking params** — на уровне кампании, не группы.

## 8. Ретаргетинг и LAL (опция)

**Обязательно?** Нет. Ретаргет/LAL включаем если:

- Есть накопленный трафик на сайте (> 5 000 визитов/мес).
- Счётчик Метрики связан с Директом.
- Пользователь явно попросил расширенную кампанию («премиум», «полная», «LAL обязательно»).

Иначе — стартуем без ретаргета, добавляем в подветке **OR.3** (Расширение аудиторных таргетов) после накопления данных.

### Типы аудиторных таргетов для РСЯ

- **LAL** — похожие аудитории на базе конвертеров (требует счётчик с 1000+ конверсий).
- **Ретаргетинг** — посетители сайта без заявки, окно 30–60 дней.
- **Интересы Директа** — «Недвижимость», «Жильё», «Ипотека».
- **Ключевые** — базовая семантика из R4.

## 9. Минус-слова (РСЯ отличаются от поиска)

**Общие для всех РСЯ** (ставим на кампании через `negative_keywords`):

```
!отделка, !ремонт, !своими руками, !как выбрать, !как купить,
```

## 10. Площадки (blocked_placements)

⚠️ **Дефолт — применяем стандартный чёрный список Этажи (~400 площадок) сразу при создании.** Раньше было «при создании пусто, чистка через 3 дня» — это дарило бюджет мусорным площадкам (мобильные игры, пиратские приложения, развлекательные домены) в самые дорогие дни обучения автостратегии.

- **R7.5 (новый шаг):** сразу после `add_campaign` вызвать `apply_blocked_placements(client_login, campaign_id)` — он применит ~400 площадок Этажи через `excludedsites.set` за один вызов.
- Через **3 дня после запуска** + далее **раз в 2 недели** — дополнительная чистка через `get_custom_report` с dimension `Placement` → фильтр площадок с расходом без конверсий (> 500 ₽ без единой конверсии) → дозаблокировать через `set_excluded_sites` (с объединённым списком: дефолт + новые).
- Шаг OR.1 в ветке оптимизации.

## 11. Видео-генерация (auto, server-side)

Yandex Direct автоматически создаёт видео-креативы из текстов и загруженных картинок объявления — через свой ИИ-видеогенератор на серверной стороне. Происходит после старта показов, обычно в течение нескольких часов / суток. **Явного API-toggle для включения/выключения этой опции нет** (проверено по API v5: ни в `Campaigns.Settings`, ни в `AdGroupSettings`).

Что делает скилл:
- Не трогаем ничего по части видео при создании — генерация включится автоматически после активации кампании пользователем.
- Если у объявлений есть валидные картинки 1:1 и 16:9 (что наш `add_ad_image` валидирует локально перед загрузкой) — Yandex использует обе пропорции при сборке видео.
- Если пользователь жалуется «видео не появилось через сутки» — проверить через `get_creatives` / `creatives` MCP-инструмент, что видео-креативы сгенерировались. Если нет — обычно причина в формате/контенте картинок.

## 12. Чеклист валидации перед apply

Прогонять перед любым `add_campaign` в ветке R:

- [ ] `search_strategy = SERVING_OFF`
