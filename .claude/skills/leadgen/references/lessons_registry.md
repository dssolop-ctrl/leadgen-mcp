# Lessons Registry

## API gotchas (Яндекс Директ)

### −1. Автопилот: ownership через state.yaml, internal-метки запрещены. (2026-05-18)

**Инцидент.** На пилоте Новосибирск-ии (РК `709914995`, `709915110`) автопилот по старой логике поставил на ad-уровне служебные метки: `autopilot:managed`, `city:novosibirsk-ai`, `topic:vtorichka`, `channel:search` / `channel:rsya` плюс бизнес-метки Этажей. Эти служебные метки попали в каталог Этажей и в отчёт Метрики как **отдельные «направления»**, в результате чего одна и та же кампания одновременно учитывалась в нескольких категориях, и агрегация statistics ломалась. Дополнительно — `topic:<>` и `channel:<>` дублировали бизнес-метки (`Вторичка` ↔ `topic:vtorichka`, `РСЯ` ↔ `channel:rsya`), засоряя фильтры в UI Директа (lesson #32 — частный случай этого же).

**Причины.**
- Раннее проектирование ownership-предиката шло «через метки в Директе»: `add_labels(campaign_id, ["autopilot:managed", "city:<>"])` ставился при adopt / create_draft, по нему же фильтровался reconcile.
- Метки `topic:<>`/`channel:<>` ставились «для удобства интроспекции», хотя обе величины уже выводимы из `BiddingStrategy` (channel) и имени кампании / бизнес-меток (topic).
- Каталог меток Этажей — общий и плоский: любая поставленная метка появляется в Метрике/отчётах Этажей как направление. Internal-метки таким образом обязательно ломают агрегацию.

**Правильно.** Источник правды ownership — **`state.yaml.campaigns[]`** (per-city, operational), **не метки в Директе**.

| Где числится | Ownership |
|---|---|
| `state.yaml.campaigns[<id>].ownership == "managed"` | `managed` — бот свободно действует |
| `state.yaml.campaigns[<id>].ownership == "released"` | `released` — read-only, история |
| `city.yaml.holdout.campaign_ids[]` содержит `<id>` | `holdout` — read-only forever |
| Кампания в API, нет ни в state, ни в holdout | `foreign` — read-only |

- Adoption / создание DRAFT — это **запись в `state.yaml`**, не `add_labels` на стороне Директа.
- Release — `state.yaml.campaigns[<id>].ownership = released`.
- `topic` / `channel` хранятся в `state.yaml.campaigns[<id>]` (operational). При adoption выводятся: `channel` из `BiddingStrategy` (Search/Network OFF), `topic` из бизнес-меток через mapping в `leadgen/config/labels.md` + из конвенции имени `<Город> | <Канал> | <Тематика> | …`.
- Бизнес-метки Этажей (`Лидген`, тематика, направление, канал) автопилот **читает** для mapping и ставит при `campaign.create_draft.*`, но **никогда** не модифицирует существующие на adopted/foreign-кампаниях.
- Чистка устаревших internal-меток на existing-кампаниях — действие `campaign.cleanup_internal_labels` (использует MCP `remove_labels`), однократный проход на adopt + однократный cleanup по городу.

**Как применять:**
- В любых ветках автопилота (`branches/*`) и action_catalog — **никогда** не вызывать `add_labels` / `set_banner_labels` с `autopilot:*` / `city:*` / `topic:*` / `channel:*`.
- Перед apply любого action с `entity_type=campaign` — резолвить ownership через state + holdout, не через `get_labels`.
- При обнаружении на adopted-кампании старых internal-меток (флаг `extra_internal_labels_pending_cleanup`) — поставить в очередь `campaign.cleanup_internal_labels`.
- Связанный lesson #32 (не дублировать topic/channel) — теперь поглощён этим: правило сильнее (запрещены **все** internal-метки, не только дубликаты).

### 0. РСЯ: BidCeiling — это клик, не CPA. И группы — после семантики, не до. (2026-04-29)

**Инцидент.** Тестовый прогон создания РСЯ-кампании в Омске (вторичка) показал три проблемы одновременно: (1) три группы (Общие + 1-комн + 2-комн) появились по упоминанию пользователя, без проверки семантики и посадочных; (2) `network_bid_ceiling ≈ 1800 ₽` — нереально высокий потолок цены клика; (3) target form CPA ≈ 6000 ₽ — взят из benchmark без sanity-check vs network ожидание (3500 ₽ для tier_1 вторички).

**Причины.**
- Старая формула в `rsya_defaults.md`: `BidCeiling = tCPA × 1.5`. Концептуально неверно — путает потолок клика с целью конверсии. tCPA = 1871 (звонок) × 0.7 × 1.5 = 1965 → ≈1800. Реальный клик-флор для Омск-вторички — 30–100 ₽.
- В `create-rsya.md` R5 минимум групп описывался как «Общие + 1 сегментная», что агент трактовал как «можно делать сегментные если пользователь сказал». Не было хард-правила «семантика ДО решения о группах + проверка посадочной».
- `get_conversion_values` возвращает `BenchmarkFormCPA` (факт) и `TargetFormCPA = benchmark` без override. Скилл не делал sanity-check vs `network_benchmarks.json` (где для вторички tier_1 = 3500 ₽).
- В R7 было прописано `add_ad × 3` — ниже минимума обучения автостратегии РСЯ (нужно ≥5).

**Правильно.**
- BidCeiling формула: `tCPA_form × CR_estimate × ceiling_factor` ≈ `tCPA × 0.05 × 2.5` = `tCPA × 0.125`, с clamp по tier-флору и хард-капом `tCPA × 0.30`. Полностью описано в `references/rsya_defaults.md` раздел «BidCeiling (максимальная ставка клика)».
- Группы ТОЛЬКО после R4 (семантика собрана) + R5 (проверка ≥3 фраз и наличия посадочной для каждого упомянутого сегмента). Если сегмент не проходит — сворачивается в Общие-купить.
- В R2.5 — обязательный sanity-check `BenchmarkFormCPA / NetworkExpected[theme/tier]` с **детерминированным выбором target** (без пауз даже в auto): ratio < 1.3 → benchmark; ratio 1.3–2.0 → `NetworkExpected`; ratio > 2.0 → `NetworkExpected × 0.9`. (Поправка 2026-04-29: ранее в этом блоке стояла пауза «спросить пользователя» при ratio > 1.3 — она ломала автономную работу. Сейчас правило детерминированное, в режиме `review` пользователь может перебить через `target_form_cpa_override`.)
- В R5 группу под сегмент создаём при ≥3 фразах в R4. Наличие посадочной — мягкий сигнал (использовать как `href`), не блокирующий: если посадочной нет — fallback на базовую тематическую (`/realty/` для вторички), группу всё равно создаём.
- `add_ad × 5–8` для стандартных групп; до 45 для LAL.

**Как применять:** на каждом R7-вызове `add_campaign` для РСЯ — проверить, что `network_bid_ceiling ≤ tCPA × 0.30` и что N групп обоснован данными из R4/R5 (не репликой пользователя). После создания — `summarize_campaign_snapshot(campaign_id)` для read-back.

### 1. `ads.add` ≠ `ads.get` по расширениям

`ads.get` возвращает `TextAd.AdExtensions` (объекты), но `ads.add` принимает только `TextAd.AdExtensionIds`.

**Почему:** асимметрия API между чтением и записью. Любой клон-скрипт обязан конвертировать `AdExtensions → AdExtensionIds`.
**Как применять:** при копировании объявления между группами/кампаниями.

### 2. `update_ad` вместо `delete + add`

Для изменения текста, заголовков, ссылок — всегда `update_ad` (partial update).

**Почему:** удаление объявления уничтожает `sitelink_set_id`, `ad_extension_ids`, `ad_image_hash`, видеодополнения, цены и другие ручные настройки.
**Как применять:** при ротации креативов, исправлении текстов, смене посадочной.

### 3. `get_campaigns` без `states` = мусор

Без фильтра API возвращает все кампании (архивные, завершённые, выключенные).

**Почему:** дефолт бесполезен для оперативной работы; контекст засоряется неактуальными записями.
**Как применять:** для "покажи кампании" — всегда `states: ["ON"]`.

### 4. Тип-специфичные FieldNames в `get_campaigns`

Вложенные поля (`BiddingStrategy`, `PlacementTypes`, `TrackingParams`) НЕ возвращаются через обычный `FieldNames` — нужны `text_campaign_field_names` / `dynamic_text_campaign_field_names` / `unified_campaign_field_names`.

**Почему:** API разделяет общие поля кампании и тип-специфичные.
**Как применять:** чтобы получить стратегию/places/trackingparams — пробрасывай нужный `*_campaign_field_names`.

### 5. Бюджет — недельный и в рублях

MCP-параметр `daily_budget_amount` — на самом деле недельный бюджет в рублях.

**Почему:** API требует микроюнитов, но MCP-сервер сам конвертирует; путаница с dailynaming.
**Как применять:** передавать число как есть (например, 1000 для 1000 руб/нед). НЕ умножать на 1 000 000.

### 6. `attribution: "LYDC"` — обязательно для отчётов

Дефолт API — LC. Данные между чатами будут несовместимы, если не фиксировать атрибуцию.

**Почему:** отчёты с разной атрибуцией нельзя сравнивать.
**Как применять:** в `get_campaign_stats` и `get_search_queries` всегда `attribution: "LYDC"`.

### 7. UTM-метки — через `tracking_params` группы

Ставить на уровне группы, а не кампании.

**Почему:** работает для основной ссылки + быстрых ссылок; на уровне кампании не всегда пробрасывается.
**Как применять:** после `add_adgroup` — `update_adgroup(adgroup_id, tracking_params=...)`.

### 8. После `resume_campaign` — обязательный read-back

Только `ResumeResults: success` не гарантирует показов.

**Почему:** кампания может остаться в `State=OFF` из-за `StatusPayment`, `StatusClarification`, `StatusModerated`.
**Как применять:** после `resume_campaign` — `get_campaigns` с проверкой `State`, `Status`, `StatusPayment`, `StatusClarification`. Если `State=OFF` — назвать точный блокер, не писать «запустил».

### 9. `add_labels` работает на объявления, не на кампании

Метки назначаются на `banner_ids`, отображаются на уровне групп.

**Почему:** API v4 Live, а не v5.
**Как применять:** после создания всех объявлений всех кластеров — один `add_labels` на весь список banner_ids.

### 10. AVERAGE_CPA без 10+ конверсий/нед — контрпродуктивно

Стратегия либо режет показы (мало трафика → замкнутый круг), либо превышает CPA в разы.

**Почему:** документация Яндекса прямо требует 10+ конверсий/нед для обучения.
**Как применять:** старт — `WB_MAXIMUM_CONVERSION_RATE`, переход на `AVERAGE_CPA` только после 2-3 стабильных недель.

### 11. Изменения > 30% бюджета — согласование + чт-пт

Резкие изменения вызывают «разведку» стратегии.

**Почему:** в понедельник разведка съедает недельный бюджет. В чт-пт попадает на начало нового цикла.
**Как применять:** любое изменение >30% — подтверждение пользователя, ставить задачу на чт-пт.

### 12. `get_*` для массового аудита не безлимитный

`SelectionCriteria.CampaignIds` в `get_ads`/`get_adgroups` имеет лимит.

**Почему:** API возвращает `4001: Превышено допустимое количество идентификаторов`.
**Как применять:** для cabinet-wide audit — батчить CampaignIds (по 10-50 за раз).

## Баланс и квоты

### 13. Баланс недоступен через API v5

Через MCP `get_account_balance` — только для агентского аккаунта, не для client-login.

**Почему:** API v5 не предоставляет Finance endpoint для client-сессии.
**Как применять:** баланс по конкретному городу — только через интерфейс Директа или Finance API v4 (другой токен).

### 14. Wordstat — дневная квота

`wordstat_*` инструменты имеют дневной лимит.

**Почему:** официальное API Wordstat квотируется по app.
**Как применять:** перед тяжёлой серией сборов — `wordstat_user_info` для остатка лимита. При L1→L2 waves батчи по 10 фраз.

## Объявления и структура

### 15. Кроссгрупповые дубли копирайта — баг

Тексты объявлений должны соответствовать ключам своей группы.

**Почему:** Яндекс в РСЯ учитывает релевантность текста группе; дубль снижает CTR и качество.
**Как применять:** при создании — индивидуальные заголовки для каждого кластера (гео, интент).

### 16. Группы обязаны быть семантически чистыми

Смешение интентов (купить + продать, покупатель + продавец) в одной группе — баг.

**Почему:** невозможно написать один текст, релевантный разным интентам; CTR падает.
**Как применять:** разделять кластеры перед написанием объявлений, не во время.

### 17. Креативный долг нельзя прятать за A/B-тестом

Если в группе крутятся почти одинаковые тексты — это не A/B, а долг.

**Почему:** настоящий A/B требует осмысленного отличия (angle / promise / CTA / pain).
**Как применять:** до роста ставок или broad-экспансии — отдельный replacement/test-план.

## Scope и планирование

### 18. Priority-РК ≠ разрешение игнорировать остальные

Если пользователь просит «оптимизировать кабинет» и отдельно называет «усилить кампанию X», это приоритет очереди, а не сужение scope.

**Почему:** account-wide оптимизация требует обзора всех РК; ложное сужение маскирует проблемы.
**Как применять:** default scope = все активные РК канала; приоритеты — внутри очереди.

### 19. Plan-only ≠ default mode для оптимизации

Если пользователь просит «оптимизировать / исправить / внедрить» — default = план + apply + self-check.

**Почему:** остановка на review-pack без blocker-а = незавершённая работа.
**Как применять:** если не пошёл в apply — явно назвать blocker (manual gate / запрет live / нет данных).

## Источники правды

### 20. Live API > старая markdown-документация

Если live-data конфликтует с локальными docs — верить live.

**Почему:** кабинет меняется независимо от документации; устаревшие docs ведут к ошибочным решениям.
**Как применять:** перед любым planning — свежий `get_campaigns` / `get_campaign_stats`.

### 21. Файл кампании ≠ source of truth для фактов

В `campaigns/<slug>.md` — гипотезы, история обсуждений, решения. Фактические изменения — в БД истории через MCP.

**Почему:** файл кампании редактируется вручную, может расходиться с реальностью; БД истории — append-only.
**Как применять:** проверка «что реально менялось» — через `get_change_history(campaign_id)`.

## R6.5 — генерация изображений

### 22. Flux на OpenRouter игнорирует image_size

**Симптом (тест 2026-05-05):** запрос `aspect_ratio=1:1` или `16:9` с `model="black-forest-labs/flux.2-pro"` возвращает 1024×768 (4:3) независимо от параметра. Все 4 картинки в наборе — одного размера 4:3.

**Причина:** в OpenRouter chat-completions API параметр `image_size` для Flux трактуется как «hint, можно игнорировать». Flux 2 PRO выдаёт нативные пропорции своего пайплайна, ≈4:3 по умолчанию.

**Как применять:**
- Дефолтная модель R6.5 = `google/gemini-3-pro-image-preview` (с 2026-05). Gemini уважает aspect лучше, хотя тоже промахивается на 1:1 в ≈30% случаев.
- **Серверный auto-crop** в MCP `generate_image` / `generate_banner_set` гарантирует точный aspect: если модель вернула не тот ratio — сервер сам кропает по центру до запрошенного. В ответе появляется поле `auto_cropped: {from_w, from_h, to_w, to_h}`. Это нормальный happy path, не ошибка.
- `aspect_warning` в ответе — теперь означает «crop тоже не сработал» (редкий edge case → regen).
- Flux можно использовать осознанно для отдельных стилевых сцен (`model="black-forest-labs/flux.2-pro"`) — auto-crop спасёт результат, но cost/latency выше.

### 23. «Генерация» = 1 API-вызов, не 1 visual scene

**Симптом (тест 2026-05-05):** агент вызвал `generate_banner_set(aspect_ratios="1:1,16:9", n_variants=2)` — получил 4 картинки на одну сцену, израсходовал ¼ бюджета 20-лимита впустую.

**Причина:** размытое понимание «лимита 20 генераций». Агент думает «сцен = 20», а на деле API-вызовов = 20.

**Как применять:**
- 1 API-вызов = 1 PNG = 1 единица в лимите 20.
- `generate_banner_set(aspect_ratios=A, n_variants=V)` = `len(A) × V` API-вызовов.
- Дефолт для РСЯ: `aspect_ratios="1:1,16:9"`, `n_variants=1` → 2 картинки на сцену.
- Считай оценочно ДО запуска: `N_unique_scenes × 2 (форматы) ≤ 20`. С учётом регенераций — закладывай ≤ 16.
- Переиспользование scene между группами (через `AdImageHash`) НЕ тратит лимит дважды — это всё ещё одна генерация.

### 25. Yandex Direct ratio-checker строже заявленных ±2%

**Что мы знаем о мире:** API Директа отклоняет картинки с ratio, отстоящим даже на 0.8% от target (16:9 фактический 1.792 vs целевой 1.778 → ошибка 5004 «Размер изображения некорректен»). Допуск близок к ±0.5% или требует точного значения.

**Как с этим жить:**
- Серверный auto-crop в `generate_image` / `generate_banner_set` (Go-код) уже даёт **точные** ratio: 1:1 = квадрат, 16:9 = 1024×576 (ровно 1.7778), 4:3, 3:2 — точно. После crop локальный PIL-ресайз не нужен.
- Если по каким-то причинам передаёшь свою картинку без auto-crop — приведи к точным размерам сам, не полагайся на ±2% допуск.

### 26. GoalId=13 в read-back — нормальное поведение

**Симптом:** после создания кампании `get_campaigns` показывает `GoalId: 13` в `WbMaximumConversionRate`, не передаваемые `priority_goals`.

**Причина:** при передаче нескольких целей API ставит `GoalId=13` (системная цель «любое целевое действие»), а `priority_goals` хранятся отдельно и не возвращаются без `field_names="..., PriorityGoals"`.

**Как применять:**
- Это нормальное поведение, не ошибка. При read-back для проверки `priority_goals` — добавить `unified_campaign_field_names="PriorityGoals"` или `text_campaign_field_names="PriorityGoals"`.

### 24. R5 без закрытого R4 = регресс

**Симптом (повторяющийся):** агент формирует группы по сегментам, упомянутым пользователем («1-комн», «2-комн»), не вызывая `check_search_volume` для семантики. Получаются группы из 3-4 фраз «из головы», статистика wordstat игнорируется.

**Причина:** в `branches/create-rsya.md` шаг R5 идёт сразу после R4, и текстовое предупреждение «R4 → R5, не наоборот» не воспринимается как блокирующий гейт. Агент видит сегменты в запросе пользователя — и строит группы, минуя R4.

**Как применять:**
- В R5 теперь HARD GATE-таблица: `check_search_volume` вызван, ≥12 фраз с Shows≥100, фразы привязаны к сегментам через wordstat-данные. Без этого R5 не открывается.
- В R4 чек-лист закрытия (`check_search_volume` ✓, ≥12 фраз ✓, таблица фраз ✓, `get_negative_keyword_guidance` ✓) — обязан быть в сообщении пользователю **перед** переходом в R5.
- Если агент уже в R5 без закрытого R4 — `STOP`, вернуться, не пытаться «достроить».

### 27. Объектные группы по комнатности/типу — hard rule, не опция

**Симптом (2026-05-15, прогон novosibirsk-ai-2026-05-15-1327):** автопилот создал search-кампанию по vtorichka для Новосибирска. В группу «Общий» свалил 9 ключей разной комнатности (купить квартиру, однушку, двушку, трёшку, однокомнатную, двухкомнатную, трёхкомнатную, студию, вторичка) с одним общим заголовком «Купить квартиру в Новосибирске» и одной общей посадочной `/realty/`. Из 5 групп ни одна не была объектной — только интент-кластеры (Общий, Ценовой, Финансовый, С городом, Готовое жильё). Это даёт низкий Quality Score, рост CPC, потерю CTR.

**Причина:** в `create-search.md` шаг 5 формулировка про объектные кластеры была мягкой («если кампания шире одного типа») и не воспринималась как блокер. Агент пропустил Раунд 2/3 расширения семантики через `check_search_volume` (артефакт «только Phrase без Shows» — известный баг), не запросил частотность отдельно по однушкам/двушкам/трёшкам/студиям, не разделил их на отдельные группы.

**Как применять:**
- В `create-search.md` шаг 5 — **HARD RULE**: если в семантике есть ≥1 фраза с типологическим маркером (комнатность для квартир, тип объекта для загородки/коммерческой) — обязательная отдельная группа объявлений. Запрещено объединять разные типы в одной группе.
- Заголовки и посадочные ОБЯЗАНЫ быть дифференцированы (`Купить однушку` → `/realty/?rooms=1`, не `Купить квартиру` → `/realty/`).
- Catch-all «Общий» получает в `negative_keywords` адгруппы все типологические маркеры, вынесенные в свои группы. Иначе каннибализация.
- Валидация перед C6: если ≥1 типологическая фраза в семантике, но нет соответствующей объектной группы → fail, перестроить кластеризацию.

### 28. add_ad для RSYA без ad_image_hash — фатальная ошибка, не warning

**Симптом (2026-05-15, прогон novosibirsk-ai-2026-05-15-1327):** автопилот создал RSYA-кампанию с 4 текстовыми объявлениями без `ad_image_hash`. API не вернул ошибку: TextAd без картинки — валидная сущность в Direct API. Однако в РСЯ инвентаре ≈80-90% слотов — баннерные (240×400, 728×90, 1080×607 и др.), там TextAd без картинки не отображается. Фактический охват кампании — 10-20% от потенциального. В Telegram-summary это было подано как «нюанс, добавим в следующем прогоне», а не как блокер «кампания не работает».

**Причина:** API не делает кросс-валидацию «тип кампании = RSYA → ad_image_hash обязателен». В `create-rsya.md` отсутствовал hard-gate, требующий проверки image_hash перед каждым `add_ad`. Финальный аудит после R7 (проверка всех ads через `get_ads` с полем AdImageHash) тоже не был обязательным шагом.

**Как применять:**
- В `create-rsya.md` R7 шаг e — **HARD RULE**: для каждого `add_ad` в RSYA-кампании `ad_image_hash` обязателен. Pre-flight assert перед вызовом.
- Шаг d (картинки) теперь — БЛОКЕР: если R6.5 не выполнен и нет AdImageHash, дальнейшие шаги недоступны.
- Новый шаг 5 в R7 — финальный аудит через `get_ads(..., text_ad_field_names="AdImageHash")`: если хоть один TextAd без AdImageHash → кампания не сдаётся пользователю как готовая.
- Лимит 20 генераций на кампанию учитывает: 2 visual scenes × 2 формата (1:1, 16:9) × 1 variant = 4 генерации — норма для минимальной RSYA-кампании.

### 29. manage_ads (suspend/archive) не работает для DRAFT-объявлений

**Симптом (2026-05-15, прогон novosibirsk-ai-2026-05-15-1327-fix):** попытка `manage_ads(action=archive)` и `manage_ads(action=suspend)` для DRAFT-объявлений (созданных в SUSPENDED-кампании, не прошедших модерацию) возвращает ошибку Code 8300 «Неверный статус у объекта: Объявление является черновиком и не может быть заархивировано/остановлено».

**Причина:** в API Direct массовые операции `ads.archive` / `ads.suspend` применимы только к объявлениям, которые УЖЕ были опубликованы (как минимум прошли модерацию). DRAFT-объявления находятся в собственном статусе «черновик» — для них недоступны переходы из активных состояний.

**Как применять:**
- **До активации кампании (DRAFT):** удалить ошибочные объявления невозможно через manage_ads. Варианты: (а) оставить как есть — DRAFT не показывается, пока кампания не активирована; (б) удалить через прямой API (нет MCP-инструмента сейчас); (в) пересоздать adgroup без них (тяжёлый путь).
- **После активации кампании:** проблемные ad можно архивировать через `manage_ads(action=archive)` — действует штатно.
- В `create-rsya.md` R7.5 финальный аудит RSYA: если найдены TextAd без AdImageHash на стадии DRAFT, **сделать пометку в campaign file и в next_action**: после активации кампании выполнить cleanup через `manage_ads(action=archive)`. До активации ничего не предпринимать (DRAFT всё равно не показывается).
- Если кампания уже активна и обнаружены text-only ad в RSYA — `manage_ads(action=archive, ad_ids=...)` работает сразу.

### 30. RSYA-визуалы: каждой adgroup свои сцены (default), переиспользование — exception

**Симптом (2026-05-15, прогон novosibirsk-ai-2026-05-15-1327-fix):** агент сгенерил 2 визуальные сцены (V1 интерьер + V3 фасад) и положил **те же сцены** в обе adgroup-ы RSYA-кампании vtorichka: «Общий» и «Финансовый». В Финансовой группе (ипотека, рассрочка, маткапитал) должны были быть финансовые визуалы — ключи в руках, семья с документами, расчёт на калькуляторе. Вместо этого получили V1 интерьер + V3 фасад в обеих группах, просто варьируя 1:1/16:9 форматы между ad. Лимит 20 генераций использован только на 5/20 — было 15 генераций запаса для уникальных сцен, но они не были использованы.

**Причина:** в `create-rsya.md` R6.5 раздел «Переиспользование» подталкивал к переиспользованию visual через `AdImageHash` ради экономии лимита — пример «V3 фасад → 3 группы». А раздел «Сколько надо» рекомендовал «1 visual на все 3 объявления группы (часто оправдано) — 2 генерации» и «без переиспользования» подавался как только-если-есть-бюджет. Эта формулировка задавала переиспользование как default. Семантическая релевантность (Финансовая группа = финансовый визуал) была подсознательно подавлена «экономией».

**Как применять:**
- В `create-rsya.md` R6.5 переписаны разделы «Сколько надо в типичной кампании» и «Переиспользование» — теперь default: **каждая adgroup получает свои уникальные visual scenes**.
- Переиспользование visual между группами — exception, только для рациональных оснований (LAL/ретаргет, сегментные группы того же интента, упор в 20-лимит при 5+ группах).
- Запрещено переиспользование между разными интентами (Общий ↔ Финансовый) — смазывает emotional anchor группы.
- Добавлена карта «Группа/интент → визуальные сцены»: Финансовый = V7 ключи + V_family, Загородка = V4 дом + V5 участок и т.д.
- Планирование начинается с верхнего числа уникальных сцен (по 2 на группу), потом корректируется если упор в 20-лимит.

### 31. RSYA: объектные сегментные группы по комнатности/типу — обязательны (тот же hard rule, что для search)

**Симптом (2026-05-15, прогон novosibirsk-ai-2026-05-15-1327-fix3):** в RSYA-кампании vtorichka все 7 типологических ключей (купить однушку, двушку, трёшку, однокомнатную, двухкомнатную, трёхкомнатную, и т.д.) лежали в одной группе «Общий» вместе с не-типологическими (купить квартиру, квартира новосибирск). Объектных групп Однушки/Двушки/Трёшки/Студии не было — как и в первой версии поисковой кампании, которую я закрывал в lesson #27. Lesson #27 был применён только к `create-search.md`, но не отражён в `create-rsya.md` — формально playbook не требовал объектной разбивки в RSYA.

**Причина:** lesson #27 был сформулирован в контексте поиска (где интент — точный, и группы по интенту/типу — основа структуры). В `create-rsya.md` R5 кластеризация декларировалась «по аудитории, не по интенту», и типологические маркеры подразумевали мягкое правило «≥3 фраз на сегмент». Это позволяло свернуть Однушки/Двушки/Трёшки в Общие-купить, теряя дифференциацию интентов и переиспользуя один V1 интерьер на все типы.

**Как применять:**
- В `create-rsya.md` R5 — добавлено HARD RULE на объектные сегментные группы: если в R4 есть ≥1 фраза с типологическим маркером (комнатность для квартир, тип объекта для загородки/коммерческой), эти фразы обязаны быть в отдельной adgroup. RSYA-порог =1 фраза (не ≥3, как у аудиторных сегментов): маркер типа — жёсткий сигнал интента, не аудиторная характеристика.
- В таблице типов групп: «Объектные сегментные» — отдельный тип с HARD-меткой, отличный от «Аудиторных сегментных» (премиум/котлован/ЖК) где ≥3 порог сохраняется.
- Каждая объектная группа получает **свой V1 интерьер с соответствующей планировкой** (студия / 1-комн / 2-комн / 3-комн). Cross-reuse V1 между разными комнатностями запрещён (комбинация lesson #30 + #31).
- Catch-all «Общие-купить» получает в negative_keywords все типологические маркеры, вынесенные в объектные группы.
- Это правило симметрично для search (lesson #27) и для RSYA (lesson #31). Любое обновление одного из них синхронизируется со вторым.

### 32. Автопилот: минимум internal-меток на campaign/ad (НЕ дублировать бизнес-метки)

**Симптом (2026-05-15, прогон novosibirsk-ai-2026-05-15-1327):** автопилот при `add_labels` ставил **7 меток** на search-кампанию (`autopilot:managed, city:novosibirsk-ai, topic:vtorichka, channel:search, Лидген, Вторичка, Покупатель`) и **8 меток** на RSYA-кампанию (плюс `РСЯ`). Метки `topic:vtorichka` и `channel:search/rsya` **дублируют** бизнес-метки `Вторичка` и `РСЯ`/`Поиск` — те же значения в разной форме. Сводки в Метрике/UI засорены: фильтры показывают и `Вторичка`, и `topic:vtorichka` одновременно. Оператор обоснованно поднял это как баг.

**Причина:** в `branches/apply.md` и `references/action_catalog.md` (leadgen-autopilot скилл) набор labels для `campaign.create_draft.in_*_topic` был жёстко прописан как `[autopilot:managed, city:<city>, topic:<topic>, channel:<channel>]` плюс бизнес-метки. Логика «дублирующего хранения» имела смысл изначально (быстрый scoping в коде автопилота через готовые internal-метки без парсинга бизнес-меток), но на практике эти данные уже есть в `state.yaml.campaigns[<id>].{topic,channel}` (operational source of truth) и в `<Тематика>` имени кампании (по конвенции `<Город> | <Канал> | <Тематика> | <Детали> | [<посадка>]`).

**Как применять:**
- В `branches/apply.md` автопилот ставит **минимум** internal-меток: `autopilot:managed` (must — ownership predicate) + `city:<city>` (must при ≥2 городов в аккаунте, optional при моноаккаунте).
- **Не ставить** `topic:<topic>` и `channel:<channel>` — они дублируют бизнес-метки.
- Бизнес-метки берутся стандартно из `leadgen/config/labels.md` (Лидген + тематика + направление + канал-метка) — без изменений.
- При reconcile значения `topic`/`channel` выводятся из:
  1. `state.yaml.campaigns[<id>].{topic,channel}` — primary source.
  2. Mapping в `leadgen/config/labels.md` секция «Mapping: business label → internal topic / channel» — fallback при потере state.
  3. Из `BiddingStrategy.Search/Network` (для channel) и из имени кампании (для topic).

**Cleanup существующих кампаний с лишними internal-метками:**
- В текущем MCP **нет инструмента** `remove_labels` / `set_banner_labels` — есть только идемпотентный `add_labels`. Удаление меток с ad возможно через API v4 Live `DeleteBannerTags`, но MCP-инструмент пока не реализован. **TODO для server/** — выделено в отдельную задачу.
- Временный workaround: оператор чистит лишние метки вручную через UI Директа.
- Существующие кампании, у которых уже есть `topic:`/`channel:` метки — оставлять как есть до релиза MCP-инструмента; они не блокируют работу автопилота, просто засоряют сводки.
