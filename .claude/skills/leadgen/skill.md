# Leadgen — управление рекламой недвижимости (Этажи)

Специализированный скилл для агентства недвижимости «Этажи». Оркестрирует MCP-инструменты Яндекс Директа, Метрики и Wordstat для создания, анализа и оптимизации рекламных кампаний.

**Компания:** Этажи — крупнейшее агентство недвижимости России. 70+ городов, домен etagi.com.
**MCP-сервер:** `leadgen-mcp` (Яндекс Директ + Метрика + Wordstat). Endpoint и режим подключения — см. [`config/mcp_endpoint.md`](config/mcp_endpoint.md).
**Аккаунт Директа:** `etagi click` (default) — агентский, 76 клиентов-городов.
**Аккаунт Метрики:** `etagi.agent` — 467 счётчиков, 70 активных городов.

## Коммуникация с пользователем (сквозное правило)

**Действует всегда — на каждом шаге любой ветки.**

### 1. UI-имена вместо API-констант

В чате, отчётах, `campaigns/<slug>.md`, превью C5.5, сообщениях к пользователю — используй имена, которые видит рекламодатель в кабинете Яндекс Директа, а НЕ API-константы.

- ✅ «Стратегия "Максимум конверсий"» / «Атрибуция "Последний значимый переход из Яндекс.Директа"» / «Расширенный геотаргетинг отключён»
- ❌ «Стратегия `WB_MAXIMUM_CONVERSION_RATE`» / «attribution=LYDC» / «ENABLE_AREA_OF_INTEREST_TARGETING=false»

**В параметрах вызовов MCP-инструментов (JSON, строковые значения `search_strategy`, `attribution` и т.п.) — ОСТАВЛЯЙ API-имена.** Иначе API сломается.

Полный маппинг API ↔ UI (стратегии, атрибуция, настройки, типы кампаний, статусы, автотаргетинг) — см. [`references/ui_naming.md`](references/ui_naming.md). Загружай этот файл перед написанием отчётов и документации кампаний.

### 2. Не использовать внутренние термины скилла

Пользователь НЕ знает внутреннюю нумерацию веток/шагов скилла. Никогда не пиши в чате/отчётах:

- ❌ «Применим подветку 3.2» / «Перейдём к Шагу C5» / «По ветке ОПТИМИЗАЦИЯ» / «O3.6», «C5.5»
- ✅ «Добавим новые ключевые через анализ поисковых запросов» / «Теперь разобью семантику на группы» / «Следующий аудит кампании»

Ссылки на шаги скилла допустимы только в моих внутренних заметках (рабочие todo, комментарии к коду), но НЕ в user-facing тексте.

### 3. Имена целей, счётчиков, меток

Всегда пиши человеческое имя + ID в скобках (если нужен). Не кидай голые ID.

- ✅ «цель "Коллтрекинг, реальный звонок" (500485089)», «счётчик Метрики основной (22325545)»
- ❌ «goal_id 500485089», «counter_id 22325545»

## Процессный контур: режимы работы, валидация, отчётность

**Сквозные правила для всех веток.** Определяют что, когда и в каком порядке делать вокруг самого рабочего действия.

### Режим работы — определяется в первом сообщении

Перед погружением в задачу явно назови режим. Если пользователь не указал — выведи из формулировки и зафиксируй.

| Режим | Когда | Что разрешено |
|---|---|---|
| `research` | «покажи», «аудит», «как дела», «посмотри» | Только read-only: `get_*`, `check_*`, Метрика/Wordstat чтения |
| `pre-apply-review` | «подготовь план», «что предложишь», «собери изменения» | Всё из research + составление пакета правок, НО без `add_/update_/manage_/delete_*` |
| `live-apply` | «внеси», «примени», «добавь», «запусти», «оптимизируй» + явное согласие | Мутирующие вызовы только по подтверждённому плану |
| `post-apply-monitoring` | После `live-apply` — через 1-3 дня и 7 дней | Read-back + сверка с прогнозом + новая задача если нужно. **Если правили ставки/стратегию/объявления** — запустить `.claude/skills/serp-monitor/scripts/track_position.sh` по топ-3 ключам, чтобы видеть связку изменение → позиция → показы. |

**Формат фиксации в начале работы:**

```
Режим: pre-apply-review
Scope: кампания omsk_poisk_vtorichka_odnokomnatnye (id 89167235), окно 7 дней
Подтверждение до apply обязательно.
```

**Смена режима в середине работы** — только с явного «ок» пользователя. Нельзя «по пути» переходить из research в live-apply.

### Автономный режим (auto vs review)

**В дополнение к режимам выше**, внутри `live-apply` есть два подрежима — зависят от явных триггеров в запросе пользователя.

| Подрежим | Триггеры в промпте | Поведение |
|---|---|---|
| **review** (по умолчанию) | триггеров auto нет | Агент делает шаги последовательно, на шагах с human-checkpoint (C5.5, R5.5 preview кластеров; R6.5 preview картинок; финальный apply) — останавливается, показывает результат, ждёт «ок» |
| **auto** | «запусти», «сам», «без модерации», «автоматически», «без подтверждения», «до конца», «не спрашивай» | Агент пропускает preview-шаги и apply-подтверждение; идёт весь флоу без остановок; фиксирует результат в отчёт в конце |

**Правила:**

1. Подрежим определяется в начале работы и фиксируется явно в первом ответе: «Подрежим: auto — выполняю без подтверждений».
2. Смена `review → auto` в середине — только по явному запросу пользователя («продолжай сам», «дальше без спроса»).
3. Даже в `auto` **обязательны** все чеклисты validation mesh перед мутирующими вызовами (Domain/Coverage/Apply-safety). Если чек не прошёл — стоп, отчёт о блокере. Auto ≠ «игнорировать валидацию».
4. `auto` не бывает в `research` и `pre-apply-review` — там всё read-only.
5. В `auto` режиме картинки РСЯ после генерации проходят только **автовалидацию** (размер, разрешение, формат) без ручного preview. Ложатся сразу в `adimages/add`.

### Перед live-apply: три чеклиста (Validation Mesh)

Прогоняй их ПЕРЕД любым мутирующим вызовом. Результат включай в pre-apply summary отдельным блоком.

**1. Domain validator — корректен ли сам пакет?**
- Минус-слова не конфликтуют с текущими целевыми ключевыми
- Новые ключи не каннибализируют уже работающие фразы
- Тексты прошли `copy_blacklist.md` (+ автопроверка `scripts/check_copy.py` при большом объёме)
- Лимиты соблюдены (title ≤ 56, title2 ≤ 30, text ≤ 81, callout ≤ 25)
- UTM-метки есть на уровне группы

**2. Coverage validator — не пропустил ли блок аудита?**
- Если пользователь просил «оптимизировать кампанию / кабинет» — проверил ли я: SQR / ключи / минус-слова / объявления / ставки / стратегию / бюджет / расписание / структуру / посадочные?
- Что сознательно вынесено в out-of-scope — явно помечено

**3. Apply-safety validator — готов ли к откату?**
- Есть свежий read-back состояния ДО изменений (сохранён в отчёт)
- План post-apply read-back готов (какие `get_*` вызову после apply)
- Rollback-сценарий понятен (что делать, если CPA вырастет в 2× за 3 дня)
- Окно apply учитывает правило чт-пт для изменений, влияющих на обучение (>30% бюджета, смена стратегии/цели, новые объявления)

**Если хоть один чеклист не пройден — apply запрещён, блокер назван явно.**

### После live-apply: обязательный отчёт (было → сделал → стало)

Работа НЕ считается завершённой, пока не показан отчёт.

**Структура отчёта:**

```
### Было (до изменений)
- Кампания <название> (id <id>), режим <стратегия>, бюджет <X> ₽/нед
- [затронутые сущности: campaign_id / adgroup_id / ad_id — сколько и какие]
- Ключевые метрики до: CPA <X>, CTR <X>%, конверсий/нед <X>

### Сделал
- [точная мутация]: update_campaign(budget=Y)
- [точная мутация]: add_keywords([...], adgroup_id=...)
- Что сознательно не трогал: <список>

### Стало (read-back)
- get_campaigns → State=ON, Status=ACCEPTED, бюджет=Y (подтверждено API)
- get_keywords → добавлено N фраз, все в статусе ACCEPTED
- [предупреждения, если что-то не прошло модерацию]

### Запись в историю
- Вызвал update_daily_summary(city="<город>") с итогом дня
- log_change_event(...) — если это критичное изменение стратегии/цели/бюджета
```

**Обязательный read-back после `resume_campaign`:** `get_campaigns(campaign_ids=[id])` с проверкой `State/Status/StatusPayment/StatusClarification`. Без этого запуск НЕ считается подтверждённым. Если `State=OFF` — назвать точный блокер из live API.

### Формат результата задачи (Output Contract)

Любой аудит/волна оптимизации оформляется по единой структуре внутри `campaigns/<slug>.md`. Это даёт сравнимость между циклами и возможность собрать HTML-отчёт.

**Обязательные разделы в записи аудита/оптимизации:**

1. **Scope** — что проверено
   - Окно анализа (даты)
   - Атрибуция: LYDC
   - Кампания/группы/объявления в scope
   - Что out-of-scope и почему

2. **Findings** (таблица) — что найдено
   | entity_type | entity_id | entity_name | verdict | confidence | evidence |
   |---|---|---|---|---|---|
   - verdict: `ok` / `attention` / `critical` / `opportunity` / `waste`
   - confidence: `high` / `medium` / `hypothesis-only`
   - evidence: конкретная цифра (CPA, CTR, расход) или ссылка на строку в source-данных

3. **Proposed changes** (таблица) — что предложено
   | change_id | entity_type | entity_id | current | proposed | reason | expected_effect | risk | confidence | approval_needed |
   |---|---|---|---|---|---|---|---|---|---|
   - expected_effect: `intraday` / `1-3d` / `7d+` для горизонта
   - confidence: `high-confidence operational` / `medium-confidence optimization` / `hypothesis-only`
   - approval_needed: `yes` (для критичных: смена цели, стратегии, >30% бюджета, новые объявления)

4. **Applied changes** (если режим был live-apply) — что сделано
   - Список мутаций с IDs
   - Read-back результаты
   - Отличия от плана (если были)

5. **Next steps** — что дальше
   - Монитор-окно (через сколько дней следующая проверка)
   - Триггеры для реакции (CPA > X → пауза, конверсий < Y → новая семантика)

**Для больших аудитов** — отдельный файл `campaigns/<slug>/<YYYY-MM-DD>_audit.md` с той же структурой; в основном файле кампании — ссылка и краткая строка-итог.

**HTML-отчёт** рендерится из этой структуры через `scripts/render_audit_report.py` (см. ниже).

### Логирование изменений в централизованную историю

После любого `live-apply` действия — вызвать MCP-инструмент `update_daily_summary(city)` с кратким описанием сделанного за день. Формат записи: одна запись на город на день, UPDATE в течение дня, INSERT на новый день (вчерашняя не перезаписывается).

Для отдельных критичных изменений (смена цели, смена стратегии, пауза/возобновление кампании, изменение бюджета >30%) — дополнительно `log_change_event(...)` с полным до/после.

Подробнее — см. секцию «История изменений» ниже в SKILL.md.

### Шаг 4. Сбор семантики (waves L1 → L2)

**Цель:** 30–80 ключевых фраз для типичной кампании. 14 фраз — слишком мало, 200+ — переспам. Ядро из get_semantic_cluster — лишь отправная точка.

**Канонический порядок:** `ПРОДУКТ-МАПА → МАСКИ L1 → РЕВЬЮ → WAVE 1 (Wordstat) → GAP-АНАЛИЗ → МАСКИ L2 → WAVE 2 → ВАЛИДАЦИЯ → КЛАСТЕРИЗАЦИЯ`.

**Уровни масок:**
- **L1** — однословные root-маски (широкий ландшафт; задача — увидеть неожиданные ветки спроса).
- **L2** — двухсловные product-маски (`{продукт} × {модификатор}`). Формируются из gap-анализа Wave 1.
- **L3/L4** — typo и competitor-маски — добавлять отдельной волной по запросу.

**Правила:**
- Wave 1 в 9 из 10 случаев строится на однословных (L1) масках.
- Wave 2 допускает двухсловные (L2). Трёхсловные маски — НЕ default.
- `check_search_volume` принимает до 10 фраз за раз; батчить соответственно.
- Всегда передавать `region_ids` города.
- Перед тяжёлым сбором — проверить квоту: `wordstat_user_info`.
- Между парсингом и анализом — жёсткий разрыв: не смешивать выгрузку с классификацией в одном вызове.

#### Раунд 0: Продукт-мапа (~5 минут, mental-model)

Для типичных тематик (вторичка, новостройки) уже лежит в `library/` и `mcp/semantic_clusters`. Для нестандартной (коммерческая, редкий город) — зафиксировать в голове:
- официальные названия: «квартира», «апартаменты»
- разговорные: «двушка», «однушка», «трёшка»
- жаргон/сленг: «хрущёвка», «сталинка», «брежневка»
- ошибки/латиница: «квартира», «kvartira»
- тендерные: — (для недвижки не актуально)

#### Раунд 1: Базовая матрица (seeds) → это L1 + частично L2 маски

```
1. Вызови get_semantic_cluster(theme="<тематика>")
   → modifiers, objects, extra_keys, negative_keywords

2. Построй ОСНОВНУЮ матрицу: {модификатор} × {объект} БЕЗ города
   Пример (вторичка):
     "купить квартиру", "купить однушку", "купить двушку недорого",
     "двухкомнатная квартира цена", "однокомнатная вторичка",
     "купить квартиру в ипотеку", "приобрести квартиру", ...
   → ~15-25 seed-фраз

   ⚠️ НЕ добавляй город к каждой фразе. Геотаргетинг работает через region_ids
   кампании/группы. Город в ключе заглушает трафик от людей, которые уже в Омске,
   но не пишут «омск» в запросе («купить двушку недорого» — главный транзакционный
   трафик, который терялся раньше).

3. Построй МИНИ-матрицу с уточнением города (для отдельного мини-кластера):
   Пример (Омск):
     "квартира омск", "купить квартиру омск", "двушка омск недорого"
   → 5-10 фраз, не больше

4. Собери модификаторы по ВСЕМ типам прибавок (library/selling_modifiers.md),
   а не только «купить» и «недорого»:
   - Транзакционные: купить, покупка, приобрести
   - Ценовые: недорого, дешево, цена, стоимость
   - Финансовые: в ипотеку, в рассрочку, с маткапиталом
   - Каталог: объявления, варианты, база
   - Условия продавца: от хозяина, от собственника, без посредников
   - Источники (высокий интент): авито, циан, домклик
   Из каждого типа — 1-2 фразы в seeds.

5. Если есть extra_keys в кластере get_semantic_cluster — добавь их БЕЗ города.
```

#### Раунд 2: Расширение через Wordstat (КЛЮЧЕВОЙ ЭТАП)

```
1. Вызови check_search_volume(
     phrases="<все seed-фразы через запятую>",
     region_ids="<geo_region_id города>"
   )
   → API вернёт для каждой фразы блок SearchedWith — реальные запросы пользователей

2. Из SearchedWith собери ВСЕ фразы с Shows >= 10 (для малых городов >= 5)

3. Какие фразы ценны в SearchedWith:
   - Гео-уточнения: район, микрорайон, улица ("квартира кировский район омск")
   - Ценовые: "до 2 млн", "недорого", "дешевые"
   - Типовые уточнения: "с балконом", "с ремонтом", "от собственника"
   - Синонимы: "однушка" ↔ "1 комнатная" ↔ "однокомнатная"
   - Long-tail: "купить однокомнатную квартиру омск вторичное жилье"

4. Отфильтруй нерелевантные — сравни с negative_keywords из кластера:
   - Аренда/сдача: снять, сдам, аренда, посуточно
   - Информационные: как, почему, отзывы, рейтинг, форум
   - Чужие бренды: авито, циан, домклик (если не в наших ключевых)
   - Другая тематика: новостройка, застройщик, жк (для вторички)
   - Нецелевые: ремонт, дизайн, курсовая, реферат
```

#### Раунд 3: Второй уровень расширения (опционально)

```
Если в Раунде 2 обнаружены высокочастотные фразы (Shows > 100),
которых НЕ было в seeds — запусти ещё один check_search_volume по ним.

Пример: seed "купить квартиру омск" → SearchedWith показал
"квартира левый берег омск" (Shows 320) — это новый кластер,
у которого свои вложенные запросы.

Лимит: не более 2 раундов расширения (seeds → round2 → round3).
Больше — diminishing returns.
```

#### Раунд 4: Финализация и подготовка к кластеризации

```
1. Объедини все собранные фразы, убери дубли (регистр, порядок слов)
2. Отсортируй по Shows (частотности) — от большей к меньшей
3. Проверь каждую на релевантность:
   - Соответствует тематике кампании?
   - Покупательский интент (не информационный)?
   - Подходит к посадочной странице?

4. Собери итоговый список минус-слов:
   - negative_keywords из get_semantic_cluster
   - + новые нерелевантные паттерны, обнаруженные в SearchedWith
   - Минус-слова идут на уровень кампании (negative_keywords в add_campaign)

5. Результат → массив фраз [{phrase, shows}] для Шага 5 (кластеризация)

6. ВАЛИДАЦИЯ перед передачей в C5:
   □ Доля фраз БЕЗ города в ключе ≥ 70% от общего числа (гео — через region_ids, не словом)
   □ Самый частотный запрос без города (типа «купить двушку недорого», «купить однокомнатную»)
     попал в выборку и не был отфильтрован как нерелевантный
   □ Присутствуют фразы хотя бы из 4 разных типов модификаторов
     (транзакционный / ценовой / финансовый / каталог / условия продавца)
     — см. library/selling_modifiers.md
   □ Мини-кластер «с уточнением города» содержит 5-10 фраз и вынесен отдельно
   □ В negative_keywords добавлен блок «Чужие города РФ» из mcp/negative_keywords.md
     (кроме собственного города и его словоформ)
   □ Итоговое число минус-слов кампании ≥ 150 (см. mcp/negative_keywords.md → шаблон сборки)

   Если чек не прошёл — вернись на Раунд 1 или Раунд 2 и добери недостающее.
```

**Важно:** check_search_volume принимает до 10 фраз за раз (лимит API). Разбивай на батчи по 10. Всегда передавай region_ids города — без гео данные будут по всей России и нерелевантны.

### Шаг 5. Кластеризация семантики → структура групп объявлений

**Ключевой шаг.** Собранную семантику нельзя складывать в одну группу — нужно разбить на кластеры, каждый из которых станет отдельной группой объявлений с индивидуальным контентом.

#### Принципы кластеризации

```
Агент САМОСТОЯТЕЛЬНО решает, как разбить семантику. Ориентиры:

1. ГЕО-КЛАСТЕРЫ (приоритет если обнаружены)
   Если в SearchedWith найдены фразы с районами/микрорайонами — это отдельные кластеры.
   Каждый район → своя группа с:
   - Названием района в заголовке объявления
   - Фильтром района в посадочной URL (если сайт поддерживает, см. build_landing_url)
   - Ключевыми только по этому району

   Пример (Омск, вторичка, однокомнатные):
   | Кластер | Ключевые (примеры) | Заголовок | Посадочная |
   |---------|-------------------|-----------|------------|
   | Нефтяники | "квартира нефтяники омск", "однушка нефтяники" | Однушки в Нефтяниках | /realty/?rooms=1&district=... |
   | Кировский | "квартира кировский район омск" | Однушки в Кировском | /realty/?rooms=1&district=... |
   | Левый берег | "квартира левый берег омск" | Однушки на Левом берегу | /realty/?rooms=1&district=... |

2. ИНТЕНТ-КЛАСТЕРЫ (ОБЯЗАТЕЛЬНО проверь КАЖДЫЙ тип из library/selling_modifiers.md)

   | Тип | Маркеры фраз | Акцент заголовка |
   |---|---|---|
   | 1. Транзакционный | купить, покупка, приобрести, оформить | "Купить квартиру в {город}" |
   | 2. Ценовой | недорого, дешево, цена, стоимость, от ... т.р. | "Квартиры от 1.5 млн", "Недорого" |
   | 3. Финансовый | в ипотеку, в рассрочку, с маткапиталом | "Квартира в ипотеку" |
   | 4. Каталог | объявления, варианты, база, подборка | "Актуальная база квартир" |
   | 5. Условия продавца | от хозяина, от собственника, без посредников | "От собственников" |
   | 6. Источники (конкуренты) | авито, циан, домклик | "Квартиры в {город} — Этажи" (без упоминания конкурента в Title) |
   | 7. Срочность/состояние | срочно, готовая, с ремонтом, под ключ | "Готовые квартиры" |

   ПРАВИЛО: если в собранной семантике есть фразы типа согласно порогу из
   library/selling_modifiers.md (≥ 3 фразы ИЛИ ≥ 1 фраза с Shows ≥ порога) —
   ВЫДЕЛИ в отдельный кластер.

   ⚠️ НЕ сворачивай всё в 2 кластера («Купить» и «Недорого»). Это была главная
   ошибка тестовой кампании — теряется 5 интент-типов и финансовый кластер
   (ипотека/рассрочка) в первую очередь.

3. ОБЪЕКТНЫЙ КЛАСТЕР (если кампания шире одного типа)
   - Однушки, двушки, студии — если все в одной кампании

4. ОБЩИЙ кластер (catch-all)
   Широкие фразы без уточнений: "квартира омск", "однокомнатная омск"
   → Общий заголовок без гео-уточнения, базовая посадочная
```

#### Алгоритм кластеризации

```
1. СНАЧАЛА извлеки гео-кластеры:
   - Пройди по всем фразам, найди упоминания районов/микрорайонов
   - Группируй: все фразы с "нефтяники" → кластер "Нефтяники"
   - Минимум для гео-кластера: 3+ фразы ИЛИ 1 фраза с Shows >= 50

2. ЗАТЕМ остаток разбей по интенту:
   - Фразы с "купить/покупка" → прямой интент
   - Фразы с "цена/недорого/дешев" → ценовой
   - Фразы с "вторичка/собственник/хозяин/без посредников" → условия
   - Остальное → общий кластер

3. ОЦЕНИ размер кластеров:
   - Если кластер < 3 фраз и Shows < 50 — влей в ближайший по смыслу
   - Если кластер > 40 фраз — подумай, можно ли разбить ещё
   - Оптимально: 5-20 фраз на кластер

4. ДЛЯ КАЖДОГО кластера определи:
   - Название группы (для add_adgroup): "{Тематика} — {кластер}"
   - Ключевые фразы (для add_keywords)
   - Посадочный URL: базовый + фильтры (если есть в site_filters.md)
   - Специфика заголовка: что добавить/изменить в Title

5. ОБЯЗАТЕЛЬНО: создай группу для КАЖДОГО кластера с ≥ 3 фразами.
   Не ограничивайся только гео-кластерами — интент-кластеры (ценовой,
   финансовый, каталог, условия продавца, срочность, catch-all) тоже
   ДОЛЖНЫ стать отдельными группами. Каждый кластер = отдельная группа объявлений.

6. ВАЛИДАЦИЯ ПОСЛЕ РАЗБИЕНИЯ:
   □ Число кластеров ≥ 4 (2 — слишком мало, не ловим все интенты)
   □ Каждый тип из library/selling_modifiers.md либо имеет свой кластер,
     либо фраз этого типа не было в семантике (подтверждено Wordstat)
   □ Обязательно должен быть финансовый кластер для кампаний вторички/новостроек
     — ипотека даёт основной объём конверсий
   □ Выделен отдельный мини-кластер «Уточнение города» (5-10 фраз с «омск»/«{город}»)
   □ Есть catch-all «Общий» для широких фраз без уточнений

   Если ≤ 3 кластеров — вернись на C4 и добери семантику по недостающим типам
   через check_search_volume.
```

### Шаг 5.5. Детальное превью для отсмотра спецом (ОБЯЗАТЕЛЬНО)

**Зачем:** верхнеуровневая таблица кластеров (C5) не даёт спецу возможности
посмотреть каждый ключ + его частотность. Без этого часть мусора улетает
в боевую кампанию и вычищается уже на расходе. Превью закрывает это

```

   )
   // ⚠️ UTM ставим НА КАМПАНИЮ (tracking_params в add_campaign), а не на каждую группу.
   // Проще менять, реже нужно разграничение по группам.

   // Без ссылок объявление теряет 20-30% CTR.
   add_sitelinks(

   )
   → sitelink_set_id  // используем один и тот же ID во всех add_ad

3. УТОЧНЕНИЯ — общий набор на всю кампанию (до цикла по кластерам):
   // Загрузить library/extensions.md → список уточнений по тематике.
   // add_ad_extension ОТДЕЛЬНО для каждого уточнения (4-8 штук):
   add_ad_extension(client_login=<login>, callout_text="Без комиссии")
     → ad_extension_id_1
   add_ad_extension(client_login=<login>, callout_text="Юридическая проверка")
     → ad_extension_id_2
   // ... всего 4-8 уточнений
   → ad_extension_ids = "<id1>,<id2>,<id3>,..."  // строка через запятую для add_ad

4. ЦИКЛ ПО КЛАСТЕРАМ — для каждого кластера:

   a) add_adgroup(
        client_login=<login>,
        campaign_id=<campaign_id>,
        name="<Тематика> — <Название кластера>",
        region_ids=[<регион>]
      )

   b) add_keywords(
        client_login=<login>,
        adgroup_id=<adgroup_id>,
        keywords="<ключевые ЭТОГО кластера через запятую>"
      )
      // ОБЯЗАТЕЛЬНЫЙ ШАГ — без ключевых группа работает только на автотаргетинге!
```

> **Чеклист кампании:** sitelinks (один набор на кампанию) ✓ → уточнения (один общий список) ✓
> **Чеклист на каждую группу:** adgroup ✓ → keywords ✓ → UTM ✓ → autotargeting ✓ → 3 ads (уже с sitelinks+уточнениями) ✓

> ⚠️ **Нюанс API v5 (важно):** API v5 `TextAdUpdate` напрямую не принимает
> `AdExtensionIds` (в отличие от `TextAdAdd`) — уточнения меняются через
> `CalloutSetting` с операциями ADD/REMOVE/SET. MCP-инструмент `update_ad`
> это скрывает: передавай `ad_extension_ids` — он сам транслирует в
> `CalloutSetting.AdExtensions[Operation:SET]` (полная замена набора).
> Тем не менее предпочитай указывать `ad_extension_ids` сразу в `add_ad`
> при создании — меньше запросов и нет риска частичной записи.
> Привязать/заменить уточнения на существующих объявлениях — см. ВЕТКА 3, подветка 3.8.
> **Именование групп:** `"Вторичка Однушки — Общий"`, `"Вторичка Однушки — Нефтяники"`, `"Вторичка Однушки — Кировский"` и т.д.

<!-- C8 — кросс-минусация -->
### Шаг 8. Кросс-минусация между группами (ОБЯЗАТЕЛЬНО)

**Зачем:** если в группе «Нефтяники» ключевая "купить квартиру нефтяники омск", а в группе «Общий» — "купить квартиру омск", то запрос "купить квартиру нефтяники омск" может сработать в обеих группах. Кросс-минусация гарантирует, что более точная группа забирает свой трафик.

```
1. Собери все ключевые фразы по группам (уже есть из Шага 7)

2. Для каждой пары групп найди пересечения:
   - Извлеки уникальные слова-уточнения каждого кластера
     (район, тип объекта, ценовой модификатор)
   - Эти слова → минус-слова для ДРУГИХ групп

3. Типичные кросс-минусы:
   - Гео-кластер "Нефтяники" → в группу "Общий" добавь минус "нефтяники"
   - Гео-кластер "Кировский" → в группу "Общий" добавь минус "кировский"
   - Ценовой кластер "недорого" → в группу "Общий" добавь минус "недорого","дешев"
   - Каждый гео-кластер минусует другие районы

4. Применение:
   add_negative_keywords(
     client_login=<login>,
     adgroup_id=<id группы "Общий">,
     keywords="нефтяники,кировский,левый берег"
   )
   // Повторить для каждой группы — минусуем чужие уточнения

5. ПРОВЕРКА: ни одна ключевая фраза группы не должна содержать
   собственные минус-слова этой группы (иначе фраза заблокируется)
```

> **Правило:** кросс-минусация делается ПОСЛЕ создания всех групп, когда полная картина ключевых известна.

<!-- R4 — семантика для РСЯ -->
### Шаг R4. Семантика для РСЯ (широкая)

**Принцип:** в РСЯ показываемся на тематических сайтах — нужны фразы **категории**, а не точные транзакционные.

```
1. get_semantic_cluster(theme="<тематика>") → база модификаторов и объектов
2. Wave 1: тематические фразы БЕЗ города, с широкими модификаторами
   Пример для вторички: "купить квартиру", "квартиры", "двушка",
   "вторичка", "квартира цена", "квартира ипотека"
   → 15-25 фраз (меньше чем в поиске)
3. check_search_volume(phrases=..., region_ids=<город>)
   → оставить фразы с Shows ≥ 100 (в РСЯ важен охват, не точность)
4. Добавить интересы/автотаргетинг через update_autotargeting
   (в R7 - EXACT=YES, ALTERNATIVE=YES)
5. Минус-слова: из `rsya_defaults.md` раздел 9 (инфо-интент блокируется)
```

**НЕ делаем в РСЯ:**
- Мелко-частотные long-tail (< Shows 50) — лишние фразы не дают охват, только шум.
- Точные конкурентные [циан, авито] — в РСЯ не работает как в поиске.
- Гео-районы в ключах — работает через region_ids.

**Результат:** 10–30 широких фраз на кампанию.

### Шаг R5. Кластеризация по аудитории

Группы формируются **по типу аудитории**, не по интенту:

| Группа (обязательные) | Что внутри | Когда добавлять |
|---|---|---|
| **Общие-купить** | Все широкие фразы + автотаргетинг EXACT+ALTERNATIVE | Всегда |
| **По сегменту** (1-комн / 2-комн / конкретный ЖК) | Фразы с указанием сегмента | Если есть посадочная под сегмент |
| **LAL** | Те же фразы + look-alike сегмент Метрики | Опция: только если > 5 000 визитов/мес на сайте |
| **Ретаргет** | Те же фразы + retargeting_list «был на сайте, не оставил заявку» | Опция: при наличии трафика |

```
3. Информационные запросы → в минус (сверить с минус-словами из get_semantic_cluster)
4. ПОДТВЕРЖДЕНИЕ пользователя перед применением
5. update_campaign(campaign_id, negative_keywords=<список>)
```

#### 3.2 Новые ключевые (упущенный спрос)

```
1. get_search_queries → запросы с конверсиями, но не в ключевых
2. check_search_volume(phrases="<найденные фразы>", region_ids=<гео>)

4. ПОДТВЕРЖДЕНИЕ пользователя (показать таблицу: фраза + Shows)
5. add_keywords(adgroup_id=<id>, keywords="<фразы через запятую>")
```

<!-- O3.3 — A/B тест объявлений -->
#### 3.3 A/B тест объявлений (низкий CTR)

```
2. Загрузить library/titles.md, library/texts.md
3. SERP-разбор конкурентов (опц., если tokens.env настроен):
   - Взять 3-5 топ-ключей группы (из get_keywords, сортировка по показам)
   - Запустить .claude/skills/serp-monitor/scripts/search.sh --query "<ключ>" --region <geo>
   - Выжать из top-8: повторяющиеся УТП (цена, ипотека, срок), CTA, эмодзи/триггеры

4. Создать новый вариант с гипотезой (учитывая SERP-разбор, если был)
5. ПОДТВЕРЖДЕНИЕ пользователя
6. add_ad(adgroup_id, ...)
7. Через 2-3 недели: сравнить, остановить проигравшего
```

```
Условие: 10+ конверсий/нед стабильно 2-3 недели подряд
1. ПОДТВЕРЖДЕНИЕ пользователя (лучше в чт-пт)
```

#### 3.5 Бюджет

<!-- O3.6 — кросс-минусация -->
#### 3.6 Кросс-минусация (каннибализация между группами)

**Используй когда:** группы внутри одной кампании конкурируют за одни и те же запросы — трафик уходит в менее релевантную группу.

```
1. ДИАГНОСТИКА:
   get_keywords(campaign_ids=<id>) → собери все ключевые по группам
   get_search_queries(campaign_ids=<id>, date_from=<30 дней>, attribution="LYDC")
   → Найди запросы, которые сработали в НЕСКОЛЬКИХ группах
   → Определи: в какой группе запрос более релевантен (по заголовку/посадочной)

2. ПОСТРОЕНИЕ кросс-минусов:
   - Для каждой группы определи уникальные слова-маркеры
     (район, тип объекта, ценовой модификатор)
   - Эти маркеры → минус-слова для ВСЕХ остальных групп

3. ПРИМЕНЕНИЕ:
   add_negative_keywords(
     client_login=<login>,
     adgroup_id=<id>,
     keywords="<чужие маркеры через запятую>"
   )

4. ПРОВЕРКА:
   - Ни одна ключевая фраза группы не содержит собственные минус-слова
   - Широкая группа ("Общий") минусует ВСЕ уточнения из специализированных групп
```

<!-- O3.8 — привязка sitelinks/уточнений к существующим объявлениям -->
#### 3.8 Привязка sitelinks/уточнений к существующим объявлениям

**Используй когда:** аудит показал, что у объявлений пустой `AdExtensions` или
нет `SitelinkSetId` — например, в старой кампании или после миграции.

```
1. ДИАГНОСТИКА:
   get_ads(campaign_ids=<id>, states="ON",
     text_ad_field_names="SitelinkSetId,AdExtensions")
   → Найди объявления без привязки (пустой массив AdExtensions, отсутствует SitelinkSetId)

2. ПОДГОТОВКА НАБОРОВ (если их ещё нет на аккаунте):
   - Быстрые ссылки: add_sitelinks(...) → sitelink_set_id
   - Уточнения: add_ad_extension(callout_text=...) × 4-8 штук → ad_extension_ids
   - Повторно используемые наборы искать через get_sitelinks / get_ad_extensions

3. ПРИВЯЗКА — по одному update_ad на объявление:
   update_ad(
     client_login=<login>,
     ad_id=<banner_id>,
     sitelink_set_id=<sitelink_set_id>,
     ad_extension_ids="<id1>,<id2>,<id3>,..."
   )
   // update_ad транслирует ad_extension_ids в CalloutSetting SET (полная замена набора).
   // Можно передать оба поля одним вызовом — или только одно, если второе уже есть.

4. ПРОВЕРКА:
   get_ads(ad_ids="<список>",
     text_ad_field_names="SitelinkSetId,AdExtensions")
   → Убедись что у всех объявлений AdExtensions заполнен и SitelinkSetId != 0
```

> **ПРАВИЛО:** для НОВЫХ кампаний — всегда делай привязку через `add_ad`
> (параметры `sitelink_set_id` и `ad_extension_ids`), а `update_ad` используй
> ТОЛЬКО для ретрофита старых объявлений или смены набора уточнений.

<!-- O3.9 — миграция UTM с уровня группы на уровень кампании -->
#### 3.9 Миграция UTM с уровня группы на уровень кампании

**Используй когда:** старые кампании имеют `tracking_params` на группах (старая схема)
вместо кампании. Миграция упрощает редактирование UTM — одно место вместо N групп.

```
1. ДИАГНОСТИКА:
   get_campaigns(campaign_ids=<id>, field_names="Id,Name,TextCampaign")
   → посмотри TextCampaign.TrackingParams — пусто?

   get_adgroups(campaign_ids=<id>, field_names="Id,Name,TrackingParams")
   → собери все tracking_params групп

   Если campaign.TrackingParams пустой И у всех групп одинаковый UTM → годится для миграции.
   Если у групп РАЗНЫЕ UTM — сначала уточни с пользователем: они осознанно разные или
   это исторический дрейф?

2. ПОДТВЕРЖДЕНИЕ пользователя (хард-гейт: изменение UTM = изменение атрибуции).

3. ПРИМЕНЕНИЕ:
   // Шаг 3a: поставить UTM на кампанию
   update_campaign(
     client_login=<login>,
     campaign_id=<id>,
     tracking_params=<общий UTM из get_utm_config>
   )

   // Шаг 3b: очистить UTM с ВСЕХ групп кампании
   for adgroup_id in adgroup_ids:
     update_adgroup(
       client_login=<login>,
       adgroup_id=<adgroup_id>,
       clear_tracking_params=true
     )

4. ПРОВЕРКА:
   get_campaigns(campaign_ids=<id>, field_names="Id,TextCampaign")
     → TextCampaign.TrackingParams заполнен
   get_adgroups(campaign_ids=<id>, field_names="Id,TrackingParams")
     → все TrackingParams пустые

5. ФИКСАЦИЯ в campaigns/<slug>.md: отметить миграцию UTM с датой.
```

> **Риск:** атрибуция Директа и Метрики непрерывна — миграция не ломает статистику,
> т.к. tracking_params рендерится в URL при клике, а не сохраняется в логах задним числом.
> Но делать лучше в чт-пт и после подтверждения пользователя.

<!-- O3.10 — отключение динамических мест показа -->
#### 3.10 Отключение динамических мест показа (DynamicPlaces=NO)

**Используй когда:** аудит O0 показал, что у кампании `DynamicPlaces` = "YES" или не установлен.
По договорённости с SEO — ВСЕГДА должно быть "NO".

**Контекст:** «Динамические места показа» — это показы в дополнительных местах
(Яндекс.Карты, Маркет, другие сервисы и партнёры). SEO Этажи договорились с
маркетингом: такие показы отключаем — они размывают атрибуцию и пересекаются
с органикой.

API-путь (из кабинета): `BiddingStrategy.Search.PlacementTypes.DynamicPlaces`
Значения: `YES` / `NO`. Применимо к `TextCampaign`, `DynamicTextCampaign`, `UnifiedCampaign` (ЕПК).

```
1. ДИАГНОСТИКА:
   // Сначала определи тип кампании:
   get_campaigns(campaign_ids=<id>, field_names="Id,Name,Type")
   → Type: TEXT_CAMPAIGN / DYNAMIC_TEXT_CAMPAIGN / UNIFIED_CAMPAIGN / ...

   // Затем запроси PlacementTypes через соответствующий type-specific FieldNames:
   get_campaigns(
     campaign_ids=<id>,
     field_names="Id,Name,Type",
     text_campaign_field_names="BiddingStrategy"            // если Type=TEXT_CAMPAIGN
     // dynamic_text_campaign_field_names="BiddingStrategy" // если DYNAMIC_TEXT_CAMPAIGN
     // unified_campaign_field_names="BiddingStrategy,PlacementTypes" // если UNIFIED_CAMPAIGN (ЕПК)
   )
   → TextCampaign.BiddingStrategy.Search.PlacementTypes.DynamicPlaces
   Если "YES" или отсутствует → требуется выключение.

   ⚠️ НЮАНС API: обычный field_names НЕ возвращает PlacementTypes — он вложен в
   TextCampaign.BiddingStrategy. Поэтому нужен отдельный text_campaign_field_names.

2. ПРИМЕНЕНИЕ (без подтверждения, т.к. это жёсткая договорённость):
   update_campaign(
     client_login=<login>,
     campaign_id=<id>,
     search_strategy=<текущая стратегия, передать для активации блока>,
     disable_dynamic_places=true
   )
   // disable_dynamic_places=true → search.PlacementTypes.DynamicPlaces = "NO"

   ⚠️ ВАЛИДАЦИЯ: API при update валидирует объект целиком. Если у кампании
   есть ограничения по сочетанию стратегии/мест (например, нельзя отключить
   DynamicPlaces при определённой стратегии) — вернётся ошибка валидации.
   В таком случае:
     a) Прочитай текущее состояние (шаг 1).
     b) Покажи пользователю, попроси согласовать изменение стратегии.
     c) Если нельзя изменить — зафиксируй в campaigns/<slug>.md как исключение.

3. ПРОВЕРКА:
   get_campaigns(
     campaign_ids=<id>,
     field_names="Id,Type",
     text_campaign_field_names="BiddingStrategy"  // или соответствующий тип
   )
   → убедись, что DynamicPlaces = "NO"

4. ФИКСАЦИЯ в campaigns/<slug>.md — отметь миграцию с датой.
```

> **Важно:** в `add_campaign` этот флаг уже включён по умолчанию — новые кампании
> автоматически создаются с `DynamicPlaces=NO`. Эта подветка нужна только для
> миграции существующих кампаний.

<!-- O3.7 — расширение структуры -->
#### 3.7 Расширение структуры (добавление групп в существующую кампанию)

**Используй когда:** нужно добавить новые группы объявлений в уже работающую кампанию (новые районы, новые кластеры, новая семантика).

```
Шаг А. Контекст существующей кампании (ОБЯЗАТЕЛЬНО)

1. Получи текущие группы, объявления, ключевые:
   get_adgroups(campaign_ids=<id>)
   get_ads(campaign_ids=<id>, states="ON",
     text_ad_field_names="Title,Title2,Text,Href")
   get_keywords(campaign_ids=<id>)

2. ОПРЕДЕЛИ СКОУП кампании из названия и ключевых:
   - Кампания "Вторичка | Однокомнатные" → собирай семантику ТОЛЬКО по однокомнатным
   - Кампания "Вторичка" (без уточнения) → все типы комнатности допустимы
   - Кампания "Загородка | Дома" → только дома, не участки и не дачи

3. ФИЛЬТРУЙ семантику по скоупу:
   - Если кампания по однокомнатным — исключи из ключевых: "2 комнат", "двушк",
     "двухкомнат", "3 комнат", "трешк", "трехкомнат", "4 комнат", "студи"
   - Если кампания по домам — исключи: "участ", "дач", "таунхаус"
   - Применяй фильтр НА КАЖДОМ этапе: при сборе seeds, при обработке SearchedWith,
     при финальной проверке перед кластеризацией

4. НЕ ДУБЛИРУЙ существующие ключевые — проверь перед add_keywords

Шаг Б. Сбор и кластеризация

→ Используй Шаги 4-5 из ВЕТКИ 1 (Создание), но с учётом скоупа из Шага А.
→ Исключи кластеры, которые уже есть в кампании.

Шаг В. Создание групп

→ Используй Шаги 6-8 из ВЕТКИ 1 (Создание) для новых кластеров.
→ Чеклист на каждую группу: adgroup ✓ → keywords ✓ → UTM ✓ → autotargeting ✓ → 3 ads ✓ → sitelinks ✓ → уточнения ✓ → кросс-минусация ✓
```

<!-- OR.1 — чистка площадок -->
### OR.1. Чистка площадок-дренажей

**Когда:** CPA выше порога ИЛИ через 3 дня после запуска (обязательно).

```
1. get_custom_report(
     report_type="CUSTOM_REPORT",
     field_names="CampaignName,Placement,Impressions,Clicks,Cost,Conversions",

     date_to=<сегодня-1>,

   - Расход > 200 ₽ И CTR < 0.05%      → кандидат на блок
3. Показать пользователю список (в review) или сразу блокировать (в auto):
   add_blocked_placements(campaign_id, places=<список>)
```

<!-- OR.2 — A/B картинок -->
### OR.2. A/B тест картинок

**Когда:** CTR ниже нормы (< 0.3%) ИЛИ накопилось > 10 000 показов без роста конверсий.

```
1. get_ad_stats (по объявлениям кампании) → найти объявления с худшим CTR

4. add_ad_image + update_ad (ad_image_hash=<новый>) — заменить картинку в худших объявлениях
   ИЛИ добавить новые объявления (add_ad ×3 с новыми картинками), старые приостановить (manage_ads action=suspend)
5. Мониторить 7 дней, сравнить CTR старых vs новых
```


<!-- OR.5 — корректировки по устройствам/полу/возрасту -->
### OR.5. Корректировки bid_modifiers

```
3. Мониторить 7 дней
```

## Правила безопасности

1. **Изменения стратегии/цели/бюджета >30%** — ТОЛЬКО с подтверждением пользователя
2. **Лучше в чт-пт** — изменения вызывают «разведку» алгоритма
3. **Не удалять объявления** — только update_ad для изменений
4. **Атрибуция** — всегда LYDC в отчётах
5. **Бюджет** — в РУБЛЯХ, недельный, НЕ микроюниты
6. **Юридическая проверка** — все тексты через LEGAL.md + `references/copy_blacklist.md`
7. **Пополнять `references/lessons_registry.md`** при обнаружении нового API-гочи

## История изменений (централизованная БД через MCP)

Общая БД на весь агентский аккаунт. Все специалисты пишут в неё — не в локальные файлы скилла.

### Архитектура

```
agency_account (etagi click)
    └── city_login (e-12345)
            └── campaign_id (89167235)
                    └── events (change_events) — детальная история
            └── date (2026-04-20)
                    └── daily_summary — краткое саммари дня по городу
```

**Файл БД:** `server/data/change_history.db` (SQLite, в `.gitignore`).
**Два слоя:**
- `change_events` — append-only детальный лог критичных мутаций
- `daily_summaries` — одна запись на город на день (UPDATE в течение дня, INSERT на новый)

### MCP-инструменты

| Инструмент | Когда вызывать |
|---|---|
| `log_change_event` | После успешного read-back критичного live-изменения (смена цели/стратегии, бюджет >30%, пауза/запуск, добавление объявления) |
| `update_daily_summary` | В конце рабочей сессии по городу — одним вызовом с кратким итогом дня |
| `get_change_history` | Когда смотришь что менялось по кампании/городу — ДО новых правок (чтобы понимать контекст) |
| `get_daily_summary` | Когда нужен обзор «что делали на этой неделе» или «что было вчера» по городу |

### Правила записи

**Что писать в `log_change_event`:**
- Смена `goal_id` / `priority_goals`
- Смена стратегии (`WB_MAXIMUM_CONVERSION_RATE` → `AVERAGE_CPA` и обратно)
- Изменение недельного бюджета >30%
- `suspend_campaign` / `resume_campaign` / `archive_campaign`
- Добавление нового `add_ad` (A/B-тест)
- Массовое добавление минус-слов (>50 за раз)
- Любое live-изменение, где нужно будет понять «что именно я поменял 2 недели назад»

**Что НЕ писать:**
- Повседневные `get_*` (чтения)
- Мелкие корректировки текстов уже работающих объявлений
- Изменения в локальных файлах кампаний

**Формат вызова log_change_event:**

```
log_change_event(
  agency_account="etagi click",
  city_login="<из get_city_config>",
  city="omsk",
  campaign_id="89167235",
  campaign_name="Поиск | Вторичка | Покупатель",
  entity_type="strategy",
  entity_id="89167235",
  action_type="update",
  tool_name="update_campaign",
  before_value="WB_MAXIMUM_CONVERSION_RATE, budget=500",
  after_value="AVERAGE_CPA, cpa=850, budget=1500",
  reason="10+ конверсий/нед 3 недели подряд, переход на AVERAGE_CPA",
  correlation_key="task-2026-04-20-omsk-strategy-switch"
)
```

**Формат вызова update_daily_summary (в конце сессии):**

```
update_daily_summary(
  city_login="<из get_city_config>",
  city="omsk",
  agency_account="etagi click",
  summary="Добавил 47 минус-слов из SQR в campaign 89167235, обновил текст 3 объявлений в группе Нефтяники. CPA за 7 дней 820 ₽ — в пределах нормы.",
  operator_name="<имя специалиста, если знаешь>",
  event_count=4,
  mode="append"
)
```

**Правило `mode`:**
- `append` (default) — дописать к сегодняшней записи. Используй если продолжаешь работу по тому же городу в том же дне.
- `replace` — заменить. Используй только если начинаешь саммари заново (например, случайно записал не то).

### Чтение истории перед работой

Перед началом оптимизации по кампании — запустить:

```
get_change_history(campaign_id="<id>", limit=20)
```

Это даёт контекст: что менялось недавно, чтобы не откатывать чужие изменения и не тратить время на то, что уже пробовали.

Перед началом сессии по городу:

```
get_daily_summary(city_login="<login>", limit=7)
```

— что делали последнюю неделю.
