# Leadgen — управление рекламой недвижимости (Этажи)

Специализированный скилл для агентства «Этажи». Оркестрирует MCP-инструменты Яндекс Директа, Метрики и Wordstat для создания, анализа и оптимизации рекламных кампаний.

**Компания:** Этажи — крупнейшее агентство недвижимости России. 70+ городов, домен etagi.com.
**MCP-сервер:** `leadgen-mcp` (Яндекс Директ + Метрика + Wordstat + VK Ads + image generation).
**Аккаунт Директа:** `etagi click` (default) — агентский, 76 клиентов-городов.
**Аккаунт Метрики:** `etagi.agent` — 467 счётчиков, 70 активных городов.

> **Структура скилла.** Этот файл — только роутер и сквозные правила. Детали каждой ветки лежат в `branches/` и подгружаются **лениво** после роутинга (см. раздел «Правила ленивой загрузки» ниже). Не читай бранчи заранее «на всякий случай».

**Не грузи файлы «на всякий случай».** Для типичной задачи достаточно этого `skill.md` + одного бранча + одного-двух references.

Правила (в порядке выполнения):

1. **Прочитай skill.md → определи ветку через Роутер 1 и канал через Роутер 2.** Если что-то не ясно — спроси пользователя, не угадывай.
2. **Загрузи ровно один бранч** — из таблицы роутера. Не комбинируй ветки заранее (например при «создай РСЯ» не грузи `branches/optimize-rsya.md`).
3. **Подгружай `references/*.md` точечно** — файл читай в момент, когда доходишь до шага, которому он нужен:
   - `references/rsya_defaults.md` → на R2 (конфиг РСЯ).
   - `references/image_prompts.md` → **только** на R6.5 (генерация картинок).
   - `references/city_benchmarks.md` → при проверке CPA на A4 / при задании `tCPA` на C2/R2.
   - `references/copy_blacklist.md` → перед `add_ad` (юр-проверка текстов).
   - `references/ui_naming.md` → перед написанием отчётов/документации.
   - `references/lessons_registry.md` → **только** при ошибке API или вопросе «почему падает».
   - `references/site_structure.md` → при подборе посадочной, если MCP `build_landing_url` не покрыл.
4. **Library — по триггеру.** `library/titles.md`/`texts.md` — на C6 (контент поиска). `library/banner_titles.md`/`banner_texts.md` — на R6 (контент РСЯ). `library/extensions.md`/`selling_modifiers.md`/`display_urls.md` — когда эти типы расширений нужны.
5. **MCP-инструменты — источник правды.** Всё, что отдаёт MCP (`get_city_config`, `get_conversion_values`, `get_utm_config`, `get_semantic_cluster`, `get_sitelink_templates`, `build_landing_url`) — бери оттуда, **не читай файлы-дубли**. Файлов-справочников по этим данным в скилле нет (были — вычищены при рефакторинге апр 2026).
6. **`flow-steps.md` — для ревизии скилла, не для каждой задачи.** Это обзорная карта шагов. При выполнении — читай бранч, а не этот файл.

**Ссылки на файлы из этого skill.md** — это навигация, не сигнал «грузи сейчас». Грузи по правилам выше.

## Процессный контур: режимы работы, валидация, отчётность

**Сквозные правила для всех веток.** Определяют что, когда и в каком порядке делать вокруг самого рабочего действия.

### Режим работы — определяется в первом сообщении

| Режим | Когда | Что разрешено |
|---|---|---|
| `research` | «покажи», «аудит», «как дела», «посмотри» | Только read-only: `get_*`, `check_*`, Метрика/Wordstat чтения |
| `pre-apply-review` | «подготовь план», «что предложишь», «собери изменения» | Всё из research + составление пакета правок, БЕЗ `add_/update_/manage_/delete_*` |
| `live-apply` | «внеси», «примени», «добавь», «запусти», «оптимизируй» + явное согласие | Мутирующие вызовы только по подтверждённому плану |
| `post-apply-monitoring` | Через 1-3 и 7 дней после `live-apply` | Read-back + сверка с прогнозом. При правке ставок/стратегии/объявлений — `serp-monitor/track_position.sh` по топ-3 ключам |

**Формат фиксации в начале работы:**

```
Режим: pre-apply-review
Scope: кампания omsk_poisk_vtorichka_odnokomnatnye (id 89167235), окно 7 дней
Подтверждение до apply обязательно.
```

Смена режима в середине работы — только с явного «ок» пользователя.

### Автономный режим (auto vs review)

Внутри `live-apply` есть два подрежима — определяются триггерами в запросе.

| Подрежим | Триггеры в промпте | Поведение |
|---|---|---|
| **review** (по умолчанию) | триггеров auto нет | Агент останавливается на human-checkpoint (C5.5 / R5.5 preview кластеров; R6.5 preview картинок; финальный apply), показывает результат, ждёт «ок» |
| **auto** | «запусти», «сам», «без модерации», «автоматически», «без подтверждения», «до конца», «не спрашивай» | Пропускает preview и apply-подтверждение; идёт весь флоу, фиксирует результат в отчёт в конце |

Правила:
1. Подрежим фиксируется в первом ответе: «Подрежим: auto — выполняю без подтверждений».
2. Смена `review → auto` в середине — только по явному запросу.
3. Даже в `auto` все чеклисты Validation Mesh (Domain/Coverage/Apply-safety) обязательны. Auto ≠ «игнорировать валидацию».
4. `auto` не бывает в `research` и `pre-apply-review`.
5. В `auto` для РСЯ картинки после генерации проходят только автовалидацию (размер, разрешение, формат) без ручного preview.

### Перед live-apply: три чеклиста (Validation Mesh)

**1. Domain validator — корректен ли пакет?**
- Минус-слова не конфликтуют с работающими ключевыми
- Новые ключи не каннибализируют действующие фразы
- Тексты прошли `references/copy_blacklist.md`
- Лимиты (title ≤ 56, title2 ≤ 30, text ≤ 81, callout ≤ 25)
- UTM на уровне группы/кампании

**2. Coverage validator — не пропустил ли блок?**
- Если просили «оптимизировать кампанию» — проверил ли SQR / ключи / минус-слова / объявления / ставки / стратегию / бюджет / расписание / структуру / посадочные?
- Что вынесено в out-of-scope — явно помечено

**3. Apply-safety validator — готов ли к откату?**
- Свежий read-back состояния ДО изменений
- План post-apply read-back готов
- Rollback-сценарий понятен (что делать если CPA вырастет в 2× за 3 дня)
- Окно apply учитывает правило чт-пт для >30% бюджета, смены стратегии/цели, новых объявлений

**Если чеклист не пройден — apply запрещён, блокер назван явно.**

### После live-apply: обязательный отчёт

Работа не считается завершённой без отчёта.

```
### Было (до изменений)
- Кампания <название> (id <id>), режим <стратегия>, бюджет <X> ₽/нед
- [затронутые сущности]
- Метрики до: CPA <X>, CTR <X>%, конверсий/нед <X>

### Сделал
- [точная мутация]: update_campaign(budget=Y)
- Что сознательно не трогал: <список>

### Стало (read-back)
- get_campaigns → State=ON, Status=ACCEPTED (подтверждено API)
- [предупреждения если не прошло модерацию]

### Запись в историю
- update_daily_summary(city="<город>") с итогом дня
- log_change_event(...) — если критичное изменение
```

Обязательный read-back после `resume_campaign`: `get_campaigns(campaign_ids=[id])` с проверкой `State/Status/StatusPayment/StatusClarification`. Без этого запуск не считается подтверждённым.

### Формат результата (Output Contract)

Любой аудит/волна оптимизации — в `campaigns/<slug>.md` по структуре:

1. **Scope** — что проверено (окно, атрибуция LYDC, сущности, что out-of-scope и почему).
2. **Findings** — таблица `entity_type | entity_id | entity_name | verdict | confidence | evidence`. `verdict ∈ {ok, attention, critical, opportunity, waste}`. `confidence ∈ {high, medium, hypothesis-only}`.
3. **Proposed changes** — таблица `change_id | entity | current | proposed | reason | expected_effect | risk | confidence | approval_needed`. `expected_effect ∈ {intraday, 1-3d, 7d+}`.
4. **Applied changes** (если live-apply) — мутации с IDs + read-back + отличия от плана.
5. **Next steps** — монитор-окно + триггеры для реакции.

Для больших аудитов — отдельный `campaigns/<slug>/<YYYY-MM-DD>_audit.md`; в основном файле кампании — ссылка и строка-итог. HTML-отчёт рендерится через `scripts/render_audit_report.py`.

## История изменений (централизованная БД через MCP)

Общая БД на весь агентский аккаунт. Все специалисты пишут в неё — не в локальные файлы скилла.

**Архитектура:**

```
agency_account (etagi click)
    └── city_login (porg-…)
            └── campaign_id (89167235)
                    └── events (change_events) — детальная история
            └── date (2026-04-20)
                    └── daily_summary — краткое саммари дня
```

**БД:** `server/data/change_history.db` (SQLite, git-ignored).
**Слои:** `change_events` (append-only детали), `daily_summaries` (одна запись на город на день).

**MCP-инструменты:**

| Инструмент | Когда вызывать |
|---|---|
| `log_change_event` | После успешного read-back критичного live-изменения (смена цели/стратегии, бюджет >30%, пауза/запуск, новое объявление) |
| `update_daily_summary` | В конце сессии по городу — одним вызовом с итогом дня |
| `get_change_history` | Перед новыми правками по кампании — чтобы видеть контекст |
| `get_daily_summary` | Когда нужен обзор «что делали на этой неделе» |

**Что писать в `log_change_event`:** смена `goal_id` / `priority_goals` / стратегии; изменение недельного бюджета >30%; `suspend/resume/archive_campaign`; новый `add_ad`; массовое добавление минус-слов (>50 за раз). Всё, где нужно будет понять «что я поменял 2 недели назад».

**Что НЕ писать:** повседневные `get_*`; мелкие корректировки текстов; изменения в локальных файлах.

**Перед работой** — `get_change_history(campaign_id, limit=20)` и `get_daily_summary(city_login, limit=7)` для контекста последней недели.
