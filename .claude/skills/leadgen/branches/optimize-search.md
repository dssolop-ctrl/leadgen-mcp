## ВЕТКА 3: ОПТИМИЗАЦИЯ

**Принцип:** ВСЕГДА начинай с данных из ветки АНАЛИЗ. Не оптимизируй вслепую.

<!-- O0 — базовый аудит сущностей -->
### Шаг 0. Базовый аудит всех сущностей (ОБЯЗАТЕЛЬНО)

**Перед любой точечной оптимизацией — проведи полный аудит кампании.** Проверь каждую сущность на соответствие стандартам из ВЕТКИ 1 (Создание).

```
1. ЗАГРУЗИ ВСЕ ДАННЫЕ кампании:
   get_adgroups(campaign_ids=<id>)
   get_ads(campaign_ids=<id>, states="ON",
     text_ad_field_names="Title,Title2,Text,Href,DisplayUrlPath,SitelinkSetId,AdExtensions")
   get_keywords(campaign_ids=<id>)

2. ПРОВЕРЬ КАЖДУЮ ГРУППУ по чеклисту:
   □ Keywords — есть ли ключевые? (без них группа работает только на автотаргетинге)
   □ Autotargeting — только EXACT + ALTERNATIVE? (не все категории включены?)
   □ UTM — tracking_params установлен?
   □ 3 объявления — есть ли A/B тест?
   □ Sitelinks — привязаны к объявлениям? (без них -20-30% CTR)
   □ Уточнения — привязаны к объявлениям?
   □ Заголовки — соответствуют кластеру? Гео в заголовке для гео-групп?
   □ Тексты — в лимитах (Title: 56, Title2: 30, Text: 81)?
   □ Посадочная URL — открывается, есть объекты, фильтры корректны?

3. ПРОВЕРЬ КАМПАНИЮ:
   □ Минус-слова — актуальны? Нет ли пропущенных? (≥ 150 по шаблону mcp/negative_keywords.md)
   □ Стратегия — WB_MAXIMUM_CONVERSION_RATE (старт) / AVERAGE_CPA (зрелая)?
   □ Цели — priority_goals актуальны?
   □ Метки — назначены на все объявления?
   □ ENABLE_AREA_OF_INTEREST_TARGETING — выключен?
   □ **DynamicPlaces** (BiddingStrategy.Search.PlacementTypes.DynamicPlaces) — "NO"?
     Проверь через тип-специфичные field_names (PlacementTypes не читается через обычный field_names):
       get_campaigns(
         campaign_ids=<id>,
         field_names="Id,Name,Type",
         text_campaign_field_names="BiddingStrategy"           // для TEXT_CAMPAIGN
         // dynamic_text_campaign_field_names="BiddingStrategy" // для DYNAMIC_TEXT_CAMPAIGN
         // unified_campaign_field_names="BiddingStrategy,PlacementTypes" // для ЕПК
       )
     → TextCampaign.BiddingStrategy.Search.PlacementTypes.DynamicPlaces
     Если "YES" или пусто — миграция (см. подветку O3.10).
   □ TrackingParams — на уровне кампании, не группы?
     Если у группы заполнено → миграция (см. подветку O3.9).

4. СОБЕРИ ОТЧЁТ:
   | Группа | Keywords | Auto | UTM | 3 ads | Sitelinks | Уточнения | URL ok |
   |--------|---------|------|-----|-------|-----------|-----------|--------|
   | Общий  | ✓ 15    | ✓    | ✓   | ✓ 3   | ✗ нет     | ✗ нет     | ✓      |
   | ...    | ...     | ...  | ... | ...   | ...       | ...       | ...    |

   → Проблемы → конкретные действия по исправлению
```

<!-- O1 — креативный анализ -->
### Шаг 1. Креативный анализ (ОБЯЗАТЕЛЬНО после базового аудита)

**Ты НЕ просто проверяешь по чеклисту.** После базового аудита — ПОДУМАЙ самостоятельно:

```
ОБЯЗАТЕЛЬНЫЕ ВОПРОСЫ (задай себе для каждой кампании):

1. СЕМАНТИКА — полная ли?
   - Все ли районы города покрыты гео-кластерами?
   - Есть ли ценовые кластеры (недорого, от собственника)?
   - Нет ли упущенных интентов? Запусти check_search_volume для проверки.
   - Сравни с аналогичной кампанией в другом городе — что есть у них, чего нет здесь?

2. ЗАГОЛОВКИ — эффективны ли?
   - Вызови `summarize_ads_performance(campaign_id=<id>, date_from=<30 дней>, top_n=10)` → top_by_ctr, low_ctr_candidates_for_ab.
     (Используй `get_ad_stats` только если нужны полные сырые данные.)
   - Есть ли гипотеза почему лидеры по CTR работают? Можно ли применить паттерн к другим группам?
   - Содержат ли гео-кластеры название района в Title?
   - Не устарели ли УТП (цены, акции)?

3. ПОСАДОЧНЫЕ — оптимальны ли?
   - Можно ли добавить фильтры, которых раньше не было в БД?
   - Проверь build_landing_url — появились ли новые district_id?
   - Нет ли лендингов, ведущих на пустую выдачу?

4. СТРУКТУРА — логична ли?
   - Нет ли слишком крупных групп (>40 фраз), которые стоит разбить?
   - Нет ли слишком мелких групп (<3 фразы), которые стоит объединить?
   - Все ли кластеры имеют индивидуальные заголовки?

5. ЧТО ЕЩЁ МОЖНО УЛУЧШИТЬ?
   - Идеи, не описанные в этом скилле
   - Паттерны из успешных кампаний других городов
   - Сезонные корректировки (wordstat_dynamics)
   - Новые типы объявлений или расширений

ФОРМАТ ВЫВОДА:
- Список конкретных предложений с обоснованием
- Для каждого: ожидаемый эффект + сложность реализации
- ПОДТВЕРЖДЕНИЕ пользователя перед реализацией
```

### Подветки (выбор по ситуации)

<!-- O3.1 — минус-слова -->
#### 3.1 Минус-слова (высокий CPA, мусорные запросы)

```
1. summarize_search_queries(
     campaign_ids=<id>, date_from=<30 дней назад>, date_to=<сегодня>,
     goal_ids=<основные цели>, attribution="LYDC", waste_min_clicks=5
   )
   → блок `waste`: запросы с кликами ≥5 и конверсиями=0, отсортированы по cost.
   → блок `top_by_cost`: где идёт расход без эффекта.
   Для полного TSV (если нужно копать глубже) — get_search_queries.
2. Информационные запросы → в минус (сверить с минус-словами из get_semantic_cluster)
3. ПОДТВЕРЖДЕНИЕ пользователя перед применением
4. update_campaign(campaign_id, negative_keywords=<список>)
```

<!-- O3.2 — новые ключевые -->
#### 3.2 Новые ключевые (упущенный спрос)

```
1. summarize_search_queries(..., goal_ids=<цели>) → блок `top_by_conversions`
   = запросы где есть конверсии. Сверить с действующими ключевыми кампании.
   Те, что конвертят, но не покрыты ключами — кандидаты на add_keywords.
2. check_search_volume(phrases="<найденные фразы>", region_ids=<гео>)
   → Проверить частотность + собрать SearchedWith для расширения
3. Отфильтровать нерелевантные (см. ВЕТКА 1, Шаг 4, Раунд 2 — логика фильтрации)
4. ПОДТВЕРЖДЕНИЕ пользователя (показать таблицу: фраза + Shows)
5. add_keywords(adgroup_id=<id>, keywords="<фразы через запятую>")
```

<!-- O3.3 — A/B тест объявлений -->
#### 3.3 A/B тест объявлений (низкий CTR)

```
1. summarize_ads_performance(campaign_id=<id>, date_from=<30 дней>, top_n=10)
   → блок `low_ctr_candidates_for_ab` = объявления с показами ≥200 и
     самым низким CTR. Это и есть слабые звенья.
2. Загрузить library/titles.md, library/texts.md
3. SERP-разбор конкурентов (опц., если tokens.env настроен):
   - Взять 3-5 топ-ключей группы (из get_keywords, сортировка по показам)
   - Запустить .claude/skills/serp-monitor/scripts/search.sh --query "<ключ>" --region <geo>
   - Выжать из top-8: повторяющиеся УТП (цена, ипотека, срок), CTA, эмодзи/триггеры
   - Найти «белые пятна» — что никто не использует и может выделить
4. Создать новый вариант с гипотезой (учитывая SERP-разбор, если был)
5. ПОДТВЕРЖДЕНИЕ пользователя
6. add_ad(adgroup_id, ...)
7. Через 2-3 недели: сравнить, остановить проигравшего
```

<!-- O3.4 — смена стратегии -->
#### 3.4 Смена стратегии (достаточно конверсий)

```
Условие: 10+ конверсий/нед стабильно 2-3 недели подряд
1. ПОДТВЕРЖДЕНИЕ пользователя (лучше в чт-пт)
2. update_campaign(search_strategy="AVERAGE_CPA", average_cpa=<целевой CPA>)
```

<!-- O3.5 — бюджет -->
#### 3.5 Бюджет

```
- Увеличение: не более +30% за раз, после 2+ недель стабильного CPA
- Снижение: если CPA выше нормы и оптимизация не помогла
- Сезонность: wordstat_dynamics → скорректировать по сезону
- ПОДТВЕРЖДЕНИЕ пользователя для любых изменений > 10%
```

<!-- O3.6 — кросс-минусация -->
#### 3.6 Кросс-минусация (каннибализация между группами)

**Используй когда:** группы внутри одной кампании конкурируют за одни и те же запросы — трафик уходит в менее релевантную группу.

```
1. ДИАГНОСТИКА:
   get_keywords(campaign_ids=<id>) → собери все ключевые по группам
   get_search_queries(campaign_ids=<id>, date_from=<30 дней>, attribution="LYDC")
   → Найди запросы, которые сработали в НЕСКОЛЬКИХ группах
   → Определи: в какой группе запрос более релевантен (по заголовку/посадочной)

2. ПОСТРОЕНИЕ кросс-минусов:
   - Для каждой группы определи уникальные слова-маркеры
     (район, тип объекта, ценовой модификатор)
   - Эти маркеры → минус-слова для ВСЕХ остальных групп

3. ПРИМЕНЕНИЕ:
   add_negative_keywords(
     client_login=<login>,
     adgroup_id=<id>,
     keywords="<чужие маркеры через запятую>"
   )

4. ПРОВЕРКА:
   - Ни одна ключевая фраза группы не содержит собственные минус-слова
   - Широкая группа ("Общий") минусует ВСЕ уточнения из специализированных групп
```

<!-- O3.8 — привязка sitelinks/уточнений к существующим объявлениям -->
#### 3.8 Привязка sitelinks/уточнений к существующим объявлениям

**Используй когда:** аудит показал, что у объявлений пустой `AdExtensions` или
нет `SitelinkSetId` — например, в старой кампании или после миграции.

```
1. ДИАГНОСТИКА:
   get_ads(campaign_ids=<id>, states="ON",
     text_ad_field_names="SitelinkSetId,AdExtensions")
   → Найди объявления без привязки (пустой массив AdExtensions, отсутствует SitelinkSetId)

2. ПОДГОТОВКА НАБОРОВ (если их ещё нет на аккаунте):
   - Быстрые ссылки: add_sitelinks(...) → sitelink_set_id
   - Уточнения: add_ad_extension(callout_text=...) × 4-8 штук → ad_extension_ids
   - Повторно используемые наборы искать через get_sitelinks / get_ad_extensions

3. ПРИВЯЗКА — по одному update_ad на объявление:
   update_ad(
     client_login=<login>,
     ad_id=<banner_id>,
     sitelink_set_id=<sitelink_set_id>,
     ad_extension_ids="<id1>,<id2>,<id3>,..."
   )
   // update_ad транслирует ad_extension_ids в CalloutSetting SET (полная замена набора).
   // Можно передать оба поля одним вызовом — или только одно, если второе уже есть.

4. ПРОВЕРКА:
   get_ads(ad_ids="<список>",
     text_ad_field_names="SitelinkSetId,AdExtensions")
   → Убедись что у всех объявлений AdExtensions заполнен и SitelinkSetId != 0
```

> **ПРАВИЛО:** для НОВЫХ кампаний — всегда делай привязку через `add_ad`
> (параметры `sitelink_set_id` и `ad_extension_ids`), а `update_ad` используй
> ТОЛЬКО для ретрофита старых объявлений или смены набора уточнений.

<!-- O3.9 — миграция UTM с уровня группы на уровень кампании -->
#### 3.9 Миграция UTM с уровня группы на уровень кампании

**Используй когда:** старые кампании имеют `tracking_params` на группах (старая схема)
вместо кампании. Миграция упрощает редактирование UTM — одно место вместо N групп.

```
1. ДИАГНОСТИКА:
   get_campaigns(campaign_ids=<id>, field_names="Id,Name,TextCampaign")
   → посмотри TextCampaign.TrackingParams — пусто?

   get_adgroups(campaign_ids=<id>, field_names="Id,Name,TrackingParams")
   → собери все tracking_params групп

   Если campaign.TrackingParams пустой И у всех групп одинаковый UTM → годится для миграции.
   Если у групп РАЗНЫЕ UTM — сначала уточни с пользователем: они осознанно разные или
   это исторический дрейф?

2. ПОДТВЕРЖДЕНИЕ пользователя (хард-гейт: изменение UTM = изменение атрибуции).

3. ПРИМЕНЕНИЕ:
   // Шаг 3a: поставить UTM на кампанию
   update_campaign(
     client_login=<login>,
     campaign_id=<id>,
     tracking_params=<общий UTM из get_utm_config>
   )

   // Шаг 3b: очистить UTM с ВСЕХ групп кампании
   for adgroup_id in adgroup_ids:
     update_adgroup(
       client_login=<login>,
       adgroup_id=<adgroup_id>,
       clear_tracking_params=true
     )

4. ПРОВЕРКА:
   get_campaigns(campaign_ids=<id>, field_names="Id,TextCampaign")
     → TextCampaign.TrackingParams заполнен
   get_adgroups(campaign_ids=<id>, field_names="Id,TrackingParams")
     → все TrackingParams пустые

5. ФИКСАЦИЯ в campaigns/<slug>.md: отметить миграцию UTM с датой.
```

> **Риск:** атрибуция Директа и Метрики непрерывна — миграция не ломает статистику,
> т.к. tracking_params рендерится в URL при клике, а не сохраняется в логах задним числом.
> Но делать лучше в чт-пт и после подтверждения пользователя.

<!-- O3.10 — отключение динамических мест показа -->
#### 3.10 Отключение динамических мест показа (DynamicPlaces=NO)

**Используй когда:** аудит O0 показал, что у кампании `DynamicPlaces` = "YES" или не установлен.
По договорённости с SEO — ВСЕГДА должно быть "NO".

**Контекст:** «Динамические места показа» — это показы в дополнительных местах
(Яндекс.Карты, Маркет, другие сервисы и партнёры). SEO Этажи договорились с
маркетингом: такие показы отключаем — они размывают атрибуцию и пересекаются
с органикой.

API-путь (из кабинета): `BiddingStrategy.Search.PlacementTypes.DynamicPlaces`
Значения: `YES` / `NO`. Применимо к `TextCampaign`, `DynamicTextCampaign`, `UnifiedCampaign` (ЕПК).

```
1. ДИАГНОСТИКА:
   // Сначала определи тип кампании:
   get_campaigns(campaign_ids=<id>, field_names="Id,Name,Type")
   → Type: TEXT_CAMPAIGN / DYNAMIC_TEXT_CAMPAIGN / UNIFIED_CAMPAIGN / ...

   // Затем запроси PlacementTypes через соответствующий type-specific FieldNames:
   get_campaigns(
     campaign_ids=<id>,
     field_names="Id,Name,Type",
     text_campaign_field_names="BiddingStrategy"            // если Type=TEXT_CAMPAIGN
     // dynamic_text_campaign_field_names="BiddingStrategy" // если DYNAMIC_TEXT_CAMPAIGN
     // unified_campaign_field_names="BiddingStrategy,PlacementTypes" // если UNIFIED_CAMPAIGN (ЕПК)
   )
   → TextCampaign.BiddingStrategy.Search.PlacementTypes.DynamicPlaces
   Если "YES" или отсутствует → требуется выключение.

   ⚠️ НЮАНС API: обычный field_names НЕ возвращает PlacementTypes — он вложен в
   TextCampaign.BiddingStrategy. Поэтому нужен отдельный text_campaign_field_names.

2. ПРИМЕНЕНИЕ (без подтверждения, т.к. это жёсткая договорённость):
   update_campaign(
     client_login=<login>,
     campaign_id=<id>,
     search_strategy=<текущая стратегия, передать для активации блока>,
     disable_dynamic_places=true
   )
   // disable_dynamic_places=true → search.PlacementTypes.DynamicPlaces = "NO"

   ⚠️ ВАЛИДАЦИЯ: API при update валидирует объект целиком. Если у кампании
   есть ограничения по сочетанию стратегии/мест (например, нельзя отключить
   DynamicPlaces при определённой стратегии) — вернётся ошибка валидации.
   В таком случае:
     a) Прочитай текущее состояние (шаг 1).
     b) Покажи пользователю, попроси согласовать изменение стратегии.
     c) Если нельзя изменить — зафиксируй в campaigns/<slug>.md как исключение.

3. ПРОВЕРКА:
   get_campaigns(
     campaign_ids=<id>,
     field_names="Id,Type",
     text_campaign_field_names="BiddingStrategy"  // или соответствующий тип
   )
   → убедись, что DynamicPlaces = "NO"

4. ФИКСАЦИЯ в campaigns/<slug>.md — отметь миграцию с датой.
```

> **Важно:** в `add_campaign` этот флаг уже включён по умолчанию — новые кампании
> автоматически создаются с `DynamicPlaces=NO`. Эта подветка нужна только для
> миграции существующих кампаний.

<!-- O3.7 — расширение структуры -->
#### 3.7 Расширение структуры (добавление групп в существующую кампанию)

**Используй когда:** нужно добавить новые группы объявлений в уже работающую кампанию (новые районы, новые кластеры, новая семантика).

```
Шаг А. Контекст существующей кампании (ОБЯЗАТЕЛЬНО)

1. Получи текущие группы, объявления, ключевые:
   get_adgroups(campaign_ids=<id>)
   get_ads(campaign_ids=<id>, states="ON",
     text_ad_field_names="Title,Title2,Text,Href")
   get_keywords(campaign_ids=<id>)

2. ОПРЕДЕЛИ СКОУП кампании из названия и ключевых:
   - Кампания "Вторичка | Однокомнатные" → собирай семантику ТОЛЬКО по однокомнатным
   - Кампания "Вторичка" (без уточнения) → все типы комнатности допустимы
   - Кампания "Загородка | Дома" → только дома, не участки и не дачи

3. ФИЛЬТРУЙ семантику по скоупу:
   - Если кампания по однокомнатным — исключи из ключевых: "2 комнат", "двушк",
     "двухкомнат", "3 комнат", "трешк", "трехкомнат", "4 комнат", "студи"
   - Если кампания по домам — исключи: "участ", "дач", "таунхаус"
   - Применяй фильтр НА КАЖДОМ этапе: при сборе seeds, при обработке SearchedWith,
     при финальной проверке перед кластеризацией
```
