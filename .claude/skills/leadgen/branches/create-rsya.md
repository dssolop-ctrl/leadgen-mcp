## ВЕТКА 1-R: СОЗДАНИЕ РСЯ-кампании

> **Когда использовать:** пользователь явно запросил РСЯ (или «в сетях», «баннерная», «рекламная сеть яндекса»).
> Роутер 2 (см. `../skill.md`) должен определить канал ДО входа сюда.

**Принципиальные отличия от ветки C (поиск):**

- Тип кампании — тот же, **ЕПК (UNIFIED_CAMPAIGN)**, как и для поиска. Различие — только в стратегиях `BiddingStrategy.Search`/`Network` (см. ниже). См. `METRIKA-ADS-RULES.md` → «Тип кампании: всегда ЕПК».
- Семантика ШИРЕ, не транзакционно-точная.
- Кластеризация по АУДИТОРИИ, не по интенту.
- Объявления = TEXT_AD + **картинка (AdImageHash)**. Картинки генерируем через OpenRouter (Flux / Gemini).
- Бюджет консервативный — см. `references/rsya_defaults.md` раздел 2.
- `search_strategy=SERVING_OFF`, `network_strategy=WB_MAXIMUM_CONVERSION_RATE` (старт).
- Лимит 20 генераций картинок на кампанию, переиспользование между группами.

**Перед началом — загрузи в контекст:**
- `references/rsya_defaults.md` — все дефолты настроек.
- `references/image_prompts.md` — промпт-билдер V1–V7.
- `library/banner_titles.md`, `library/banner_texts.md` — тексты РСЯ.

### Шаг R1. Определение параметров (РСЯ)

Спроси у пользователя (если не указал):

1. **Город** — какой город.
2. **Тематика** — вторичка / новостройки / загородка / ипотека / аренда / коммерческая / бренд / HR.
3. **Сегмент** — опционально (1-комн / 2-комн / конкретный ЖК / котлован / с отделкой / эконом / премиум).
4. **Бюджет** — если не задан, рассчитать по формуле из `rsya_defaults.md` раздел 2: `CPA × 10 × 1.2`.
5. **Стиль визуала** — опционально. Если не задан, выбирается по тематике (см. маппинг в `image_prompts.md`):
   - Вторичка / новостройки (купитель) → V1 (интерьер) + V3 (фасад)
   - Загородка → V4 (частный дом) + V5 (участок)
   - Ипотека → V7 (ключи) + V1
   - Коммерческая → V6
6. **Посадочная** — [site] (etagi.com) или [LP] (etagi.net). По дефолту — site с фильтрами через `build_landing_url`.
7. **Подрежим** — auto или review (см. «Автономный режим» в процессном контуре).

#### Шаг R2.5. Sanity-check benchmark vs network — детерминированный выбор target

⚠️ **Никогда не используй `BenchmarkFormCPA` напрямую как target без этого шага.** Шаг детерминированный — в режиме `auto` выполняется без пауз, в режиме `review` показывается пользователю с возможностью переопределить.

```
ratio = BenchmarkFormCPA / NetworkExpected[theme/tier]
```

Где `NetworkExpected` — значение из `server/data/network_benchmarks.json` (для вторички tier_1=3500, tier_2=2500, tier_3=1800; tier виден через `get_city_config(...).tier`).

**Правило выбора target (применяется в обоих режимах автоматически):**

| ratio | target_form_cpa | Комментарий |
|---|---|---|
| < 0.7 | `BenchmarkFormCPA` | benchmark ниже ожидаемого — норма (хорошая работа аккаунта). |
| 0.7 — 1.3 | `BenchmarkFormCPA` | benchmark в норме. |
| 1.3 — 2.0 | **`NetworkExpected`** | Ориентируемся на норму tier-а. Аккаунтный benchmark задран — не наследуем дорогие исторические периоды в новую кампанию. |
| > 2.0 | `NetworkExpected × 0.9` | Сильная аномалия (учётка с высоким спендом без атрибуции / тестовый период / неполные данные). Стартуем чуть ниже нормы, чтобы автостратегия не пыталась тянуть исторический потолок. |

То же правило — для `BenchmarkCallCPA` vs `NetworkExpected.call`. tCPA звонка всегда ниже tCPA формы.

**Как фиксировать применённое решение:**

В файле кампании (раздел «Решения по target») всегда отрази:
- `BenchmarkFormCPA` (что вернул MCP);
- `NetworkExpected` (для своего tier-а);
- `ratio`;
- какой бакет применён и итоговый `target_form_cpa`.

Это нужно для прозрачности при пост-старт-аудите и для пересчёта target после накопления первых конверсий.

**Режимы:**
- `auto` — применить правило без вопросов, проследовать дальше. Решение зафиксировать в файле кампании.
- `review` — показать пользователю расчёт + предложенный target, дождаться «ок» либо явного override (`target_form_cpa_override`).

**После выбора:**
```
target_form_cpa = <по правилу выше>
target_call_cpa = <по правилу выше>
tCPA_РСЯ_form  = target_form_cpa × 0.7   # коэффициент РСЯ от поискового
tCPA_РСЯ_call  = target_call_cpa × 0.7
priority_goals_json = [
  {"goal_id": form_goal, "value": tCPA_РСЯ_form},
  {"goal_id": call_goal, "value": tCPA_РСЯ_call}
]
```

#### Шаг R2.6. Расчёт BidCeiling (см. полную формулу в `references/rsya_defaults.md`)

⚠️ BidCeiling — это **потолок цены клика**, не CPA конверсии. **НЕ использовать `tCPA × 1.5`** — даёт ставки в 5–15 раз выше рыночных.

```
expected_click = tCPA_РСЯ_form × 0.05      # CR_estimate для РСЯ-недвижимости 5%
BidCeiling     = expected_click × 2.5       # страховой запас
BidCeiling_final = clamp(BidCeiling, tier_min, tCPA_РСЯ_form × 0.30)
```

`tier_min`: tier_1=80, tier_2=40, tier_3=20.

Пример (Омск, tier_2, форма target 2500, tCPA РСЯ = 1750):
- expected_click = 1750 × 0.05 = 87.5
- BidCeiling = 87.5 × 2.5 = 219
- clamp с tier_min=40, max=525 → **219 ₽**.

Если есть фактический CR из истории — используй его вместо 0.05.

<!-- R4 — семантика для РСЯ -->
### Шаг R4. Семантика для РСЯ (широкая)

**Принцип:** в РСЯ показываемся на тематических сайтах — нужны фразы **категории**, а не точные транзакционные.

```
1. get_semantic_cluster(theme="<тематика>") → база модификаторов и объектов
2. Wave 1: тематические фразы БЕЗ города, с широкими модификаторами
   Пример для вторички: "купить квартиру", "квартиры", "двушка",
   "вторичка", "квартира цена", "квартира ипотека"
   → 15-25 фраз (меньше чем в поиске)
3. check_search_volume(phrases=..., region_ids=<город>)
   → оставить фразы с Shows ≥ 100 (в РСЯ важен охват, не точность)
4. Добавить интересы/автотаргетинг через update_autotargeting
   (в R7 - EXACT=YES, ALTERNATIVE=YES)
5. Минус-слова: из `rsya_defaults.md` раздел 9 (инфо-интент блокируется)
```

**НЕ делаем в РСЯ:**
- Мелко-частотные long-tail (< Shows 50) — лишние фразы не дают охват, только шум.
- Точные конкурентные [циан, авито] — в РСЯ не работает как в поиске.
- Гео-районы в ключах — работает через region_ids.

**Результат:** 10–30 широких фраз на кампанию.

<!-- R5 — кластеризация по аудитории -->
### Шаг R5. Кластеризация по аудитории

⚠️ **Сначала кластеризация — потом группы.** Не предрешай группы по сегментам, упомянутым пользователем («1-комн и 2-комн»). Это сигнал, не конечное решение. Решение — после R4 (семантика собрана) и проверки посадочных.

**Алгоритм:**

1. Из R4 получены 10–30 широких фраз. Сгруппируй их по смысловой однородности (это «семантические кластеры» — не путай с группами Директа).
2. Для каждого упомянутого пользователем сегмента (если есть) проверь **жёсткое правило**:
   - **≥3 фразы в R4 под этот сегмент.**
   - Если «да» — сегмент валиден как отдельная группа.
   - Если «нет» — сегмент **сворачивается в общую группу**.
3. Дополнительно (мягкий сигнал, **не блокирующий**): проверь, есть ли посадочная под сегмент через `build_landing_url(...)` / `get_site_filters(...)`.
   - Есть посадочная → используй её как `href` объявлений группы.
   - Нет посадочной → используй базовую посадочную тематики (например, `/realty/` для вторички). Группу всё равно создаём, если правило по фразам выполнено.
4. **Базовая группа — `Общие-купить`** — всегда. Туда уходят все фразы, которые не попали в валидные сегментные группы, + автотаргетинг EXACT+ALTERNATIVE.
5. Опциональные «аудиторные» группы (LAL / Ретаргет / Интересы) — **не создаём в первой кампании**, если пользователь явно не запросил расширенную/премиум-конфигурацию.

**Типы групп:**

| Группа | Что внутри | Когда создавать |
|---|---|---|
| **Общие-купить** | Все фразы из R4 + автотаргетинг EXACT+ALTERNATIVE | Всегда (базовая). |
| **Сегментная** (1-комн / 2-комн / котлован / конкретный ЖК / премиум) | Фразы под сегмент | Только если: **≥3 фраз в R4** под сегмент. Посадочная — желательна, но не обязательна (fallback на базовую). |
| **LAL** | Те же фразы + look-alike сегмент Метрики | Опция: > 5 000 визитов/мес + явный запрос на «расширенную». |
| **Ретаргет** | Те же фразы + retargeting_list «был на сайте, не оставил заявку» | Опция: при истории трафика + явный запрос. |
| **Интерес-премиум** | Премиум-фразы + аудиторные интересы «Недвижимость» | Опция: Москва/СПб/премиум-ЖК + явный запрос. |

**Минимум для первой кампании:** одна группа `Общие-купить`. Сегментные добавляются ТОЛЬКО при выполнении правила «≥3 фразы в R4».

**Анти-паттерны (НИКОГДА):**

- ❌ Создать группы «1-комн» и «2-комн» только потому, что пользователь упомянул их в запросе. Если в семантике R4 нет ≥3 специализированных фраз — сегмент сворачивается в `Общие-купить`.
- ❌ Решить структуру групп до R4 (семантика). R4 → R5, не наоборот.
- ❌ Сворачивать сегмент только из-за отсутствия посадочной фильтра — посадочная это soft-сигнал; если фраз достаточно, выделение в группу осмыслено и без специализированной посадочной (используй базовую тематическую).

Пользователю показываем таблицу кластеров перед созданием:

```
| Группа | Фразы (N) | Посадочная | Аудиторный таргет | Объявлений |
|---|---:|---|---|---:|
| Общие-купить | 22 | /realty/ | автотаргетинг | 7 |
| 2-комн | 5 | /realty/dvuhkomnatnye-kvartiry/ (или /realty/ если нет фильтра) | — | 5 |
```

Если пользователь упомянул сегменты, но они не прошли проверку «≥3 фраз» — явно отрази в таблице:
```
| Сегменты «1-комн», «2-комн» | свёрнуты в Общие-купить (в R4 < 3 специализированных фраз) |
```

<!-- R6.5 — генерация картинок -->
### Шаг R6.5. Генерация картинок (OpenRouter)

**Лимит: 20 генераций на всю кампанию.**

```
1. Определить стилевой набор: 3-4 визуала (V1–V7) × 2 формата (1:1 + 16:9)
2. Для каждого: собрать промпт через промпт-билдер из image_prompts.md
   с инъекцией: city, theme, segment, city_hint
3. Вызвать MCP generate_banner_set(
     visual_type="V1.1",
     aspect_ratios="1:1,16:9",
     n_variants=1,
     context='{"city":"омск","theme":"вторичка",...}',
     model="black-forest-labs/flux.2-pro"  # дефолт для V1/V3/V4
   )
   → возвращает список путей + width/height + при необходимости fallback_used и aspect_warning

   ⚠️ Автоматический фолбэк: если первичная модель отклоняет запрос
   по модальностям (image-only Flux при ["image","text"]) или возвращает
   4xx — клиент сам перезапускается на google/gemini-3-pro-image-preview
   и возвращает результат с fallback_used=true + primary_error=<...>.
   Это нормально — отражай в файле кампании.

4. Автовалидация выхода:
   • Если в ответе есть aspect_warning — модель проигнорировала image_size
     (типично для Gemini), Директ почти наверняка отдаст «Размер изображения
     некорректен». Вариант (а) перегенерить с другим промптом, (б) отресайзить
     локально (файл лежит в file_path), (в) сменить модель на flux.2-pro.
   • size_bytes ≤ 10 MB, ≥ 10 KB
   • 1:1 → 450×450 — 5000×5000; 16:9 → 1080×607 — 5000×2812 (±2% по соотношению)

5. В режиме review — положить в docs/campaign_previews/<slug>/ → показать пользователю
   В режиме auto — пропустить preview, идти в R7
6. Если картинка не прошла валидацию — regen (лимит 5 регенераций на сцену из общего 20)
   Учёт квоты: неуспешные API-вызовы OpenRouter (4xx до фолбэка) в квоту НЕ идут — биллинг
   идёт только за реально полученный пиксель. Но фиксируй попытки в файле кампании
   (раздел «Креативы» → «Попытки») для прозрачности.
```

**Переиспользование:** одна картинка используется в нескольких группах через AdImageHash.
Пример: V3 фасад ЖК → группы «Общие-купить» + «LAL» + «2-комн» — одна картинка на три.

<!-- R7 — создание через API -->
### Шаг R7. Создание через API

```
1. add_campaign(
     name="<Город> | РСЯ | <Тематика> | <Сегмент> | [site|lp]",
     search_strategy="SERVING_OFF",
     network_strategy="WB_MAXIMUM_CONVERSION_RATE",
     network_bid_ceiling=<BidCeiling из R2.6, формула tCPA × 0.125 c clamp по tier-у>,
     priority_goals=<priority_goals_json из R2.5 — value=tCPA для каждой цели>,
     daily_budget_amount=<weekly_rub из R1/get_default_budgets>,
     counter_ids=<из R2>,
     tracking_params=<из R2>,
     negative_keywords=<summary_string из get_negative_keyword_guidance>,
     settings='[{"option":"ENABLE_AREA_OF_INTEREST_TARGETING","value":<true|false>}]'
   )
   → campaign_id

   ⚠️ ПРОВЕРКА ПЕРЕД ВЫЗОВОМ:
   - network_bid_ceiling ≤ tCPA × 0.30 (если выше — пересчитай R2.6, что-то не так)
   - priority_goals.value = tCPA, НЕ BidCeiling (типичная путаница)
   - daily_budget_amount в рублях (не микроюниты), недельный

2. Для каждой группы (из R5):
   a. add_adgroup(campaign_id, name, region_ids)
   b. add_keywords(adgroup_id, keywords=<из R4>)
   c. update_autotargeting(adgroup_id, exact="YES", alternative="YES")
   d. Для картинок группы:
      - add_ad_image(file_path=<путь из R6.5>, name=<имя>) → AdImageHash
        (или использовать уже загруженный hash из предыдущей группы)

      ⚠️ add_ad_image выполняет локальную валидацию ДО похода в API:
        формат (JPG/PNG/GIF), вес (10 KB — 10 MB), пиксели,
        соотношение 1:1 или 16:9 ±2%. При несоответствии вернёт ошибку
        с указанием конкретной причины (а не общее «IncorrectImageSize»
        от Директа). Если упёрся — ресайз/кроп локально и retry.
   e. **add_ad × N**, где N зависит от типа группы (см. `references/rsya_defaults.md` раздел 7):
      - стандартная группа (Общие-купить, сегментная): **5–8 объявлений** (минимум 5)
      - LAL-группа (если создана): **до 45 объявлений** — питает ML-стратегию
      - Каждое объявление = свой комбо из (заголовок 1–4) × (текст 1–3) × (визуал 1:1 или 16:9)
      - НЕ создавать только 3 объявления — это ниже порога обучения авто-стратегии РСЯ.
   f. add_sitelinks, add_ad_extension (через ad_extension_ids)

3. Если пользователь запросил ретаргет/LAL (R5 опция):
   add_audience_targets(adgroup_id, ..., audience_id)

4. Read-back: get_campaigns с text_campaign_field_names="BiddingStrategy,TrackingParams" — подтвердить настройки.
   Дополнительно: `summarize_campaign_snapshot(campaign_id)` — компактная сводка для read-back пользователю.
```

### Шаг R11. Финальное состояние: DRAFT и план мониторинга

**Кампания после R7 остаётся в DRAFT/SUSPENDED — ты её НЕ запускаешь.**
См. правило в `METRIKA-ADS-RULES.md` → «Новая кампания всегда остаётся в DRAFT».
`resume_campaign` вызывает только пользователь после ручного ревью.

```
1. get_campaigns(campaign_ids=<id>, field_names="Id,Name,State,Status") →
   ожидаемое состояние: State=SUSPENDED (или OFF), Status=DRAFT/MODERATION/ACCEPTED.
   State=ON при создании — ОШИБКА, сразу suspend_campaign и разбирайся.
2. update_daily_summary(city) с итогом — что создано, в каком состоянии.
3. log_change_event — если критичное изменение в существующей кампании.
4. Передай пользователю краткий чек-лист для ручного ревью:
   - тексты объявлений (заголовки, тексты, sitelinks, callouts);
   - креативы (V1.1, V3 и т.д. — превью в docs/campaign_previews/<slug>/);
   - посадочная (живая ли, фильтры применяются);

   - Через 3 дня от старта: чистка площадок (O-RSYA / OR.1)
   - Через 7 дней от старта: первая оценка CPA и конверсий
```
