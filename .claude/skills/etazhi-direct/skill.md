# Этажи Директ — управление рекламой недвижимости

Специализированный скилл для агентства недвижимости «Этажи». Оркестрирует MCP-инструменты Яндекс Директа, Метрики и Wordstat для создания, анализа и оптимизации рекламных кампаний.

**Компания:** Этажи — крупнейшее агентство недвижимости России. 70+ городов, домен etagi.com.
**MCP-сервер:** `leadgen-mcp` (Яндекс Директ + Метрика + Wordstat). Endpoint и режим подключения — см. [`config/mcp_endpoint.md`](config/mcp_endpoint.md).
**Аккаунт Директа:** `etagi click` (default) — агентский, 76 клиентов-городов.
**Аккаунт Метрики:** `etagi.agent` — 467 счётчиков, 70 активных городов.

### Базовые тематики

| Тематика | Направление | Посадочная | Ключевые кластеры |

| Аренда | Арендатор | /realty_rent/ | снять квартиру |
| Аренда | Арендодатель | /for_the_lessor/ | сдать квартиру |
| Коммерческая | Покупатель | /commerce/ | офис, торговое помещение, склад |
| Гаражи/Паркинги | Покупатель | /realty/?type=garage | купить гараж, паркинг |

### Направление: покупатель vs продавец

- **Покупатель** (95% кампаний) — человек ищет объект для покупки/аренды. Посадочная — каталог с фильтрами.
- **Продавец** (5%) — человек хочет разместить свой объект через Этажи. Посадочная — форма оценки/размещения.
- Это РАЗНЫЕ кампании с разными ключевыми, текстами и посадочными. Не смешивать.

### Шаг 4. Сбор семантики (многораундовое расширение)

**Цель:** 30–80 ключевых фраз для типичной кампании. 14 фраз — слишком мало, 200+ — переспам. Ядро из get_semantic_cluster — лишь отправная точка.

#### Раунд 1: Базовая матрица (seeds)

```
1. Вызови get_semantic_cluster(theme="<тематика>")
   → modifiers, objects, extra_keys, negative_keywords

2. Построй матрицу: {модификатор} × {объект} + {город}
   Пример (вторичка, Омск):
     "купить квартиру омск", "купить однушку омск", "купить двушку омск",
     "квартира омск цена", "однокомнатная омск вторичка", ...
   → ~10-20 seed-фраз (не больше, это входные данные для расширения)

3. Если есть extra_keys в кластере — добавь их с подстановкой города
```

#### Раунд 2: Расширение через Wordstat (КЛЮЧЕВОЙ ЭТАП)

```
1. Вызови check_search_volume(
     phrases="<все seed-фразы через запятую>",
     region_ids="<geo_region_id города>"
   )
   → API вернёт для каждой фразы блок SearchedWith — реальные запросы пользователей

2. Из SearchedWith собери ВСЕ фразы с Shows >= 10 (для малых городов >= 5)

3. Какие фразы ценны в SearchedWith:
   - Гео-уточнения: район, микрорайон, улица ("квартира кировский район омск")
   - Ценовые: "до 2 млн", "недорого", "дешевые"
   - Типовые уточнения: "с балконом", "с ремонтом", "от собственника"
   - Синонимы: "однушка" ↔ "1 комнатная" ↔ "однокомнатная"
   - Long-tail: "купить однокомнатную квартиру омск вторичное жилье"

4. Отфильтруй нерелевантные — сравни с negative_keywords из кластера:
   - Аренда/сдача: снять, сдам, аренда, посуточно
   - Информационные: как, почему, отзывы, рейтинг, форум
   - Чужие бренды: авито, циан, домклик (если не в наших ключевых)
   - Другая тематика: новостройка, застройщик, жк (для вторички)
   - Нецелевые: ремонт, дизайн, курсовая, реферат
```

#### Раунд 3: Второй уровень расширения (опционально)

```
Если в Раунде 2 обнаружены высокочастотные фразы (Shows > 100),
которых НЕ было в seeds — запусти ещё один check_search_volume по ним.

Пример: seed "купить квартиру омск" → SearchedWith показал
"квартира левый берег омск" (Shows 320) — это новый кластер,
у которого свои вложенные запросы.

Лимит: не более 2 раундов расширения (seeds → round2 → round3).
Больше — diminishing returns.
```

#### Раунд 4: Финализация и подготовка к кластеризации

```
1. Объедини все собранные фразы, убери дубли (регистр, порядок слов)
2. Отсортируй по Shows (частотности) — от большей к меньшей
3. Проверь каждую на релевантность:
   - Соответствует тематике кампании?
   - Покупательский интент (не информационный)?
   - Подходит к посадочной странице?

4. Собери итоговый список минус-слов:
   - negative_keywords из get_semantic_cluster
   - + новые нерелевантные паттерны, обнаруженные в SearchedWith
   - Минус-слова идут на уровень кампании (negative_keywords в add_campaign)

5. Результат → массив фраз [{phrase, shows}] для Шага 5 (кластеризация)
```

**Важно:** check_search_volume принимает до 10 фраз за раз (лимит API). Разбивай на батчи по 10. Всегда передавай region_ids города — без гео данные будут по всей России и нерелевантны.

<!-- C5 — кластеризация -->
### Шаг 5. Кластеризация семантики → структура групп объявлений

**Ключевой шаг.** Собранную семантику нельзя складывать в одну группу — нужно разбить на кластеры, каждый из которых станет отдельной группой объявлений с индивидуальным контентом.

#### Принципы кластеризации

```
Агент САМОСТОЯТЕЛЬНО решает, как разбить семантику. Ориентиры:

1. ГЕО-КЛАСТЕРЫ (приоритет если обнаружены)
   Если в SearchedWith найдены фразы с районами/микрорайонами — это отдельные кластеры.
   Каждый район → своя группа с:
   - Названием района в заголовке объявления
   - Фильтром района в посадочной URL (если сайт поддерживает, см. build_landing_url)
   - Ключевыми только по этому району

   Пример (Омск, вторичка, однокомнатные):
   | Кластер | Ключевые (примеры) | Заголовок | Посадочная |
   |---------|-------------------|-----------|------------|
   | Нефтяники | "квартира нефтяники омск", "однушка нефтяники" | Однушки в Нефтяниках | /realty/?rooms=1&district=... |
   | Кировский | "квартира кировский район омск" | Однушки в Кировском | /realty/?rooms=1&district=... |
   | Левый берег | "квартира левый берег омск" | Однушки на Левом берегу | /realty/?rooms=1&district=... |

2. ИНТЕНТ-КЛАСТЕРЫ
   - «Купить» (прямой интент) — "купить квартиру омск"
   - «Вторичка/от собственника» (уточнённый) — "вторичка омск", "от собственника"
   - «Цена/недорого» (ценовой) — "квартира омск недорого", "дешевые квартиры"

3. ОБЪЕКТНЫЙ КЛАСТЕР (если кампания шире одного типа)
   - Однушки, двушки, студии — если все в одной кампании

4. ОБЩИЙ кластер (catch-all)
   Широкие фразы без уточнений: "квартира омск", "однокомнатная омск"
   → Общий заголовок без гео-уточнения, базовая посадочная
```

#### Алгоритм кластеризации

```
1. СНАЧАЛА извлеки гео-кластеры:
   - Пройди по всем фразам, найди упоминания районов/микрорайонов
   - Группируй: все фразы с "нефтяники" → кластер "Нефтяники"
   - Минимум для гео-кластера: 3+ фразы ИЛИ 1 фраза с Shows >= 50

2. ЗАТЕМ остаток разбей по интенту:
   - Фразы с "купить/покупка" → прямой интент
   - Фразы с "цена/недорого/дешев" → ценовой
   - Фразы с "вторичка/собственник/хозяин/без посредников" → условия
   - Остальное → общий кластер

3. ОЦЕНИ размер кластеров:
   - Если кластер < 3 фраз и Shows < 50 — влей в ближайший по смыслу
   - Если кластер > 40 фраз — подумай, можно ли разбить ещё
   - Оптимально: 5-20 фраз на кластер

4. ДЛЯ КАЖДОГО кластера определи:
   - Название группы (для add_adgroup): "{Тематика} — {кластер}"
   - Ключевые фразы (для add_keywords)
   - Посадочный URL: базовый + фильтры (если есть в site_filters.md)
   - Специфика заголовка: что добавить/изменить в Title

5. ОБЯЗАТЕЛЬНО: создай группу для КАЖДОГО кластера с ≥ 3 фразами.
   Не ограничивайся только гео-кластерами — интент-кластеры (ценовой,
   от собственника, catch-all) тоже ДОЛЖНЫ стать отдельными группами.
   Каждый кластер = отдельная группа объявлений.
```

```
   - ЦЕНОВОЙ кластер → "Квартиры в Омске Недорого" / "Однушки от 1.5 млн"
   - УСЛОВИЯ кластер → "Квартиры от Собственника Омск"

6. Текст объявления:
   - Адаптируй под кластер: для гео — упомяни район
   - Для ценового — подчеркни выгоду
   - Для условий — подчеркни "без комиссии", "от хозяина"

7. Посадочная URL:
   - Вызови build_landing_url(city=<slug>, theme=<тематика>, rooms=<N>, district=<район>)
   - Инструмент сам соберёт URL: домен + базовый путь + фильтры из SQLite
   - Если район/фильтр не найден в БД — URL вернётся без этого фильтра (базовая страница)
   - **ПРАВИЛО: кластер ВСЕГДА создаётся**, даже без подходящего фильтра в БД.
     Группа объявлений создаётся с адаптированными заголовками (район в Title),
     но ведёт на базовую страницу по тематике. НЕ пропускай кластер из-за отсутствия фильтра.

8. Быстрые ссылки:
   - Общие для всех кластеров ИЛИ адаптированные (если есть смысл)

9. **ПРОВЕРКА ПОСАДОЧНЫХ URL (ОБЯЗАТЕЛЬНО)**
   - Для КАЖДОГО сгенерированного URL — открой страницу и убедись:
     a) Страница отвечает HTTP 200 (не 404, не 500, не редирект на главную)
     b) На странице есть объекты недвижимости (не пустая выдача)
     c) Фильтры применились корректно (район/комнатность/цена в выдаче)
   - Если страница не открывается или пустая — замени на базовый URL без фильтров
   - Используй браузер/fetch для проверки. Не пропускай этот шаг.

10. Юридическая проверка → LEGAL.md (все тексты всех кластеров)

Для каждого НОВОГО заголовка/текста — объясни гипотезу.
```

### Шаг 7. Создание через API (мульти-группы)

```
1. КАМПАНИЯ (одна на все кластеры):
   add_campaign(
     client_login=<login>,
     name="<Город> | <Тип> | <Тематика> | <Детали> | [<посадка>]",
     daily_budget_amount=<бюджет в рублях>,
     search_strategy="WB_MAXIMUM_CONVERSION_RATE",
     network_strategy="SERVING_OFF",
     priority_goals=<priority_goals_json из get_conversion_values>,
     counter_ids="<counter_id>",
     negative_keywords="<минус-слова через запятую>",
     start_date="<YYYY-MM-DD>",
     settings='[{"option":"ENABLE_AREA_OF_INTEREST_TARGETING","value":false}]'
   )

2. БЫСТРЫЕ ССЫЛКИ — один набор на всю кампанию (до цикла по кластерам):
   // Данные из get_sitelink_templates (Шаг 6). Минимум 4 ссылки, оптимально 8.
   // Без ссылок объявление теряет 20-30% CTR.
   add_sitelinks(
     client_login=<login>,
     sitelinks="<title1>|<desc1>|<url1>;<title2>|<desc2>|<url2>;..."
   )
   → sitelink_set_id  // используем один и тот же ID во всех add_ad

3. УТОЧНЕНИЯ — общий набор на всю кампанию (до цикла по кластерам):
   // Загрузить library/extensions.md → список уточнений по тематике.
   // add_ad_extension ОТДЕЛЬНО для каждого уточнения (4-8 штук):
   add_ad_extension(client_login=<login>, callout_text="Без комиссии")
     → ad_extension_id_1
   add_ad_extension(client_login=<login>, callout_text="Юридическая проверка")
     → ad_extension_id_2
   // ... всего 4-8 уточнений
   → ad_extension_ids = "<id1>,<id2>,<id3>,..."  // строка через запятую для add_ad

4. ЦИКЛ ПО КЛАСТЕРАМ — для каждого кластера:

   a) add_adgroup(
        client_login=<login>,
        campaign_id=<campaign_id>,
        name="<Тематика> — <Название кластера>",
        region_ids=[<регион>]
      )

   b) add_keywords(
        client_login=<login>,
        adgroup_id=<adgroup_id>,
        keywords="<ключевые ЭТОГО кластера через запятую>"
      )
      // ОБЯЗАТЕЛЬНЫЙ ШАГ — без ключевых группа работает только на автотаргетинге!

   c) update_adgroup(adgroup_id=<id>,
        tracking_params=<tracking_params из get_utm_config>
      )

   d) update_autotargeting(
        client_login=<login>,
        adgroup_id=<adgroup_id>,
        exact="YES", alternative="YES",
        broader="NO", accessory="NO", competitor="NO"
      )
      // ОБЯЗАТЕЛЬНО — без этого все категории автотаргетинга включены по умолчанию!
      // Оставляем ТОЛЬКО целевые (EXACT) + узкие (ALTERNATIVE)
```

> ⚠️ **Нюанс API v5 (важно):** API v5 `TextAdUpdate` напрямую не принимает
> `AdExtensionIds` (в отличие от `TextAdAdd`) — уточнения меняются через
> `CalloutSetting` с операциями ADD/REMOVE/SET. MCP-инструмент `update_ad`
> это скрывает: передавай `ad_extension_ids` — он сам транслирует в
> `CalloutSetting.AdExtensions[Operation:SET]` (полная замена набора).
> Тем не менее предпочитай указывать `ad_extension_ids` сразу в `add_ad`
> при создании — меньше запросов и нет риска частичной записи.
> Привязать/заменить уточнения на существующих объявлениях — см. ВЕТКА 3, подветка 3.8.
> **Именование групп:** `"Вторичка Однушки — Общий"`, `"Вторичка Однушки — Нефтяники"`, `"Вторичка Однушки — Кировский"` и т.д.

<!-- C8 — кросс-минусация -->
### Шаг 8. Кросс-минусация между группами (ОБЯЗАТЕЛЬНО)

**Зачем:** если в группе «Нефтяники» ключевая "купить квартиру нефтяники омск", а в группе «Общий» — "купить квартиру омск", то запрос "купить квартиру нефтяники омск" может сработать в обеих группах. Кросс-минусация гарантирует, что более точная группа забирает свой трафик.

```
1. Собери все ключевые фразы по группам (уже есть из Шага 7)

2. Для каждой пары групп найди пересечения:
   - Извлеки уникальные слова-уточнения каждого кластера
     (район, тип объекта, ценовой модификатор)
   - Эти слова → минус-слова для ДРУГИХ групп

3. Типичные кросс-минусы:
   - Гео-кластер "Нефтяники" → в группу "Общий" добавь минус "нефтяники"
   - Гео-кластер "Кировский" → в группу "Общий" добавь минус "кировский"
   - Ценовой кластер "недорого" → в группу "Общий" добавь минус "недорого","дешев"
   - Каждый гео-кластер минусует другие районы

4. Применение:
   add_negative_keywords(
     client_login=<login>,
     adgroup_id=<id группы "Общий">,
     keywords="нефтяники,кировский,левый берег"
   )
   // Повторить для каждой группы — минусуем чужие уточнения

5. ПРОВЕРКА: ни одна ключевая фраза группы не должна содержать
   собственные минус-слова этой группы (иначе фраза заблокируется)
```

> **Правило:** кросс-минусация делается ПОСЛЕ создания всех групп, когда полная картина ключевых известна.

<!-- O3.6 — кросс-минусация -->
#### 3.6 Кросс-минусация (каннибализация между группами)

**Используй когда:** группы внутри одной кампании конкурируют за одни и те же запросы — трафик уходит в менее релевантную группу.

```

   get_keywords(campaign_ids=<id>) → собери все ключевые по группам
   get_search_queries(campaign_ids=<id>, date_from=<30 дней>, attribution="LYDC")
   → Найди запросы, которые сработали в НЕСКОЛЬКИХ группах
   → Определи: в какой группе запрос более релевантен (по заголовку/посадочной)

2. ПОСТРОЕНИЕ кросс-минусов:
   - Для каждой группы определи уникальные слова-маркеры
     (район, тип объекта, ценовой модификатор)
   - Эти маркеры → минус-слова для ВСЕХ остальных групп

3. ПРИМЕНЕНИЕ:
   add_negative_keywords(
     client_login=<login>,
     adgroup_id=<id>,
     keywords="<чужие маркеры через запятую>"
   )

4. ПРОВЕРКА:
   - Ни одна ключевая фраза группы не содержит собственные минус-слова
   - Широкая группа ("Общий") минусует ВСЕ уточнения из специализированных групп
```

<!-- O3.8 — привязка sitelinks/уточнений к существующим объявлениям -->
#### 3.8 Привязка sitelinks/уточнений к существующим объявлениям

**Используй когда:** аудит показал, что у объявлений пустой `AdExtensions` или
нет `SitelinkSetId` — например, в старой кампании или после миграции.

```
1. ДИАГНОСТИКА:
   get_ads(campaign_ids=<id>, states="ON",
     text_ad_field_names="SitelinkSetId,AdExtensions")
   → Найди объявления без привязки (пустой массив AdExtensions, отсутствует SitelinkSetId)

2. ПОДГОТОВКА НАБОРОВ (если их ещё нет на аккаунте):
   - Быстрые ссылки: add_sitelinks(...) → sitelink_set_id
   - Уточнения: add_ad_extension(callout_text=...) × 4-8 штук → ad_extension_ids
   - Повторно используемые наборы искать через get_sitelinks / get_ad_extensions

3. ПРИВЯЗКА — по одному update_ad на объявление:
   update_ad(
     client_login=<login>,
     ad_id=<banner_id>,
     sitelink_set_id=<sitelink_set_id>,
     ad_extension_ids="<id1>,<id2>,<id3>,..."
   )
   // update_ad транслирует ad_extension_ids в CalloutSetting SET (полная замена набора).
   // Можно передать оба поля одним вызовом — или только одно, если второе уже есть.

4. ПРОВЕРКА:
   get_ads(ad_ids="<список>",
     text_ad_field_names="SitelinkSetId,AdExtensions")
   → Убедись что у всех объявлений AdExtensions заполнен и SitelinkSetId != 0
```

> **ПРАВИЛО:** для НОВЫХ кампаний — всегда делай привязку через `add_ad`
> (параметры `sitelink_set_id` и `ad_extension_ids`), а `update_ad` используй
> ТОЛЬКО для ретрофита старых объявлений или смены набора уточнений.

<!-- O3.7 — расширение структуры -->
#### 3.7 Расширение структуры (добавление групп в существующую кампанию)

**Используй когда:** нужно добавить новые группы объявлений в уже работающую кампанию (новые районы, новые кластеры, новая семантика).

```
Шаг А. Контекст существующей кампании (ОБЯЗАТЕЛЬНО)

1. Получи текущие группы, объявления, ключевые:
   get_adgroups(campaign_ids=<id>)
   get_ads(campaign_ids=<id>, states="ON",
     text_ad_field_names="Title,Title2,Text,Href")
   get_keywords(campaign_ids=<id>)

2. ОПРЕДЕЛИ СКОУП кампании из названия и ключевых:
   - Кампания "Вторичка | Однокомнатные" → собирай семантику ТОЛЬКО по однокомнатным
   - Кампания "Вторичка" (без уточнения) → все типы комнатности допустимы
   - Кампания "Загородка | Дома" → только дома, не участки и не дачи

3. ФИЛЬТРУЙ семантику по скоупу:
   - Если кампания по однокомнатным — исключи из ключевых: "2 комнат", "двушк",
     "двухкомнат", "3 комнат", "трешк", "трехкомнат", "4 комнат", "студи"
   - Если кампания по домам — исключи: "участ", "дач", "таунхаус"
   - Применяй фильтр НА КАЖДОМ этапе: при сборе seeds, при обработке SearchedWith,
     при финальной проверке перед кластеризацией

4. НЕ ДУБЛИРУЙ существующие ключевые — проверь перед add_keywords

Шаг Б. Сбор и кластеризация

→ Используй Шаги 4-5 из ВЕТКИ 1 (Создание), но с учётом скоупа из Шага А.
→ Исключи кластеры, которые уже есть в кампании.

Шаг В. Создание групп

→ Используй Шаги 6-8 из ВЕТКИ 1 (Создание) для новых кластеров.
→ Чеклист на каждую группу: adgroup ✓ → keywords ✓ → UTM ✓ → autotargeting ✓ → 3 ads ✓ → sitelinks ✓ → уточнения ✓ → кросс-минусация ✓
```

## Правила безопасности

1. **Изменения стратегии/цели/бюджета >30%** — ТОЛЬКО с подтверждением пользователя
2. **Лучше в чт-пт** — изменения вызывают «разведку» алгоритма
3. **Не удалять объявления** — только update_ad для изменений
4. **Атрибуция** — всегда LYDC в отчётах
5. **Бюджет** — в РУБЛЯХ, недельный, НЕ микроюниты
6. **Юридическая проверка** — все тексты через LEGAL.md

## Источники данных: MCP-инструменты vs файлы скилла

### MCP-инструменты (вызываются через сервер, НЕ загружают контекст)

| Инструмент | Заменяет файл | Что возвращает |
|---|---|---|
| `get_city_config(city)` | — | counter_id, client_login, domain, geo_region_id |
| `get_utm_config(city, theme, placement)` | — | Готовый tracking_params с utm_content |
| `get_conversion_values(client_login, counter_id)` | — | goal_id, value, priority_goals_json |
| `get_sitelink_templates(theme, city)` | — | Наборы быстрых ссылок с доменом города |
| `get_semantic_cluster(theme)` | — | Модификаторы, объекты, минус-слова |
| `get_blocked_placements()` | — | 400+ заблокированных площадок РСЯ |
| `build_landing_url(city, theme, district?, rooms?, ...)` | — | Готовый URL посадочной с фильтрами из SQLite |
| `upsert_site_filters(city, filter_type, name, url_value)` | — | Запись фильтров в БД (районы, комнатность, цена) |
| `get_site_filters(city?, filter_type?)` | — | Просмотр доступных фильтров города |

### Файлы скилла (загружаются в контекст при необходимости)

| Файл | Описание | Когда загружать |
|---|---|---|
| `config/campaign_naming.md` | Конвенция именования кампаний | При создании кампании |
| `config/labels.md` | Метки кампаний для аналитики | При создании кампании (ОБЯЗАТЕЛЬНО) |

| `library/extensions.md` | Уточнения (callouts) | При создании кампании |
| `references/site_structure.md` | URL-дерево посадочных etagi.com | При выборе посадочной |
