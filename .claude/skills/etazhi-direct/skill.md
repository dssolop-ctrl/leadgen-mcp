# Этажи Директ — управление рекламой недвижимости

Специализированный скилл для агентства недвижимости «Этажи». Оркестрирует MCP-инструменты Яндекс Директа, Метрики и Wordstat для создания, анализа и оптимизации рекламных кампаний.

**Компания:** Этажи — крупнейшее агентство недвижимости России. 70+ городов, домен etagi.com.
**MCP-сервер:** `leadgen-mcp` (Яндекс Директ + Метрика + Wordstat).
**Аккаунт Директа:** `etagi click` (default) — агентский, 76 клиентов-городов.
**Аккаунт Метрики:** `etagi.agent` — 467 счётчиков, 70 активных городов.

### Базовые тематики

| Тематика | Направление | Посадочная | Ключевые кластеры |
|---|---|---|---|
| Вторичка | Покупатель (95%) | /realty/ | купить квартиру, однушка, двушка |

| Аренда | Арендатор | /realty_rent/ | снять квартиру |

### Направление: покупатель vs продавец

- **Покупатель** (95% кампаний) — человек ищет объект для покупки/аренды. Посадочная — каталог с фильтрами.
- **Продавец** (5%) — человек хочет разместить свой объект через Этажи. Посадочная — форма оценки/размещения.
- Это РАЗНЫЕ кампании с разными ключевыми, текстами и посадочными. Не смешивать.

## Перед любой работой

### Определить город и конфигурацию

1. **Город** — спроси если не указан
2. **client_login** — найди в `config/counters.md` или через `get_agency_clients`
3. **counter_id** — найди в `config/counters.md` или через `metrika_get_counters(account="etagi.agent")`
4. **goal_id** — получи через `metrika_get_goals(counter_id, account="etagi.agent")`, выбери по приоритету из `config/goals.md`

### Принцип рассуждения

**Ты НЕ копировальный аппарат.** При каждом действии:
- Загрузи данные по аналогичным кампаниям (тот же тип в других городах)
- Проанализируй что работает (CTR, конверсии) и что нет
- Предложи улучшения с конкретным обоснованием
- Для каждого нового заголовка/текста — объясни гипотезу

## ВЕТКА 1: СОЗДАНИЕ кампании

### Шаг 1. Определение параметров

Спроси у пользователя (если не указал):

1. **Город** — какой город запускаем?
2. **Тематика** — что рекламируем?
3. **Тип размещения** — выбор:

| Тип | API | Когда использовать |
|---|---|---|
| Поиск | TEXT_CAMPAIGN (network=SERVING_OFF) | Горячий спрос, высокий интент |
| РСЯ | TEXT_CAMPAIGN (search strategy=SERVING_OFF) | Охват, ретаргетинг, имидж |
| ЕПК / Мастер | UNIFIED_CAMPAIGN (API v501) | Автоматизированная, для тестов |
| Динамические | DYNAMIC_TEXT_CAMPAIGN | Большая база объектов, фид |
| Медийная | CPM_BANNER_CAMPAIGN | Охват, узнаваемость, CPM |

4. **Бюджет** — недельный, в рублях
5. **Посадочная** — [site] (etagi.com) или [LP] (etagi.net)

### Шаг 2. Подготовка конфигурации

```
1. Загрузи config/counters.md → counter_id города, client_login
2. Вызови metrika_get_goals(counter_id, account="etagi.agent", conditions="form_sum_leads,received_real_calls,all_calls") → goal_id по приоритету из config/goals.md
3. Загрузи config/campaign_naming.md → имя кампании и UTM
```

### Шаг 3. Анализ аналогов

**ОБЯЗАТЕЛЬНЫЙ ШАГ. Не пропускай.**

```
1. Найди аналогичные кампании в кабинете:
   get_campaigns(client_login=<другой город>, states="ON", field_names="Id,Name")
   → Фильтруй по тематике (Поиск | Вторичка, Поиск | Загородка и т.д.)

2. Получи статистику лучшей аналогичной:
   get_campaign_stats(client_login=<город-донор>, campaign_id=<id>,
     date_from=<30 дней назад>, date_to=<вчера>,
     field_names="CampaignName,Impressions,Clicks,Cost,Ctr,Conversions,CostPerConversion",
     goal_ids=<goal_id_донора>, attribution="LYDC")

3. Получи объявления донора:
   get_ads(client_login=<город-донор>, campaign_ids=<id>,
     text_ad_field_names="Title,Title2,Text,SitelinkSetId,AdExtensions")

4. Напиши секцию "Анализ и гипотезы":
   - Что работает у аналога (конкретные цифры)
   - Что можно улучшить и почему
   - Новые идеи с обоснованием
```

### Шаг 4. Сбор семантики

```
1. Загрузи library/semantic_clusters.md → базовые кластеры по тематике (если файл существует)
2. Расширь через check_search_volume(phrases="<базовый запрос>") для конкретного города
3. Проверь частотность ключевых
4. Загрузи library/negative_keywords.md → типовые минус-слова
5. Дополни минус-словами из анализа поисковых запросов аналога (если есть данные)
```

### Шаг 5. Создание контента

```
1. Загрузи library/titles.md → заголовки по тематике
2. Загрузи library/texts.md → тексты объявлений
3. Загрузи library/sitelinks.md → быстрые ссылки
4. Загрузи library/extensions.md → уточнения

Для каждого элемента:
- Выбери лучшие из библиотеки, адаптируй под город
- Предложи минимум 1 НОВЫЙ вариант
- Для каждого нового — объясни гипотезу
- Проверь лимиты символов

5. Юридическая проверка → LEGAL.md
```

### Шаг 6. Создание через API

```
1. add_campaign(
     client_login=<login>,
     name="<Город> | <Тип> | <Тематика> | <Детали> | [<посадка>]",
     daily_budget_amount=<бюджет в рублях>,
     search_strategy="WB_MAXIMUM_CONVERSION_RATE",
     network_strategy="SERVING_OFF",  // для поиска
     goal_id=<goal_id>,
     counter_ids="<counter_id>",
     negative_keywords="<минус-слова через запятую>"
   )

2. add_adgroup(
     client_login=<login>,
     campaign_id=<id>,
     name="<Группа — тематика>",
     region_ids=[<регион>],
     keywords=["фраза 1", "фраза 2", ...],
     autotargeting_categories=[
       {category: "EXACT", value: true},
       {category: "ALTERNATIVE", value: true},
       {category: "BROADER", value: false},
       {category: "ACCESSORY", value: false},
       {category: "COMPETITOR", value: false}
     ]
   )

3. update_adgroup(adgroup_id=<id>,
     tracking_params=<UTM из config/campaign_naming.md — полный шаблон с utm_content>
   )

4. add_ad × 2 (два варианта для A/B теста)

5. add_sitelinks + привязать к объявлениям

6. add_ad_extension (уточнения) + привязать
```

### Шаг 7. Метки кампании (ОБЯЗАТЕЛЬНО)

```
1. Загрузи config/labels.md → определи метки по тематике и направлению
2. Вызови add_labels(campaign_ids=<id>, labels="Лидген,Вторичка,Покупатель", client_login=<login>)
3. Проверь результат через get_labels(campaign_ids=<id>, client_login=<login>)
```

> **Без меток статистика не попадёт в отчёты.** Не пропускай этот шаг.
> Метки работают через API v4 Live (GetCampaignsTags / UpdateCampaignsTags).

### Шаг 8. Документирование

Создай файл `campaigns/<slug>.md` по шаблону `campaigns/template.md`.
Запиши гипотезы агента — для сравнения при анализе.

## ВЕТКА 2: АНАЛИЗ кампаний

### Источники данных

| Источник | Инструмент | Когда |
|---|---|---|
| **Яндекс Директ API** | `get_campaign_stats` | Расход, клики, показы, CTR, конверсии |
| **Яндекс Метрика API** | `metrika_get_direct_report` | Визиты, отказы, глубина, конверсии |
| **Внешний дашборд** | _(будущий MCP #2)_ | Агрегированные данные, быстрая сводка |

### Flow

```
1. ОПРЕДЕЛИТЬ СКОУП
   - Одна кампания (по ID или имени) → детальный аудит
   - Все кампании города → дашборд
   - Все города → сводка

2. СБОР СТАТИСТИКИ
   get_campaign_stats(
     client_login=<login>,
     campaign_id=<id>,  // или без → все кампании
     date_from=<7/30 дней назад>,
     date_to=<вчера>,
     field_names="CampaignName,Impressions,Clicks,Cost,Ctr,AvgCpc,Conversions,CostPerConversion",
     goal_ids=<goal_id>,
     attribution="LYDC"
   )

3. ДАННЫЕ МЕТРИКИ (для детального аудита)
   metrika_get_direct_report(
     counter_id=<id>,
     account="etagi.agent",
     date1=<7 дней назад>,
     date2=<вчера>,
     utm_campaign=<campaign_id>
   )

4. ОЦЕНКА ЗДОРОВЬЯ
   | Метрика | GOOD | ATTENTION | CRITICAL |
   |---------|------|-----------|----------|
   | CPA | < порог из benchmarks | < порог × 1.5 | > порог × 1.5 |
   | CTR (поиск) | > 5% | 3-5% | < 3% |
   | CTR (РСЯ) | > 0.5% | 0.3-0.5% | < 0.3% |
   | Отказы | < 30% | 30-50% | > 50% |

5. ДИАГНОСТИКА ПРОБЛЕМНЫХ
   - Поисковые запросы: get_search_queries → waste / opportunities
   - CTR объявлений: get_ad_stats → какое проигрывает
   - Ключевые: get_criteria_stats → неэффективные фразы

6. ОТЧЁТ
   - Сводная таблица с светофором
   - Конкретные рекомендации (входные данные для ОПТИМИЗАЦИИ)
   - Обновление campaigns/<utm>.md
```

## ВЕТКА 3: ОПТИМИЗАЦИЯ

**Принцип:** ВСЕГДА начинай с данных из ветки АНАЛИЗ. Не оптимизируй вслепую.

### Подветки (выбор по ситуации)

#### 3.1 Минус-слова (высокий CPA, мусорные запросы)

```
1. get_search_queries(campaign_ids=<id>, date_from=<30 дней>, attribution="LYDC")
2. Анализ: расход > 0, конверсии = 0 → кандидаты в минус
3. Информационные запросы → в минус (сверить с references/intent_patterns.md)
4. ПОДТВЕРЖДЕНИЕ пользователя перед применением
5. update_campaign(campaign_id, negative_keywords=<список>)
```

#### 3.2 Новые ключевые (упущенный спрос)

```
1. get_search_queries → запросы с конверсиями, но не в ключевых
2. check_search_volume(phrases="<фразы>") → проверить частотность
3. ПОДТВЕРЖДЕНИЕ пользователя
4. add_keywords(adgroup_id, keywords=[...])
```

#### 3.3 A/B тест объявлений (низкий CTR)

```
1. get_ad_stats → определить слабое объявление
2. Загрузить library/titles.md, library/texts.md
3. Создать новый вариант с гипотезой
4. ПОДТВЕРЖДЕНИЕ пользователя
5. add_ad(adgroup_id, ...)
6. Через 2-3 недели: сравнить, остановить проигравшего
```

#### 3.4 Смена стратегии (достаточно конверсий)

```
Условие: 10+ конверсий/нед стабильно 2-3 недели подряд
1. ПОДТВЕРЖДЕНИЕ пользователя (лучше в чт-пт)
2. update_campaign(search_strategy="AVERAGE_CPA", average_cpa=<целевой CPA>)
```

#### 3.5 Бюджет

```
- Увеличение: не более +30% за раз, после 2+ недель стабильного CPA
- Снижение: если CPA выше нормы и оптимизация не помогла
- Сезонность: wordstat_dynamics → скорректировать по сезону
- ПОДТВЕРЖДЕНИЕ пользователя для любых изменений > 10%
```

---

## Правила безопасности

1. **Изменения стратегии/цели/бюджета >30%** — ТОЛЬКО с подтверждением пользователя
2. **Лучше в чт-пт** — изменения вызывают «разведку» алгоритма
3. **Не удалять объявления** — только update_ad для изменений
4. **Атрибуция** — всегда LYDC в отчётах
5. **Бюджет** — в РУБЛЯХ, недельный, НЕ микроюниты
6. **Юридическая проверка** — все тексты через LEGAL.md
