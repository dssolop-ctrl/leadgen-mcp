## 1. Фильтр района в посадочной URL (гео-кластеры без фильтра)

### Замечание
> «Ссылку на фильтр по району не сделала в группах по району»

### Корень проблемы
- `mcp/site_filters.md` документирует только `rooms` и `installment`. Для районов/микрорайонов явно помечено `ТРЕБУЕТ ИССЛЕДОВАНИЯ`.
- В `skill.md` на шаге C6 п.7 прописано: *«если район/фильтр не найден в БД — URL вернётся без этого фильтра (базовая страница)»* → агент послушно падает на базовый URL, район в Title остаётся, но юзер попадает в общий каталог.
- `upsert_site_filters` и автоматическая работа с БД фильтров есть (MCP сервер уже поддерживает seed через git-трекнутый `filter_values.json`), но **в флоу нет шага разведки недостающих фильтров**.

## 2. УТП по направлениям (allow/deny-list)

### Замечание
> «"Без комиссии с покупателя" уточнение по вторичке неверное, надо делать чтоб какие то УТП можно на такие то направления, а такие то нельзя вот туда»

### Корень проблемы
- В `library/extensions.md` есть разделы по тематикам (Вторичка, Новостройки и т.д.), но библиотека не содержит **жёстких запретов** вида «это уточнение запрещено на теме X».
- «Без комиссии» — стоит в разделе «Новостройки» (где комиссия реально 0 для покупателя), но агент может схватить его как «общеподходящее» и поставить на вторичку, где это ложь (комиссия с покупателя есть).
- Юридически это тоже рискованно — ложная выгода = потенциальная претензия от ФАС.

### План доработки

**2.1. Переструктурировать `library/extensions.md`**

Ввести каноническую структуру:

```markdown
| Текст | Символов | Разрешено на | Запрещено на | Почему запрещено |
|---|---|---|---|---|
| Без комиссии | 12 | Новостройки, Коммерческая (аренда) | **Вторичка, Загородка** | Реальная комиссия есть |
| Сделка без риска | 16 | Все | — | |
| Юридическая проверка | — | Все | — | |
| ... | | | | |
```

Или:

```markdown
## Вторичка
### ✅ Разрешены
| Текст | Символов |
|---|---|
| ... |

### ❌ ЗАПРЕЩЕНЫ (типовые ошибки)
| Текст | Почему |
|---|---|
| Без комиссии | Комиссия с покупателя есть |
| Рассрочка 0% | Только для новостроек |
| Квартиры от застройщика | Это новостройка, не вторичка |
```

Deny-list должен быть у **каждой** тематики с явным объяснением «почему нельзя».

**2.2. Хард-чек в `skill.md` шаг C6**

Добавить в C6 п.4 (после выбора уточнений):

```
ПРОВЕРКА УТП ПЕРЕД ЮРАВВОДОМ:
1. Для каждого выбранного уточнения проверь таблицу «Запрещено на» в library/extensions.md
2. Если текущая тема кампании есть в deny-list → ЗАМЕНИ уточнение
3. Если таблица не содержит темы — явно отметь уточнение как «проверить
   вручную» и спроси пользователя перед add_ad_extension
```

**2.3. Расширить запреты (гипотезы на подумать)**

- **Вторичка / покупатель:** нельзя — «Без комиссии», «От застройщика», «Рассрочка 0%», «Семейная ипотека» (если нет реальной программы для вторички в Этажи).
- **Загородка:** нельзя — «Без комиссии» (если комиссия есть), «Квартиры» (путает с вторичкой).
- **Новостройки:** нельзя — «От собственника», «Вторичка», «Без посредников» (этого нет для новостроек — там застройщик и есть посредник).
- **Аренда:** нельзя — «Без комиссии» для жилой (обычно комиссия агента есть), «Ипотека» (не ипотечная реклама).
- **HR:** нельзя всё про сделки/объекты.

Спец должен **валидировать этот список** — добавлять доп. запреты после каждого запуска, которые всплывают на отсмотре.

**2.4. Связь с `LEGAL.md`**

В `LEGAL.md` (рядом с общими запрещёнными терминами) добавить подраздел «Фактические запреты по направлениям» — синхронизировать с allow/deny матрицей.

### Затрагиваемые файлы
- `.claude/skills/etazhi-direct/library/extensions.md` — переструктурировать
- `.claude/skills/etazhi-direct/skill.md` — Шаг C6 проверка УТП
- `LEGAL.md` — фактические запреты по направлениям (корневой файл)

### Приоритет
**P1** — прямое вранье в объявлении = блокер на модерации и риск ФАС.

## 3. Ключи без обязательной привязки к городу

### Замечание
> «ключи везде используются только с привязкой города "омск", т.е. основной трафик теряем по "купить двушку недорого", "купить двухкомнатную квартиру"»

### Корень проблемы
- В `mcp/semantic_clusters.md` п.1 правила: *«в ключи вставляй название города (например, «купить квартиру Тюмень»)»*.
- В `skill.md` Шаг C4 Раунд 1: *«Построй матрицу: {модификатор} × {объект} + {город}»* + пример «купить квартиру омск».
- Логика ошибочная: гео-таргетинг в Директе отсекается через `region_ids` кампании/группы. Добавляя город в каждый ключ, мы теряем трафик от людей, которые **не уточняют город** в запросе (а они уже и так в Омске — гео их отсекает).
- Запросы вида «купить двушку недорого» с Shows тысячами — главный транзакционный трафик. Мы его не ловим.

### План доработки

**3.1. Переписать принцип `semantic_clusters.md`**

Заменить п.1 правил:

- ❌ Было: «`{город}` / `ГОРОД` — заменяй на конкретный город и его варианты (Тюмень / в Тюмени / тюменский)»
- ✅ Стало:

```markdown
1. **ГЕО-таргетинг идёт через region_ids кампании, НЕ через слово в ключе.**
   Город в ключе добавляй ТОЛЬКО для уточняющих запросов, где пользователь
   сам пишет город (например, «квартира омск», «купить квартиру омск от хозяина»).

   - БАЗА: {модификатор} × {объект} БЕЗ города
     Примеры: «купить однокомнатную», «купить двушку недорого»,
     «двухкомнатная квартира цена», «купить квартиру в ипотеку»
   - ГЕО-УТОЧНЁННЫЕ: {модификатор} × {объект} × {город}
     Примеры: «купить квартиру омск», «квартира омск от хозяина»
     → отдельный мини-кластер «Брендозапросы города»

2. **Защита от перелива в другие города** — через минус-слова.
```

## 4. Продающие прибавки → больше интент-кластеров

> «групп тоже маловато, продающие прибавки только две, "недорого" и "купить", есть еще же дешево, покупка, приобрести, цена, в ипотеку, объявления»
>
> «крч возможно надо добавить список определенных прибавок»

### Корень проблемы
- `semantic_clusters.md` → модификаторы для вторички: `купить, недорого, без посредников, авито, циан, домклик, покупка, цена, продажа, стоимость, от хозяина, объявления, дешевая` — **список есть**, но агент на C5 свернул всё в 2 интент-кластера (Купить / Недорого).

**Корень:**

**План:**

Обработчик:
```go
if dup := common.GetString(req, "display_url_path"); dup != "" {
```
